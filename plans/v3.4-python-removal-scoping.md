# v3.4-python-removal — retire the Python CLI (scoping)

**Status:** DRAFT v1 (pre-mediator-review) · **Authored:** 2026-04-21 · **Author:** claude (lead-drafting at owner request) · **Trigger:** owner conversation post-v3.3 — "i'm ok with full removal as long as we can test them" · **Positions as:** immediate follow-on to `plans/v3.3-symmetric-federation-scoping.md` — v3.3 landed 7 items built on the existing Python CLI; v3.4 removes the Python dependency without regressing behavior.

> **Authorship note:** owner-requested lead-drafted first pass. Parity-testability is a ratified owner constraint — "as long as we can test them" — so the test harness precedes every port in Phase 0.

## 1. Goal + motivation

`scripts/peer-inbox-db.py` is ~5200 lines of Python that owns every CLI verb (session-register, peer-send, peer-broadcast, peer-round, peer-receive, peer-list, peer-watch, peer-replay, room-create, peer-reset, peer-queue, session-list, session-close, session-adopt, hook-log-session, peer-web fallback). The v3.0 Go rewrite ported the hot paths (hook, daemon, peer-web reads), but v3.2 and v3.3 kept adding Python — every write path still goes through the Python script, including the federated `/api/send` flow (Go shells out to Python per request).

**Owner motivation verbatim (2026-04-21):** *"i'm ok with full removal as long as we can test them."*

**Why now.**

- Every federated send costs **two Python startups** (~100–300ms each) — laptop's CLI + orange peer-web's shell-out. Federation made this visible.
- Python 3.14 appeared on the owner's system during v3.3 work, surfacing version-drift risk (e.g. strptime behavior changes, stdlib churn).
- Go binaries compile to a single static file; Python deploys carry a lockstep interpreter + pathing concerns. Orange's first-ever deploy (v3.x-server-deployment) tripped on this several times.
- The Go store package already knows the entire schema + goose version contract. Porting verbs reuses infrastructure, not rewrites it.

**Success criteria.**

1. `scripts/peer-inbox-db.py` is deleted from the repo.
2. Every verb runs natively in Go with identical behavior, asserted by byte-level parity tests.
3. No shell-outs from Go binaries to Python anywhere in the peer-web / daemon / hook / migrate code paths.
4. `install-global-protocol` stops distributing the Python script.
5. Tests that previously invoked `python3 scripts/peer-inbox-db.py` invoke the Go `peer-inbox` binary instead; no `python3` in test dependencies (except where Python is the *subject under test*, e.g. the claude-api skill).

**What this does NOT do.**

- Remove Python from the `extensions/peer-inbox-pi.ts` → `peer-inbox-pi` dependency chain (that's Node, not Python).
- Remove `pi` / `pi-ext` which are TypeScript+Node.
- Change any user-visible CLI surface (`agent-collab peer send` still works the same; underlying binary swaps).

## 2. Architecture: the parity seam

The heart of v3.4 is not the port — it's the parity harness that proves the port didn't regress anything.

**Key insight:** every verb reduces to *(input state, command, inputs) → (output state, stdout, stderr, exit code)*. If we can snapshot all four outputs for the Python implementation against a seeded DB, we can run the Go implementation against the same seed and diff. Byte-level equality is the gate.

This gives us:

- **Regression protection during port.** The Go port can't silently drift from Python semantics (termination tokens, turn-cap arithmetic, mention resolution, etc.).
- **Confidence at deletion time.** When Python is removed, the harness becomes pure Go-vs-Go (new Go against frozen Go snapshots from the deletion moment) — documentation of current behavior persists even without Python.

**Seam design:**

- Each verb's fixture is a JSON file under `tests/fixtures/parity/<verb>/<scenario>.json` with `{"seed": {...schema rows...}, "args": [...], "stdin": "...", "env": {...}}`.
- Golden output fixture: `<verb>/<scenario>.expected.json` with `{"stdout": "...", "stderr": "...", "exit": N, "db_after": {...schema rows...}}`.
- Harness: `tests/parity/run.sh <verb> <scenario> <impl>` runs either Python or Go against the seed + diffs against expected. Exits non-zero on any divergence.
- CI hook: every PR runs the full suite across both implementations.

**What the harness cannot prove** — documented explicitly:

- **Unix-socket push timing.** Channel pushes to live peers are timing-sensitive and async. Tested with a mock listener that records payloads; exact timing not asserted.
- **Process-tree walks** (ancestor PID discovery for session-id inference). Dependent on harness invocation environment; tested via injected PPID fixture.
- **MCP server spawn** (interaction with Claude Code's CLI). Integration-tested with the existing `daemon-harness.sh` pattern, not unit-tested.

These edge cases are ported last, after the harness has proven itself on 80% of the surface.

## 4. Out of scope

- **Schema changes.** v3.4 is behavioral-parity-only. If a verb has a latent bug, Go reproduces it. Bug fixes land separately (v3.4.x) after parity is proven.
- **CLI UX improvements.** Flag names, help text, output formatting stay byte-identical during the port. Improvements come later, with proper changelog notes.
- **Performance tuning beyond "no Python startup tax."** The Go rewrite will naturally be faster for anything that was bottlenecked on Python startup; micro-optimizations beyond that are v3.5 territory.
- **MCP server rewrite.** The peer-inbox MCP server is separate infrastructure; untouched.
- **pi / pi-ext / oozoo integration.** Node/TypeScript, unrelated.
- **Python-based tests that are *about* Python** (e.g. any future test exercising claude-api Python SDK). Those are different concerns and keep python3 on the dev machine; they just don't invoke `peer-inbox-db.py`.

## 5. Per-phase impl surfaces

### Phase 0 — Parity harness

**Fixture format** (`tests/fixtures/parity/<verb>/<scenario>.json`):

```json
{
  "description": "session-list returns empty when no sessions exist",
  "seed": {
    "goose_version": 7,
    "sessions": [],
    "inbox": [],
    "peer_rooms": []
  },
  "args": ["--cwd", "/tmp/x"],
  "stdin": null,
  "env": {
    "AGENT_COLLAB_SELF_HOST": "testhost"
  }
}
```

**Expected output fixture** (`<scenario>.expected.json`):

```json
{
  "stdout": "no sessions\n",
  "stderr": "",
  "exit": 0,
  "db_after": {
    "sessions": [],
    "inbox": [],
    "peer_rooms": [],
    "pending_outbound": []
  }
}
```

**Runner** (`tests/parity/run.sh`):

- Reads fixture + expected
- Creates scratch DB, applies migrations via `peer-inbox-migrate`
- Seeds with fixture rows
- Invokes either `python3 scripts/peer-inbox-db.py <verb>` or `go/bin/peer-inbox <verb>` with fixture args/stdin/env
- Snapshots actual output + DB state
- Diffs against expected; exits 1 on any divergence
- Also writes `<scenario>.actual.json` for human inspection on failure

**Regenerate path** (`tests/parity/regenerate.sh <verb> <scenario>`): runs the *Python* impl and writes the result as `<scenario>.expected.json`. Used once per fixture to bootstrap the golden. Not used in CI (CI is read-only).

**CI integration:** new `tests/parity-all.sh` iterates every fixture × every impl, reports a per-verb parity matrix. Fails the build on any red cell.

**First fixture:** `session-list/basic` as the worked example. Proves the harness shape; everyone else follows the same template.

### Phase 1 — `/api/send` native Go

The payoff phase. Ports the heaviest shell-out in peer-web.

- New package `go/pkg/peerinbox/` with:
  - `Send(ctx, params) (SendResult, error)` — local SQLite write with all the existing semantics (turn cap, termination, server_seq, idempotency, channel push, mention resolution).
  - `BroadcastLocal(ctx, params) ([]SendResult, error)` — fan-out variant.
  - `SendRemote(ctx, params, remoteCfg) (SendResult, error)` — HTTP POST to another peer-web; mirrors Python's `_peer_send_remote`.
  - `FlushOutboundQueue(ctx, homeHost) (FlushStats, error)`.
- `handleSend` in `peer-web/server/send.go` rewrites to call these instead of shelling out.
- Owner auto-register stays (still calls `peerinbox.RegisterSession` instead of shelling out).
- Bearer validation already landed in v3.3 Item 7 — no change there.

**Parity gate:** every existing peer-send CLI fixture must pass with the Go `peer-web /api/send` handler producing the same DB state the Python `peer-send` produced, given the same inputs.

### Phase 2 — Read-only verbs

Ordering: `session-list` → `peer-list` → `peer-replay` → `peer-watch` → `peer-receive`. Each lands with its fixture suite.

`peer-watch` is the only long-running verb — ports to Go's `fsnotify` instead of Python's poll loop.

### Phase 3 — Write verbs

Ordering: `session-register` (needed by fixture seeding anyway), `room-create`, `peer-reset`, `peer-queue`, then `peer-send`, `peer-broadcast`, `peer-round`.

The v3.3 federation code (home_host lookup, remotes.json, outbound queue, ULID, bearer-token passing through HTTP) ports along with `peer-send`.

### Phase 4 — Edge verbs

`hook-log-session` is already effectively in Go (the hook binary writes to the DB). The verb lives in Python for CLI-invocation parity. Port is a thin wrapper.

`session-adopt` / `session-close` are straightforward.

Python `peer-web` fallback: the Go peer-web already handles everything the Python version does. Phase 4 confirms no fixture calls Python peer-web; then the fallback goes away.

### Phase 5 — Flip default

- Bash wrapper `scripts/agent-collab` dispatches: `${AGENT_COLLAB_IMPL:-go}` selects binary.
- `install-global-protocol` installs only the Go binary by default; passes `--install-python-fallback` opts in to the script.
- Doc update: `docs/PEER-WEB-GUIDE.md`, `AGENT-COLLABORATION.md`, `README.md` stop mentioning the Python script as the primary path.

### Phase 6 — Delete Python

- `rm scripts/peer-inbox-db.py`.
- Remove Python from bootstrap installer dep checks.
- Remove `AGENT_COLLAB_IMPL` env var.
- Parity fixtures stay: they now diff current Go against frozen Go snapshots of what the semantics *were* at Python-removal time.

## 6. Risks + mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Unix-socket push semantics differ subtly between Python and Go. | Silent message loss to live peers. | Phase 0 harness includes a mock socket listener; Phase 3 tests assert exact byte payloads. |
| Python's argparse defaults / exit codes drift. | User-facing behavior change. | Parity fixtures assert exit codes byte-for-byte. Go CLI uses `flag` or a compatible library (spf13/cobra) with hand-verified parity. |
| Process-tree walks for session-id discovery are OS-dependent. | Breaks on Linux vs macOS differently. | Port with explicit per-OS paths; parity tests run on both in CI. |
| MCP spawn integration breaks in production. | Daemon can't launch Claude sessions. | `daemon-harness.sh` already integration-tests this; extend to Go binary during Phase 4. |
| Hidden couplings between verbs (e.g. session-register side effects read by peer-send). | Verb-at-a-time porting misses integration bugs. | Parity tests include multi-verb scenarios (e.g. register + send + list). |
| Go needs a ULID library (currently Python stdlib implementation). | Dependency creep. | Reuse the same 15-line Crockford-base32 impl in Go; zero deps. |
| Tests still invoke `python3 scripts/peer-inbox-db.py`. | Can't actually delete Python. | Phase 0 adds a linter that greps for `python3.*peer-inbox-db`; fails CI after Phase 5. |

## 7. Ratification + sequencing

- **Phase 0 is the gate.** No porting code lands before the harness is green with one working fixture. This is the ratified owner-constraint's enforcement point.
- **Phase 1 is the motivator.** Biggest performance + deploy-simplicity win; probably the commit the owner cares about most.
- **Phases 2–4 are grind.** Each verb is a self-contained unit of port-and-fixture work.
- **Phase 5 + 6 are close-out ceremony.** Low-risk if the parity harness has done its job.

## 8. Open questions

1. **Which Go CLI library?** `flag` (stdlib) is minimal; `spf13/cobra` is richer but a dep. Python's argparse is closer to cobra in feature set. Recommendation: use `flag` — it's already in use for other Go binaries in this repo, and keeping zero-dep simplifies the deletion story.
2. **Python script in git history.** After deletion, `git log scripts/peer-inbox-db.py` still works for anyone who wants to read the original. No tombstone file needed.
3. **Renaming `peer-inbox` binary.** Today `go/cmd/peer-inbox/` exists but is used only for some subcommands. Post-v3.4 it becomes the canonical CLI. Consider renaming `go/cmd/peer-inbox/` → `go/cmd/agent-collab/` for clarity, or keep the name to avoid churn. Decide at Phase 5.
4. **Binary distribution.** Should `install-global-protocol` download pre-built binaries (from a release) or require `go` at install time? Phase 5 gates this.
