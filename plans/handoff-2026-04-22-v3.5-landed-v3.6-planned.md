## v3.5 recap — what shipped

**Architecture:** `sessions.channel_socket` now holds a URI (or bare
unix path for legacy). Dispatch by scheme:

- `unix:///path/.sock` → `sendOverUnixSocket` (same-host, kernel push)
- bare path `/path/.sock` → same (legacy format)
- `http(s)://host/api/channel-push?label=X&pair_key=Y` → HTTP POST

**New endpoint:** `POST /api/channel-push` on every peer-web. Receives
a push from a remote host, looks up the local session with matching
`(label, pair_key)`, forwards to its unix socket. Becomes the **per-host
channel gateway**.

**CLI:** `session-register --channel-uri <uri>` overrides auto-detect.

**Zero schema migration** — column reinterpreted.

**Key files:**
- `go/pkg/store/sqlite/send.go` — `pushChannel` URI dispatcher
- `go/pkg/store/sqlite/system_event.go` — same dispatch
- `go/pkg/store/sqlite/channel_push.go` — `ForwardLocalPush` store method
- `go/cmd/peer-web/server/channel_push.go` — `/api/channel-push` handler
- `go/cmd/peer-inbox/session_register.go` — `--channel-uri` flag

## v3.6 — what's planned next

Full spec in `plans/v3.6-multi-room.md`. The short version:

**Problem:** a single Claude session already CAN be in multiple rooms
today (share `channel_socket` across multiple `sessions` rows), but the
channel MCP resolves self from the socket with `LIMIT 1` — it always
picks the first row. So:

- Incoming `<channel>` tags are mis-stamped with wrong pair_key.
- `peer_inbox_reply` always sends into the wrong room.

Concrete failure observed in S3 above: my session was in two rooms,
orange sent into `v35-live-...`, but the `<channel>` header showed
`pair_key="federation-demo-..."` (the first-registered one).

**Fix (4 pieces, ~2 hours implementation):**

1. **Sender injects `meta.pair_key`** in `send.go` + `system_event.go`
   push payloads. One-line additions.

2. **HTTP gateway passthrough**: `/api/channel-push` copies the
   `?pair_key=` query param into `meta.pair_key` if not already set by
   the sender (legacy compat). ~5 lines in `channel_push.go`.

3. **Channel MCP reads `meta.pair_key`** on incoming to stamp the
   correct `<channel>` header. Returns a LIST for `_resolve_self_from_socket`.
   Tracks "last incoming pair_key" as default reply context.
   ~30 lines in `peer-inbox-channel.py`.

4. **`peer_inbox_reply` tool gains optional `pair_key` arg.** Default
   = last-incoming. Explicit override wins. Error when ambiguous.

**Zero schema change.** The multi-row-per-socket pattern already works.

**Test matrix:**

- M1: two rooms on same host, same socket — both deliverable with
  correct pair_key; default reply targets last-sender's room; explicit
  pair_key overrides.
- M2: multi-room across federation — local unix push + federated HTTP
  push both carry correct pair_key.
- M3: ambiguity error path — fresh session with two rooms and no prior
  incoming errors with a helpful "pass pair_key=..." message.
