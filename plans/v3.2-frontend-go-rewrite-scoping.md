## 1. Goal + motivation

Replace the embedded Python SPA at `cmd_peer_web` (`scripts/peer-inbox-db.py`, symbol-keyed; ~1200 LOC including inline HTML template) with a Go binary that (a) navigates across all rooms + cwd-scoped pairs on one machine instead of requiring `--pair-key` at process start, and (b) removes Python from the web-serving path.

**Two owner-surfaced pains, one fix:**

1. **Single-scope lock.** Today the server takes `--pair-key KEY` at startup and renders exactly that room; cross-room visibility requires killing the process and relaunching with a different key. For an owner tracking multiple live rooms (v0.3+ pattern), this is the dominant friction point.
2. **Python maintenance aversion.** Owner-stated preference. Every patch to `cmd_peer_web` is throwaway work if rewrite is inevitable.

**Why one fix works for both.** A rewrite lets multi-room navigation ship as the first-class entry point rather than a patch on top of the single-scope lock. Patching Python first = same UX work, twice.

**Why now, not in v3.2-α.** The v3.2-α workbench (findings panel + stance stripe + convergence replay) is gated on v3.1 write-time redaction per `v3.2-frontend-brainstorm.md` §Safety-gate, because v3.2-α exposes *new* structured data on localhost:8787. This rewrite does NOT add new structured data — it re-serves the existing `/scope.json` + `/pairs.json` + `/rooms.json` + `/messages.json` shapes from Go, plus a new multi-room index view over data already readable from `~/.agent-collab/sessions.db`.

**Gate applicability — drafter closes, owner-delegated (Q2, see §11 SR8):** the rewrite expands the aggregation surface — one process serves all rooms instead of one — but does NOT widen the data shape rendered on :8787. The brainstorm's §Safety-gate ruling had two prongs: **(prong 1)** :8787 is broader than filesystem access, therefore any port-visible surface counts as exposure; **(prong 2)** workbench structured additions (findings panel, stance stripe, lean-history) compound that exposure. Prong 1 already fires against Python's :8787 today; rewrite doesn't worsen prong 1 in a meaningful direction (one port serving all rooms is not obviously worse than N ports each serving one room — same data, same reach, fewer attack surfaces for operators auditing localhost binds). Prong 2 requires new structured data shapes; this cluster ships none — only reformats the existing `/messages.json` payload. Drafter's close: **does not trip the gate; rewrite can ship before v3.1 redaction.** Mitigation `--only-pair-key K` flag preserved for operators wanting narrow-exposure semantics regardless. Mediator may re-open on absorption; §11 SR8 captures the closing argument.

**Why Go, not Node/Rust/other.** `go/cmd/peer-inbox` and `go/cmd/daemon` already share the SQLite backend via `go/pkg/store/sqlite`. Reusing that store package is the lowest-friction path; introducing a third runtime would contradict the v3.0 Go consolidation direction.

## 4. Per-item impl surfaces

**Item 1 — `go/cmd/peer-web` binary.**
- Surface: new `go/cmd/peer-web/main.go` + likely `go/cmd/peer-web/server/` package for handler separation.
- Reuses: `go/pkg/store/sqlite` for DB access (same connection pattern as `go/cmd/peer-inbox`), `embed.FS` for static assets, `html/template` for the index page. Detail-page HTML can be ported as a single `embed`-ed file first; move to templates only if parameter binding demands it.
- Entry: `peer-web --port 8787 [--db PATH]`. No `--pair-key` startup flag; scope is per-request via URL.

**Item 2 — multi-room index route.**
- Surface: `GET /` handler + new JSON endpoint (e.g. `/index.json`) the JS client polls for room-list updates.
- Query: union of (a) every row in `peer_rooms` with `room_key LIKE 'pk:%'` (pair-key rooms) and (b) every distinct `(from_cwd, from_label, to_label)` triplet in `inbox` grouped by canonical pair + cwd (cwd-scoped edges). Activity dot = max-freshness across members. Unread count = `inbox.read_at IS NULL` aggregated by scope.
- Out: index template renders two sections (`rooms` + `cwd pairs`) with click-through links to detail routes.

**Item 3 — unified detail route.**
- Surface: `GET /view?scope=<pair_key|cwd>&key=<K>|&path=<urlencoded-cwd>` → same SPA as today. Query-param over path-segment encoding keeps URLs predictable and lookup-table-free.
- JS client detects scope from URL query + `/api/scope` payload; existing SPA code (`peer-inbox-db.py` `_PEER_WEB_HTML_TEMPLATE` symbol) copy-pasted into a Go-embedded `.html` file, with scope resolution swapped from process-global to per-fetch query params.

**Item 4 — API parity.**
- Surface: five handlers, one per current endpoint. Each takes scope from URL path params instead of process-start config.
- Critical invariant: JSON output shape byte-identical so the existing JS client (also being ported) works unchanged. Parity probe: curl same endpoints against Python + Go with the same DB, diff output.
- `POST /send` → see Item 6.

**Item 5 — CLI dispatch + deep-link preservation.**
- Surface: `scripts/agent-collab` `peer web` case + new Go CLI flags in `go/cmd/peer-web/main.go`.
- Dispatch: if Go binary reachable on PATH, exec that; else fall back to Python `peer-inbox-db.py peer-web`. Env `AGENT_COLLAB_WEB_IMPL=python` forces fallback for bug reports.
- Flag semantics (Go binary):
  - `--port 8787` — bind port (unchanged).
  - `--pair-key K` — deep-link shortcut: server still multi-room, but index redirects to `/view?scope=pair_key&key=K` on first visit. Browser-convenience flag.
  - `--only-pair-key K` — strict mode: server rejects requests for any scope other than pair-key K (safety mitigation per §1 gate; for operators wanting single-room exposure).
  - `--cwd PATH` — cwd-mode analogue of `--pair-key` (backwards-compatible with today's Python default).

**Item 6 — composer send path.**
- Decision (Q1 closed): shell out to `scripts/peer-inbox-db.py peer-send` / `peer-broadcast`. Matches today's in-Python self-invoke at `cmd_peer_web` `_script_self_invoke` symbol. Evidence: grep for `peer-send|peer-broadcast|peerSend|peerBroadcast` in `go/` returns only a prompt-template reference in `go/cmd/daemon/main.go` (no executable verb). Implementing Go-native send would duplicate room-cap + termination + push logic and is its own cluster.
- Implementation: `os/exec` Python subprocess with 15s timeout (matches Python `_script_self_invoke`). Reuses Python session-resolution + auto-register-owner path byte-for-byte.

## 12. v3 close — Q2 drafter-closed under owner delegation

Owner-delegated close: "you decide".

**SR8 — Q2 closed: rewrite does NOT trip the v3.2-α redaction gate.**

Closing argument:

The brainstorm §Safety-gate ruling identified two prongs of exposure amplification for v3.2 dashboard work:

1. **Port-visibility prong:** localhost:8787 is broader than filesystem access — any process with localhost network access reads bodies, vs. the filesystem-access-required baseline at `~/.agent-collab/sessions.db`.
2. **Structured-data prong:** workbench additions (findings panel, stance stripe, lean-history, convergence replay) render MORE structured data in a MORE accessible format, which compounds prong 1.

Prong 1 already fires against Python `cmd_peer_web` today — :8787 is live. The rewrite does not meaningfully worsen prong 1 in either direction:

- **Sub-prong 1a (per-port aggregation):** today's default is "one server per pair-key = N servers for N rooms"; tomorrow's default is "one server for N rooms". Aggregate data-reach across all servers is identical. A localhost attacker today could enumerate :8787, :8788, :8789, ... to find all running rooms; tomorrow they'd hit :8787 once. One port is arguably easier to firewall + audit than N ports. Net safety: wash or slight improvement.
- **Sub-prong 1b (operator control):** for operators who want narrow single-room exposure, `--only-pair-key K` replicates today's `--pair-key` semantics exactly. Preserved in §2 Item 5, gated in §7.

Prong 2 requires new structured data shapes on :8787. Rewrite ships none — every endpoint (`/api/scope`, `/api/pairs`, `/api/rooms`, `/api/messages`, `POST /api/send`) is a byte-identical port of today's Python handler. No findings table, no stance stripe, no lean-history. Turn-cap warning + stale-session filter (§2 Item 7) operate on data already in today's `/pairs.json` / `/rooms.json` payload; they're rendering-layer polish, not new extraction.

Therefore: neither prong is amplified. Gate does not trip. Rewrite can ship before v3.1 write-time redaction lands.

**What would re-open this:**
- If implementation discovers that achieving parity requires exposing a new structured shape (e.g., a materialized-unread-per-member table that didn't exist on the client before), drafter's close is void and Q2 reverts to reviewer-routed.
- If mediator absorbs v3 and disagrees with the prong analysis, mediator's ruling replaces drafter's close.
- If owner revokes delegation, same effect.

Drafter confidence: medium-high. The prong analysis is defensible against the brainstorm text as written; the weak spot is sub-prong 1a (aggregation-by-port), where reasonable reviewers could argue that surface-unification is not obviously safer than surface-dispersal. Mitigation flag shipped regardless.
