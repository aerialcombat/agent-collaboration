# v3.0 Scoping — Go Rewrite, Cloud-READY Not Cloud-YET

**Status:** scoping doc, mediated convergence (2 peer agents + human owner) · **Owner:** DJ · **Mediator:** Claude · **Peers:** alpha (claude), gamma (claude) · **Session:** dusty-island-8114

## Changelog

- **v3.0** — scoping pass. Convergence reached across 9 mediated rounds on language pick (Go), storage trajectory (SQLite today → Postgres when cloud ships), protocol artifact (protobuf v1, exercised on Unix socket from day one), and a 4-week implementation plan with a 2-week buffer.

## Goal

Rewrite the hot-path of agent-collaboration in Go **without deploying cloud infrastructure today**, while laying down the interfaces, schema columns, protocol artifacts, and env-var shapes that make cloud activation a composition (new implementations behind existing interfaces), not a rewrite.

**Primary user-visible win (Week 1):** hook latency drops ~80-150ms → ~13ms p99 (post-cold fresh-exec), ~9ms p50. Measured via `tests/hook-latency-exec.sh`. User win is 6-11× over Python baseline; sub-perceptible in practice.

**Primary structural win (Weeks 1-4):** the protocol spec is carrying real Unix-socket traffic before cloud ever ships, so "re-implementable in Elixir at scale" is operationally tested — not aspirational.

## What exists today

`scripts/peer-inbox-db.py` (3,328 LOC god file) + `scripts/peer-inbox-channel.py` (728 LOC) + `hooks/peer-inbox-inject.sh` (68 LOC) + ~1.7K LOC smoke suite. Stdlib-only Python, SQLite at `~/.agent-collab/sessions.db`, Unix sockets at `$TMPDIR/peer-inbox-*.sock`. Zero pip deps. v2.1 room-dynamics work landed; group chats + reply tool working.

Known pain:
- **Hot path:** `hooks/peer-inbox-inject.sh` pays 2× Python interpreter cold-start per prompt → ~80-500ms wasted when inbox is empty. No empty-inbox fast-path.
- **Dead code:** `poc/channels/peer-inbox-channel.py` (402 LOC) superseded by `scripts/peer-inbox-channel.py`. Delete.
- **DRY violations:** duplicated SQL in `peer-inbox-db.py` at `:1007`/`:1792`, `:1101`/`:1103`, `:1703-1722`; duplicated send/broadcast body-read at `:1330-1344` + `:1456-1470`; session-key env tuple hardcoded twice (Python `:83-88` + bash `agent-collab:92-99`).
- **Single-machine by construction:** Unix sockets + SQLite at `Path.home()`, woven through with no transport abstraction. Cross-machine requires redesign, not swap.

## Non-goals (explicit out-of-scope for v3.0)

- **No cloud deployment.** No managed Postgres, no auth service, no always-on daemon, no admin dashboard rewrite.
- **No OIDC flow / short-lived JWTs / keychain integration** today. Reserve the schema slots; defer the machinery.
- **No gRPC server.** Protocol is wire-encoded over Unix socket; the server half ships in v2.
- **No full Python→Go port.** Long-tail admin commands (`web`, `watch`, `replay`, `inspect`) stay in Python until they organically change.
- **No Rust / Elixir.** Decisively picked against for this scope (see "Language decision" below).

## Locked constraints

1. **Language: Go.**
2. **Storage v1: SQLite (unchanged). v2 trajectory: Postgres.** Schema written in Postgres-portable SQL today; CI verifies every migration applies cleanly against a `postgres:16` container.
3. **Protocol artifact: protobuf 3** at `docs/protocol/v1.proto`, versioned in repo. Used as the wire format on the Unix socket from Week 1 — not paper-only.
4. **Mode: hybrid.** Single-machine fast path is preserved at sub-10ms. Cloud is additive when it ships.
5. **Coexistence: Python survives for long-tail commands.** Shared SQLite + shared dirty-marker + protobuf contract tests keep the boundary safe.

## Language decision (summary of rounds 3-4)

- **Hook cold-start <10ms target eliminates Elixir** decisively — BEAM startup is 200-1000ms.
- **Rust's wins are below the user-perceptibility threshold** for a workload that's "open SQLite, run one UPDATE, write output." Learning cost (2-3 months) is not justified.
- **Go fits:** pure-Go SQLite (`modernc.org/sqlite`, no CGo) preserves single-binary distribution. `net/http` + `html/template` maps 1:1 to `cmd_peer_web` (`scripts/peer-inbox-db.py:1829`, 453 LOC) when that dashboard is eventually ported. Owner already runs Go in adjacent infrastructure.
- **Reconsider Elixir in 2-3 years** if the agent-infrastructure vision grows into thousands of always-on sessions with supervision-tree semantics. The protobuf spec is what preserves that optionality.

## Design spec v1 (converged Round 7)

Per area: **(a)** what we build today / **(b)** what we design for but defer / **(c)** what it becomes when cloud ships.

### 1. Identity & auth

- **(a)** `pair_key` + `session_key` continue as auth primitives. Schema adds nullable `user_id TEXT` + `workspace_id TEXT DEFAULT 'default'` columns. `Auth` interface in Go with `LocalTrust` implementation (returns `trusted=true`).
- **(b)** OIDC device-flow login, JWT issuance, refresh-token rotation, OS-keychain storage, Postgres RLS on `workspace_id`.
- **(c)** `OIDCBearer` impl slots in behind `Auth` interface. Nullable columns promoted to `NOT NULL` via migration. `pair_key` demotes to "workspace invite token."

### 2. Storage tier

- **(a)** Local SQLite only, accessed via `Store` interface (`SQLiteLocal` impl). Schema written in Postgres-portable SQL: no ROWID tricks, no `WITHOUT ROWID`, no `AUTOINCREMENT` (use `INTEGER PRIMARY KEY` which maps 1:1 to Postgres `BIGINT GENERATED ALWAYS AS IDENTITY`), no SQLite-only functions on the hot path. Migrations via `goose`. CI verifies every migration applies against `postgres:16`.
- **(b)** Postgres-specific primitives (LISTEN/NOTIFY, RLS, JSONB), connection pooling via pgbouncer, a subscription daemon mirroring cloud messages to local SQLite, write-through with async upstream flush, outbox drain logic.
- **(c)** `PostgresRemote` impl added; composite `SQLiteLocal + PostgresRemote` store. Local SQLite demotes to read-cache + outbox. Daemon arbitrates sync.

### 3. Transport

- **(a)** Unix socket only, via `Transport` interface (`UnixSocketTransport` impl). **All wire bytes are protobuf-encoded `Envelope` — even locally.** This exercises the spec under real traffic from day one.
- **(b)** gRPC server-streaming for cloud push, TLS, reconnect/backoff, SSE for the admin dashboard.
- **(c)** `GRPCClient` + `GRPCServer` impls added. Daemon holds streaming RPC per workspace. Unix socket stays as the same-machine fast path; locality router selects `UnixSocket` vs `GRPC` above the `Transport` interface.

### 4. Hook contract

- **(a)** Go hook binary (~200 LOC) ships in Week 1. Reads local SQLite only via `UPDATE ... RETURNING`. Bash `mtime`-marker fast-path short-circuits empty-inbox invocations at ~1ms. **<15ms p99 post-cold fresh-exec** budget when inbox has messages, measured via `tests/hook-latency-exec.sh`. Worth shipping today for the cold-start win alone, independent of cloud.
- **(b)** Daemon-populated local cache, cross-machine locality routing on send side (`machine_id` check), cloud-reachability degradation modes.
- **(c)** **Hook contract unchanged.** Daemon mirrors remote messages into local SQLite and touches the dirty-marker. Hook never learns cloud exists. `machine_id` column on sessions becomes load-bearing for routing in the daemon.

### 5. Delivery semantics

- **(a)** Per-pair FIFO via server-assigned `seq_id` (SQLite `INTEGER PRIMARY KEY`, maps 1:1 to Postgres `GENERATED ALWAYS AS IDENTITY`). **`idempotency_key TEXT` column in schema from day one** with `UNIQUE (workspace_id, idempotency_key)`. Client code populates the key on every send even though local delivery is effectively exactly-once today. Empty `outbox` table created but unused logic.
- **(b)** Replay-on-reconnect logic in the daemon, cross-machine dedup via Postgres UNIQUE.
- **(c)** Daemon drains outbox to cloud on reconnect. Postgres UNIQUE absorbs duplicate sends. Caller semantics unchanged.

**Rejected:** exactly-once (distributed-consensus overhead not worth the marginal value); Lamport/vector clocks (per-pair FIFO covers the real ordering need; cross-pair ordering is irrelevant for debate semantics).

### 6. Wire protocol envelope

- **(a)** **`.proto v1` written at `docs/protocol/v1.proto` in Week 1, versioned in repo.** Shape:
  ```protobuf
  message Envelope {
    uint32 version = 1;
    string type = 2;
    google.protobuf.Any payload = 3;
    AuthContext auth = 4;
    string idempotency_key = 5;
    uint64 client_seq = 6;
  }
  ```
  Versioning rules: add-only fields, reserved numbers for deprecations, capability-negotiation handshake stubbed (`Hello{version, supported_types[]}` first frame). Generated Go bindings are the only way transport moves bytes.
- **(b)** gRPC service definitions, streaming RPC shapes, auth-context population beyond `LocalTrust`.
- **(c)** Same envelope, additively extended for workspace/auth ops. Existing Unix-socket traffic keeps using the envelope unchanged. **No wire-breaking change on cloud ship** — the whole point of writing the .proto now.

**Rejected:** `oneof` for payload (breaks forward-compat on exhaustive-switch clients); ad-hoc JSON (chose protobuf for codegen + portability across Go/Elixir/Rust).

### 7. Observability

- **(a)** Structured JSON logs via `log/slog` (stdlib, Go 1.21+) to stderr. Fields: `ts, pair_key, session, event, operation, duration_ms, latency_ms, result`. Basic `/metrics` Prom endpoint on any long-running component: `hook_latency_histogram`, `send_count`, `receive_count`, `unix_socket_error_count`.
- **(b)** `outbox_depth`, `stream_reconnect_count`, per-workspace dashboards, OTel tracing with W3C context propagation.
- **(c)** Deferred metrics come online when the daemon becomes network-backed. OTel added only if a specific bug/capacity question demands it.

**Rejected today:** OTel tracing, remote log aggregation, logging message bodies (privacy + noise).

### 8. CLI command shape

- **(a)** Existing command surface unchanged. New env vars introduced as **no-ops today**: `AGENT_COLLAB_SERVER` (unset = local-only), `AGENT_COLLAB_WORKSPACE` (defaults `"default"`), `AGENT_COLLAB_OFFLINE`, `--cloud=off` flag reserved. Python CLI continues against the same SQLite + dirty-marker + schema; Go hot path + thin Go CLI subset coexist.
- **(b)** `agent-collab login/logout/whoami`, `workspace create|invite|join|leave|list`, `agent-collab daemon` subcommand, offline-mode surfacing, outbox-depth in `status`.
- **(c)** Deferred subcommands ship. `pair_key` commands alias to workspace-invite semantics during a deprecation window. `--cloud=on|auto` flag becomes functional.

### Residual micro-delta (owner's call at implementation time)

Alpha proposed `google.protobuf.Any` for payload with a typed registry (more type-safe). Gamma proposed `bytes payload` (simpler). Both avoid `oneof`. Either works; pick at implementation time.

### Week 1 — Foundation (three parallel tracks)

**Track A — Hook binary (user-visible ROI):**
- `go/cmd/hook/main.go` reads via `Store` interface (minimal surface: `ReadUnread`, `MarkRead`, `Stat`), emits JSON-compatible output, honors the existing 4KB cap. Fail-open on any error.
- `go/pkg/store/sqlite/` — Postgres-portable SQL; `modernc.org/sqlite` driver (pure Go, no CGo).
- `hooks/peer-inbox-inject.sh` stats `~/.agent-collab/inbox-dirty` (atomic `touch` on send by both Python and Go). Clean → exit ~1ms. Dirty → invoke Go binary. `AGENT_COLLAB_FORCE_PY=1` falls back to Python for per-session rollback.
- `go/cmd/hook/hook_bench_test.go` — in-process bench (amortizes driver registration; useful as a regression check, not as the DoD surface).
- `tests/hook-latency-exec.sh` — **shell wall-clock bench, fresh `exec` per iteration. This is the DoD evidence surface. Target: p99 <15ms post-cold.**

**Track B — Protocol + interfaces:**
- `docs/protocol/v1.proto` — `Envelope` + versioning rules doc at `docs/protocol/v1.md`.
- `scripts/build-proto.sh` regenerates Go bindings; bindings committed to `go/pkg/protocol/`.
- `go/pkg/auth/localtrust/` (always-true impl); `go/pkg/transport/transport.go` interface declared, no impl yet.

**Track C — Schema + CI:**
- `migrations/0001_reserve_v2_columns.sql` adds nullable `user_id`, `workspace_id DEFAULT 'default'`, `idempotency_key`, `client_seq` to messages; creates empty `outbox` table; adds `UNIQUE(workspace_id, idempotency_key)`; seeds a `schema_version` row at `10`.
- **Runtime assertion:** both Go and Python assert `schema_version >= 10` at startup; mismatch refuses to run with a migration hint (prevents old-binary-against-new-DB silent corruption).
- `.github/workflows/schema-portability.yml` runs migration against an empty `postgres:16` container, asserts no dialect errors.

**W1 DoD:** hook p99 <15ms post-cold fresh-exec (measured by `tests/hook-latency-exec.sh`, wired into CI green-bar), `.proto` frozen, schema additive, Python CLI unchanged, `tests/smoke.sh` green.

### Week 2 — Interfaces + thin Go CLI + protobuf wire (parallel tracks)

**Track C (Store):** refactor W1 hook to route all SQLite through `Store` interface. No direct `sqlite3.Open` outside `go/pkg/store/sqlite/`.

**Track D (Transport):** `go/pkg/transport/unixsocket/` — **all Go wire traffic marshals `Envelope` protobuf (not JSON), from day one.** The magic-byte dual-accept is for the **Python↔Python legacy path only** during `AGENT_COLLAB_WIRE_PROTO=1` rollout — Python listener sniffs and accepts both JSON (legacy) and protobuf (new) during W2-W6. Go-send is protobuf-only always. Do not let "temporary JSON" sneak into any Go path — the whole point of writing the `.proto` now is to stress-test it under real traffic before cloud ships.

**Track E (Auth):** `go/pkg/auth/localtrust/` wired through the hook and any Go component that reads auth context.

**Track F (thin Go CLI — core, not optional):** `go/cmd/agent-collab/main.go` subset: `peer send`, `peer receive`, `peer list`, `session register`. Writes `idempotency_key` + `client_seq` on every send. `scripts/agent-collab` bash dispatcher routes these 4 to Go, rest to Python. **`tests/parity.sh` diffs stdout + DB state between Python and Go for each ported command** (command-level parity, not just envelope roundtrip).

**Track G (Python protobuf bindings):** `scripts/build-proto.sh` extended for Python at `scripts/py/pb/`. `peer-inbox-channel.py` gains protobuf send path (behind the feature flag). Version-lock `protobuf` Python + `google.golang.org/protobuf` Go in their respective manifests.

**Track H (CI hardening):** `tests/protobuf-roundtrip.sh` — Python↔Go byte-for-byte equivalence. `.github/workflows/schema-portability.yml` applies migrations, validates clean apply + rollback against `postgres:16`.

**Wire-format invariant matrix (contract for `tests/interop.sh`):**

| Sender | Receiver | Expected wire format | Behavior |
|---|---|---|---|
| Go | Python | protobuf | Python decodes, field-match asserted |
| Python (with `AGENT_COLLAB_WIRE_PROTO=1`) | Go | protobuf | Go rejects JSON cleanly, accepts proto |
| Python (legacy) | Python | JSON | Still works until W6 cutover |
| Go | Go | protobuf | Proto only, no JSON fallback |

If any cell silently degrades to JSON, the spec lied — test should fail.

**W2 DoD:** Go hits SQLite only via `Store`; all Go wire bytes are protobuf; 4 core CLI commands on Go by default; parity + roundtrip + schema-portability CI all green; Python smoke test green post-migration.

**Risk stop:** any parity divergence → halt and fix before W3.

## Two-language coexistence — 4 explicit contracts

Python and Go must coexist safely during weeks 2-6 (or longer if long-tail stays Python). Four contracts govern the boundary:

1. **Shared SQLite schema.** Single `sessions.db`. `goose` migrations authoritative. **Additive-only rule during coexistence:** new columns nullable/defaulted, new tables, new indexes — **no renames, drops, or type changes.** A `schema_version` row asserted by both runtimes at startup (>=10 at v1); mismatch refuses to run with a migration hint.
2. **Dirty-marker file** at `~/.agent-collab/inbox-dirty` (or per-pair `~/.agent-collab/dirty.<pair_key>`). Both sides `touch` on every send. Hook stats for the fast-path short-circuit. Atomic at the filesystem level. Worst-case race: an extra hook invocation that reads a clean inbox and exits in <10ms.
3. **Protobuf canonical form on Unix socket** (post-W3). Contract tests verify byte-for-byte roundtrip equivalence. Pinned protobuf versions: `protobuf` Python + `google.golang.org/protobuf` Go.
4. **Bash dispatcher routes by subcommand.** `peer receive` → Go. `web` / `watch` / `replay` / `inspect` → Python. New transport commands → Go. User-visible command names stay identical; internal routing is invisible.

**Rollback:** `AGENT_COLLAB_FORCE_PY=1` per-session forces full Python path (hook + CLI).

**Failure modes:**
- Schema drift → mitigated by additive-only rule + Python-smoke-after-migration CI job.
- Protobuf version skew → mitigated by pinned libs + roundtrip contract tests.
- Dirty-marker race → `touch` is FS-atomic; worst case is a wasted <10ms hook exit.

## Cost / benefit

**Today pays (~1 week of design tax):**
- ~1 day writing `.proto` + versioning rules.
- ~2-3 days for `Store` / `Transport` / `Auth` interfaces with one impl each.
- Schema discipline — Postgres-portable SQL, reserved nullable columns.
- Keeping protobuf on the Unix socket (exercises the spec from day one).

**Today defers (zero operational cost):**
- No Postgres instance to host or back up.
- No OIDC / auth service to run.
- No persistent daemon architecture to supervise.
- No migration of existing workflows.
- No user-facing command changes.

**v2 gains:**
- Cloud ships as **composition of new implementations behind already-exercised interfaces.** `PostgresRemote Store` + `GRPCClient/Server Transport` + `OIDCBearer Auth` + `agent-collab daemon` binary. No upstream rewrites.
- Protocol spec already carries real traffic — v1.proto is production-tested before a server ever accepts it.
- Optionality preserved for Elixir if the agent-infrastructure vision scales into thousands of always-on sessions.

## Scope creep explicitly rejected

- Centralized audit log "for analytics" — no user scenario demands it; local export script suffices if ever needed.
- Cross-machine hot-reload of session state mid-turn — sounds cool, no scenario requires it.
- Cloud-stored full transcripts — not peer-inbox's job (Linear/Notion/git handle durable conversation storage better; peer-inbox is a routing layer).
- Admin dashboard port to Go in v1 — defer until someone touches `cmd_peer_web` (`scripts/peer-inbox-db.py:1829`) substantively.
- Cross-machine transport in v1 — the transport interface exists and is exercised; the gRPC implementation is v2.

## Design caveats

- **The .proto is the load-bearing artifact for future optionality.** Without a language-neutral spec exercised by real traffic, "re-implementable in Elixir at scale" is aspirational. With it, the door stays open. Do not skip the protobuf step to save a week.
- **Single-machine fast path must be preserved** — it remains 70-80% of usage even after cloud ships. "Always cloud" would regress A1/A2 latency from sub-ms back to 30-200ms RTT for no benefit.
- **Additive-only schema rule is non-negotiable during coexistence.** Rename/drop/type-change during the Python/Go overlap window breaks one runtime silently. Wait until Python is retired from a command before altering that command's schema.
- **Don't stand up Postgres "for future use."** The schema discipline earns the future migration; premature Postgres deployment pays operational cost before the scenario materializes. The user said: cloud-READY, not cloud-YET.
