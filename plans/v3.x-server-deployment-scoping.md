## 1. Goal + motivation

Port the agent-collab stack from macOS laptop to a Linux server so that:

1. **Agent collectives stay alive** when owner closes the laptop. Rooms keep going, daemon keeps supervising, autonomous work keeps progressing.
2. **Remote check-in** works from phone / non-primary devices, not tied to the machine where the agents live.
3. **Background autonomy** — agents can execute long-horizon tasks without owner-in-the-loop, with results waiting in peer-inbox when owner returns.

**Owner motivation verbatim (peer-inbox, 2026-04-21):** *"i love how agents are working together. i've experienced it twice today already, especially because i'm fullstack. and i don't want to miss it when i'm on the road. i want to be able to check in remotely. and i want things to be going even when i'm NOT around."*

**Why now.** v3.2 frontend-go-rewrite just landed (`f11dbc3`) — the Go peer-web is the natural remote-access interface. Before v3.2, remote access was a Python dashboard with single-pair-key scope; too narrow to be the "owner's cockpit" a server deployment needs. After v3.2, the UI is multi-room + safety-gated + mobile-adjacent. The stack is now server-ready from the UI side.

**Stack portability audit** (drafter walked the tree — see `v3.2-frontend-go-rewrite-scoping.md` §4 for the Go side):

- **Go binaries** — pure-Go SQLite driver (`modernc.org/sqlite`, no CGo). `GOOS=linux go build` cross-compiles; static single-file distributions.
- **Python CLI** (`scripts/peer-inbox-db.py`) — cross-platform, already uses `~/.agent-collab/`.
- **pi extension** (`extensions/peer-inbox-pi.ts`) — Node/TS. Linux-native.
- **Install scripts** (`scripts/install-global-protocol`) — cross-platform bash. Already writes `~/.claude/`, `~/.codex/`, `~/.gemini/` templates.
- **Session markers** — JSON files at `<cwd>/.agent-collab/sessions/`. Filesystem-native.
- **Socket channels** — Unix domain sockets. Linux-native.
- **Daemon** (`go/cmd/daemon`) — already designed as headless auto-reply supervisor; spawns Claude Code with `--print` / SDK mode on peer-inbox events. This is the key enabler: headless agent lifecycle already exists.

**Conclusion: ~90% port, ~10% server-shape wiring.** The 10% is what this cluster ships.

## 3. Out of scope

- **Multi-user / multi-tenant server** — single-operator assumption holds. Sharing with collaborators is a separate scope-doc.
- **Public internet exposure with auth** — Tailscale handles the "reachable from phone" axis without opening ports to the world. TLS + auth for non-Tailscale access is follow-on.
- **Push notifications to phone** — flagged in owner convo; `ntfy.sh` / web-push integration is separate cluster, multi-day, probably after first-deploy proves ergonomics.
- **Mobile native app + oozoo integration** — parked per owner: "not worried about it right now". Separate scope.
- **Horizontal scaling / multi-node** — single-box assumption.
- **CI for Linux-build / Linux-smoke** — manual for v1; automate later if first deploy has stuck paper-cuts.
- **Backup strategy beyond sessions.db copy** — single SQLite file, a `cron` daily copy to another volume is sufficient until it isn't.

## 4. Per-item impl surfaces

**Item 1 — `scripts/server-bootstrap.sh`.**
- Surface: new file, bash, idempotent (safe to re-run).
- Flow: (a) detect distro, (b) apt install deps, (c) install Go 1.25+ (via apt or download), (d) clone or assume repo at `/opt/agent-collab`, (e) `go build ./cmd/...` into `go/bin/`, (f) create system user `agent-collab` with home `/var/lib/agent-collab`, (g) symlink or copy binaries into `/usr/local/bin/`, (h) run `scripts/install-global-protocol` in `agent-collab` user's home, (i) write `.env` template at `/etc/agent-collab/env.example`, (j) install systemd units, (k) enable (not start — owner fills `.env` before first start).
- Exit cleanly on failure; prints next-step instructions (e.g., "fill /etc/agent-collab/env then `systemctl start agent-collab-daemon`").

**Item 2 — systemd units.**
- `agent-collab-daemon.service`: `ExecStart=/usr/local/bin/agent-collab-daemon`, `EnvironmentFile=/etc/agent-collab/env`, `User=agent-collab`, `Restart=on-failure`, `RestartSec=5`. Journal logging.
- `agent-collab-peer-web.service`: `ExecStart=/usr/local/bin/peer-web --port 8787`, same env + user + restart semantics. Binds 127.0.0.1 — Tailscale/SSH exposes externally.
- Both `WantedBy=multi-user.target`.
- **Deferred:** separate per-agent units. For v1, daemon spawns agents on peer-inbox events; no "agent-alpha.service" etc. until multi-persistent-agent use-case is real.

**Item 3 — secrets.**
- `/etc/agent-collab/env` (chmod 600, owned by `agent-collab` user).
- Contents: `ANTHROPIC_API_KEY`, optionally `OPENAI_API_KEY` + `GEMINI_API_KEY` for pi providers, `AGENT_COLLAB_MAX_PAIR_TURNS` override, `AGENT_COLLAB_INBOX_DB` if non-default.
- Template at `deploy/env.example`, gitignored-in-practice via not-shipping-the-real-one.

**Item 4 — runbook.**
- `docs/SERVER-DEPLOY.md` sections: (a) Prereqs (VPS minimum: 2 vCPU, 2GB RAM, 20GB disk; Ubuntu 22.04 LTS as reference), (b) First-time setup (git clone, bootstrap script, .env fill, start services), (c) Remote access (Tailscale install + `tailscale up` + bind check; SSH port-forward as fallback), (d) Adding agents (daemon auto-spawns on peer-inbox events; register session via `agent-collab session register`), (e) Upgrade flow (`git pull && go build && systemctl restart`), (f) Backup (`cp ~/.agent-collab/sessions.db /mnt/backup/`), (g) Troubleshooting (journalctl, common error patterns).

**Item 5 — smoke test.**
- `tests/server-deploy-smoke.sh`: runs inside a throwaway Ubuntu environment (Lima / Multipass on macOS, or target VPS directly). Calls `scripts/server-bootstrap.sh`, fills `.env` with test keys (env-var-injected), starts services, performs `agent-collab session register` + `agent-collab peer send`, asserts message lands in DB + daemon logs + peer-web `/api/messages` returns it.
- **Open q:** CI infra for running this. Defer — manual-only for v1.

**Item 6 — MCP spawn Linux probe.**
- Surface: on the freshly-deployed server, start a Claude Code CLI session with `claude --print "hello"` and verify peer-inbox MCP server gets spawned identically to macOS (inspect `claude mcp list` or equivalent). Captures a fixture so future deploys don't rediscover the surprise.
- If it fails, the failure mode likely requires `claude mcp add` config tweaks that we'd document in the runbook.

## 7. Open questions for challenge round

- **Q1** — distro breadth. Reference target is Ubuntu 22.04 LTS. Does the bootstrap need to also handle Debian 12, Fedora, Arch? Recommendation: Ubuntu-only for v1, document that other distros "probably work" but aren't gated.
- **Q2** — binary distribution. Ship binaries via GitHub releases + bootstrap downloads them, OR bootstrap builds from source? Source-build is simpler (no release pipeline), but requires Go on the server (~200MB). Recommendation: source-build for v1, switch to release artifacts if bootstrap time becomes a pain.
- **Q3** — persistent-agent model. Does v1 ship any "always-on" agents (a Claude Code running in a loop), or just the daemon that spawns per-event? Recommendation: daemon-only for v1. Persistent-loop agents are a separate design question (token cost + restart semantics + state reconciliation).
- **Q4** — access model. Tailscale only? Or also SSH port-forward + document both? Recommendation: Tailscale primary, SSH port-forward documented as fallback for users who don't want Tailscale.
- **Q5** — backup strategy. Just documented `cp` for v1, or ship a `cron` unit? Recommendation: documented only, add cron later if owner asks.
- **Q6** — systemd unit: user service vs system service. System service is simpler for auto-start on boot; user service needs `loginctl enable-linger`. Recommendation: system service, `User=agent-collab` directive.
