## v2.0 goal

**Group chats + reply tool.** Two concrete additions to v1.7's 2-person pair model:

1. **Group chats** — a `pair_key` room can hold 3+ participants, with fan-out broadcast, unified room view, and room-level turn cap / termination.
2. **Reply tool** — Claude sessions with channels loaded see incoming messages as `<channel>` tags and can reply via a first-class MCP tool, instead of being coached to shell out to `agent-collab peer send`.

The two compound: in a group chat, replies become natural tool calls that fan out to everyone.

## What we learned before writing this

- Channels is the sanctioned transport for our pattern. We're already using it (line 98 of `peer-inbox-channel.py` declares `capabilities.experimental["claude/channel"] = {}`). The hook is our fallback for sessions launched without the flag, not our primary.
- Claude Code's Channels reference also documents a **reply tool** pattern and **permission relay**. Reply tool is cheap to add; permission relay is deferred to v2.2+.
- Cross-machine is a v2.1 mission — the transport swap is small, but auth/identity/discovery require their own design pass. Don't sneak it into v2.0.

## v2.0 scope

Ordered dependencies:

### 1. Reply tool in the channel MCP (~0.5d)

Add a `peer_inbox_reply` MCP tool to `scripts/peer-inbox-channel.py`. When Claude calls it:
- Resolve the calling session's `(cwd, label, pair_key)` — the channel server knows its own Claude PID from the pending-channels registration; use that to find the session row.
- Accept `to` (optional — omit for broadcast) and `body`.
- Insert into `inbox` and push to every other peer's channel socket (fan-out on send side).

Make it the natural way to reply, not a command. The existing `peer send` shell path stays for non-channel sessions.

Smoke: channel-enabled Claude calls tool → message appears in other session's channel and in SQLite.

### 2. Fan-out broadcast in `peer send` (~0.5d)

Add a `peer broadcast` subcommand OR let `peer send` fan out when a pair_key is set and no `--to` is given. Decide which; both are cheap. Lean toward explicit `peer broadcast` to keep `peer send` semantics sharp.

- Resolve all other live peers in the sender's pair_key.
- Insert N inbox rows (one per peer), push to each channel socket.
- Smoke: 3-way and 4-way broadcast reaches everyone.

### 3. Unified room view in `peer web` (~0.5d)

In `--pair-key` mode today, the sidebar shows C(n,2) edge-pairs. Change to show **one row per room** with all participants' messages interleaved by timestamp.

- New pair-key-scoped endpoint: `/rooms.json` (list) and `/messages.json?pair_key=KEY` (interleaved).
- Sender pills stay color-keyed by label.
- Keep the edge view for v1.6 cwd mode (back-compat).

### 4. Room-level turn cap + termination (~1d)

Currently `peer_pairs` table is keyed on `(cwd, a_label, b_label)` — per-edge counters. For group chats, move accounting to per-room.

- New column or table: `peer_rooms (pair_key PK, turn_count, terminated_at, terminated_by)`.
- `peer send` / broadcast: bump the room counter once per message (not per recipient). Check room cap before sending.
- `[[end]]` terminates the whole room — all sessions reject further sends until `peer reset --pair-key K`.
- Migration additive; 2-person pairs still work under the new table (just a degenerate case of N=2).

### 5. Slash commands (~30min)

- `commands/peer.md` gains `broadcast <msg>` verb.
- Optional: `/peer reply` convenience that uses the reply tool when channels are loaded, `peer send` otherwise.

### 6. Smoke scenarios (~1d)

- 3-way broadcast reaches all peers.
- 4-way broadcast (stress fan-out).
- Reply tool: channel-loaded Claude calls tool, other peer sees message; verify the SQLite row too.
- Room-level turn cap: 20 messages total across the room blocks the 21st regardless of sender.
- `[[end]]` terminates the whole room; `peer reset --pair-key K` revives.
- Back-compat: 2-person pairs still pass all v1.7 scenarios.

## Design caveats

- **Reply tool is Claude-only.** Codex/Gemini don't speak MCP channel notifications; they keep `peer send` via shell. Design for graceful coexistence — don't require tool-reply anywhere.
- **Fan-out is sender-side.** N unicast pushes, no shared bus. Fine at realistic N (≤ 8). Don't prematurely introduce a bus.
- **Room model must not break 2-person pairs.** The v1.7 path is the hot path; v2.0 changes should be additive. Migrate `peer_pairs` → `peer_rooms` (or add a `pair_key` column) without breaking existing smoke.
- **Don't cascade session close with room termination.** Closing a single session should remove that label from the room; terminating the room should NOT kill the other sessions.
- **`peer send --to X` still works.** Directed sends remain a first-class verb. Broadcast is additive.
- **Channel push auth.** The channel server already gates by `X-Sender` in v1.4. Group chat doesn't change that; fan-out uses the same per-peer auth.

## What NOT to do in v2.0

- **No cross-machine.** Transport stays Unix-socket + local SQLite. v2.1.
- **No permission relay.** v2.2+ candidate.
- **No encryption at rest.** SQLite stays plaintext.
- **No message editing / deletion.** Inbox rows stay immutable.
- **No room admin roles or naming.** Pair key is the room identity.
- **No rich media.** Plain text bodies only, 8KB cap unchanged.
