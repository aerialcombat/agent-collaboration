# v2.0 Scoping

v1.7 closed the integration UX (`/agent-collab`, `/peer`, install
skill, zero-config Codex/Gemini). v2.0 picks the next scope from
the three ideas the handoff put out of scope for v1.7.

## Candidates

### A. Group chats — 3+ labels in one pair_key (~2 days)

Current design locks a pair_key to 2 labels via a unique
`(pair_key, label)` constraint, and `peer_pairs.turn_count` is
keyed on an ordered pair of labels.

What changes:
- Drop the "2 labels per pair" framing in docs.
- Turn cap semantics: move from per-edge to per-room (messages
  sent into the pair_key count toward one counter), or keep
  per-edge but compute all pairs on demand.
- `peer send` auto-infer breaks for ≥3 peers → require `--to`.
- `peer list` already scopes by pair_key — works unchanged.
- Slack-UI pair routes (`/messages.json?a=X&b=Y`) become room
  routes (`/messages.json?room=pair_key`) with sender-filtered
  views.
- `/agent-collab` join flow is unchanged.
- New slash form `/peer broadcast <msg>` for group fan-out (or
  have `/peer send` fan out when no `--to` and ≥2 peers present).

Migration: none — existing 2-label pairs stay valid; the unique
index stays in place (no dupe labels in a room). Only the docs
and a couple of helpers relax assumptions.

Risk: low. SQLite schema barely moves. Most of the work is UI
polish (web view + slash commands).

### B. Cross-machine (~2 weeks)

Replace the single SQLite file with a small HTTP service, or
sync SQLite databases across machines.

Unknowns: auth model, endpoint hosting, offline behavior, how
two machines discover each other. Likely needs a directory
service or user-owned relay.

Risk: high. Big surface area. v2 of the storage layer.

### C. NL-spawn — "spawn a frontend agent" (~1 week)

User says something to Claude; Claude opens a new terminal,
starts `claude` / `codex` / `gemini` with the right flags, and
registers it into the current pair_key.

Needs: AppleScript/osascript (macOS) or `wezterm cli` / tmux
(cross-platform) for spawning terminals; the new CLI session
needs to pick up `AGENT_COLLAB_SESSION_KEY` and pair_key from
env or a handoff file; the `/agent-collab` skill becomes the
target to automate.

Risk: medium. Platform fragmentation (macOS Terminal vs iTerm
vs tmux vs WezTerm) will bite.

## Recommendation

**Pick group chats (A).** Smallest scope with a real user win —
"product manager + frontend + backend" three-way rooms is a
common shape and the current primitives already support most of
it. The other two options either require a new storage tier or
get tangled in terminal fragmentation; pushing them out gives us
v2.1+ room to design them properly.

## Shape of the v2.0 PR (proposed)

1. Relax pair-cap accounting to per-room instead of per-edge, or
   document that turn caps are cheap enough to compute on demand
   across all sender/recipient pairs in a room.
2. `peer send` auto-infer: error clearly when ≥3 peers, list the
   candidates, require `--to`.
3. `peer list` gains a `--room` shortcut that prints label only
   (no cwd noise) — group-friendly output.
4. Web view: `pair_key`-scoped room view with all participants'
   messages interleaved. Sender pills stay color-keyed.
5. `/peer broadcast <msg>` slash helper — fans out to every peer
   in the room.
6. Smoke: 3-way and 4-way room scenario; turn cap under group
   semantics.
7. Docs: drop the "2 labels per pair" wording; add a short
   "rooms vs pairs" section.

## What stays out of v2.0

- Cross-machine. Still SQLite-on-one-host.
- NL-spawn. User still types `/agent-collab` in each terminal.
- Encryption at rest.
- Message editing / deletion.

Handoff for whoever picks this up: start at step 1 (turn cap
model). The answer to "per-room vs per-edge" changes the rest of
the PR.
