# v3.5 — Channel as URI (unified push transport)

## Context

Yesterday we proved cross-machine federation works end-to-end (orange ↔ laptop
via Tailscale, `peer-send --home-host orange`). The write path crosses HTTP
cleanly. The **read path** does not — orange writes an inbox row addressed to
`laptop-client`, but `laptop-client.channel_socket` on orange is empty
(unix paths are meaningless across hosts), so no live push happens into the
remote Claude's context. Remote agents only see messages if they explicitly
poll `GET /api/messages`. That's the "remote → local push gap" filed as bug #3
in the Apr 21 federation session.

## Options considered

See `plans/v3.5-channel-as-uri-options.md` for the long-form enumeration.
Short version of the twelve options evaluated:

| Option | Verdict |
|---|---|
| Everyone long-polls hub | Rejected — orange-hosted agents get second-class latency. |
| WebSocket / SSE / gRPC from every peer | Rejected — stateful server, reconnect complexity, overkill for N≤10 agents. |
| Message broker (NATS/MQTT/Redis) | Rejected — external dependency for 10-agent deployment. |
| SQLite logical replication | Rejected — massive dep, corruption risk. |
| P2P mesh (libp2p) | Rejected — doesn't solve NAT without a relay; becomes hub anyway. |
| **Channel-as-URI (this doc)** | **Accepted.** |

The deciding observation: **agent-collaboration's real deployment model is
Tailscale / LAN / VPN, not public internet**. Every node has a stable
routable address. The "NAT-defeating" long-poll assumption doesn't apply. When
every peer is reachable, push beats pull on every axis that matters (latency,
symmetry, code simplicity, debuggability).

## Architecture

The existing `sessions.channel_socket` column is a transport hint — "how to
async-push one message into this session's live context." Today it's a unix
socket path. Generalize it to a URI with a scheme-dispatched backend.

```
unix:///tmp/peer-inbox-joe/89772.sock     → sendOverUnixSocket (same-host)
http://100.75.236.84:8787/api/channel-push?label=X   → HTTP POST (cross-host)
https://...                                → HTTP POST w/ TLS
(empty)                                    → no push, receiver must poll
```

### Write path dispatcher

`go/pkg/store/sqlite/send.go` currently calls `sendOverUnixSocket(recvSock, payload)`
directly. Introduce `pushChannel(channelURI, payload) (int, error)`:

```go
func pushChannel(uri string, payload map[string]any) (int, error) {
    if uri == "" {
        return 0, errNoChannel // caller surfaces as push_status=no-channel
    }
    u, err := url.Parse(uri)
    if err != nil || u.Scheme == "" {
        // Bare path — legacy unix socket format. Backward compat.
        return sendOverUnixSocket(uri, payload)
    }
    switch u.Scheme {
    case "unix":
        return sendOverUnixSocket(u.Path, payload)
    case "http", "https":
        return sendOverHTTPChannelPush(uri, payload)
    default:
        return 0, fmt.Errorf("unsupported channel scheme: %s", u.Scheme)
    }
}
```

### Cross-host push endpoint

Every host's peer-web grows one endpoint:

```
POST /api/channel-push?label=<label>&pair_key=<key>
Authorization: Bearer <push-token>    (deferred to v3.5.1; PoC is unauthenticated)
Content-Type: application/json

{"from": "...", "body": "...", "meta": {...}}
```

Handler:
1. Look up the receiver's local session: `SELECT channel_socket FROM sessions
   WHERE label = ? AND pair_key = ?`.
2. If found and `channel_socket` is a unix path → forward via sendOverUnixSocket.
3. Return HTTP 200 on forward success, 502 on forward failure, 404 if no local
   session.

The effect: peer-web becomes a **per-host channel gateway**. Orange POSTs to
laptop's peer-web, laptop's peer-web forwards to laptop's local unix socket.

### session-register change

```
--channel-uri <uri>    explicit channel URI; overrides auto-detect
```

When `--channel-uri` is omitted, keep the existing behavior (walk process tree
for pending-channel socket; store as unix path). Auto-detection of a Tailscale
URL is deferred to v3.5.1; for now, operators registering a remote-addressable
agent pass `--channel-uri http://100.x.x.x:8787/api/channel-push?label=X`
explicitly.

### Federated registration

Today: registering a session with `--home-host orange` only seeds
`peer_rooms.home_host` on the local DB. It does NOT tell orange anything about
this session. Orange learns about the session only when someone manually
registers it there (what we did yesterday to get the auth token).

For this phase, the cross-host registration step stays manual. A future v3.5.1
item: `session register --home-host X` POSTs to `X`'s new `/api/register-peer`
endpoint to auto-create the sessions row with channel_uri set.

## Schema

**Zero migration.** The `channel_socket` column already holds free-form text.
We change its interpretation at read time (URI parser with legacy fallback),
not its storage. Existing rows with bare paths parse as unix-scheme URIs.

If later we want to be explicit, rename to `channel_uri` in a 0008 migration.
Not required for this phase.

## Test matrix

Three scenarios gate the PoC:

### S1 — Local-only (agents on same host)

Setup: two Claude sessions on laptop, same pair_key, default channel URIs
(unix sockets).

- `a` registers with unix socket URI (auto-detected).
- `b` registers with unix socket URI.
- `a` sends `"hi b"` → push_status="pushed", b's Claude context receives.

**Invariant:** behavior identical to pre-v3.5. No code paths with URL parsing
take effect. Pure backward-compat check.

### S2 — Server-only (agents on orange)

Setup: two Claude sessions on orange, same pair_key, default channel URIs
(unix sockets on orange).

- `c` registers with unix socket URI.
- `d` registers with unix socket URI.
- `c` sends `"hi d"` → push_status="pushed", d's Claude context receives.

**Invariant:** same as S1 but on orange. Confirms no regressions for
server-hosted agents. **Orange-hosted agents must remain first-class** —
never go through HTTP to reach a same-host peer.

### S3 — Mixed (laptop + orange)

Setup: one Claude session per host, federated room with home_host=orange.

- `server-agent` on orange: channel URI = `unix:///tmp/peer-inbox-joe/<pid>.sock`
  (auto-detected).
- `laptop-client` on orange (the row that orange uses to push TO laptop):
  channel URI = `http://100.75.236.84:8787/api/channel-push?label=laptop-client&pair_key=<key>`
  (manually set via `session-register --channel-uri ...`).
- `laptop-client` on laptop (its own row): channel URI = unix socket
  (auto-detected).

Flows:
- **laptop → orange** (already works today): laptop POSTs `/api/send` to
  orange; orange writes inbox row; orange's push dispatcher uses
  server-agent's unix URI → unix socket push → orange's Claude sees it. ✓
- **orange → laptop** (new via v3.5): orange writes inbox row for laptop-client;
  push dispatcher parses channel URI as `http://...`; HTTP POSTs payload to
  laptop's `/api/channel-push`; laptop's peer-web looks up its own
  `laptop-client.channel_socket` (local unix path); forwards to the unix
  socket; laptop's Claude context receives.

**Invariant:** both directions reach live Claude contexts with sub-second
latency (one HTTP round-trip, no poll). Orange's own agents still use unix
(never HTTP to themselves). Laptop's own local-to-local sends still use unix.
HTTP is only for cross-host hops.

## PoC steps (verification)

1. **Implement pushChannel dispatcher** in `go/pkg/store/sqlite/send.go` +
   `system_event.go`. Drop-in replacement for `sendOverUnixSocket` calls.
2. **Implement /api/channel-push handler** in
   `go/cmd/peer-web/server/channel_push.go`. Auth deferred — accept any push
   with a valid `label+pair_key` pointing to a locally-registered session.
3. **Add --channel-uri flag to session-register**. Default behavior unchanged.
4. **Run S1 + S2 + S3 by hand.** S3 needs orange's laptop-client row
   explicitly set to the laptop's http://... URI (manual SQL for PoC).
5. **Commit + push + document.**

## Migration plan

Because the schema is unchanged, there's no migration in the DB sense.

**Rollout:**
- v3.5 lands as a single commit: dispatcher + endpoint + flag.
- Existing unix-path values in `channel_socket` keep working (legacy fallback
  in the parser).
- New deployments with `--channel-uri http://...` start working cross-host
  immediately.

**Rollback:**
- `git revert` the commit. Unix paths still parse. No DB rollback needed.

## Deferred to v3.5.1

- **Auth on /api/channel-push.** PoC trusts any caller that can reach the
  endpoint (Tailscale/LAN/VPN-scoped). Production wants per-peer bearer tokens,
  ideally reciprocal to the existing `auth_token` pattern.
- **Auto-detection of reachable URL.** At session-register time, probe:
  - `AGENT_COLLAB_SELF_URL` env var,
  - `tailscale status --json` (parse own Tailscale IP),
  - `peer-web` listen address from the local process table,
  - compose a URL and store as channel_uri without operator intervention.
- **`/api/register-peer` on home_host.** Eliminates the manual orange-side
  session-register step. Triggered by `session register --home-host X`.
- **TLS (https:// channel URIs).** Works today if operator configures it on
  peer-web; not a v3.5 deliverable.
- **Long-poll fallback daemon.** Useful for truly unreachable agents (public
  internet, no Tailscale). Implemented in a later phase if demand exists.

## Rationale recap

| Axis | Channel-as-URI | Long-poll alternative |
|---|---|---|
| Latency (cross-host) | ~1 HTTP RTT | poll interval + 1 RTT |
| Latency (same-host) | kernel (unix socket) | kernel OR wasted HTTP loopback |
| Orange-agents-first-class | ✓ | ✗ (if uniform polling) |
| Code surface | 1 endpoint, 1 dispatcher | daemon + cursor + long-poll |
| Auth story | extends existing bearer | new token scheme or no auth |
| Debuggability | curl the endpoint | replay poll state |
| NAT-hostile networks | ✗ (needs reachable peer) | ✓ |
| Our actual deployment | ✓ (Tailscale) | overkill |
