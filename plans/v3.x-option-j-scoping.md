## Why ship Option J solo

W1 round 2 showed operational cost of push-asymmetry — idle-birch went stale during the round, gemini-peer consistently `(no-channel)` on every broadcast. Subsequent rounds on the broader v3.1 bundle would repeat that drag. Installing hooks on Codex + Gemini fixes the collaboration infrastructure first, then every downstream review round benefits. Owner's redirect sequences this correctly: infra-fix before feature-work.

## Deliverables

### Coder lane

**1. Unified hook script** — rewrite `hooks/peer-inbox-inject.sh` in place with stdin-schema detection. ONE script, NOT three per-CLI files (alpha C1 ruling in scaffold §8.2 — avoids the B1/B8 drift class that bit the UTF-8 bytes parity in W1).

Structure (adapter-at-top → shared core):

```
hooks/peer-inbox-inject.sh
├── detect CLI from stdin JSON shape (or from HOOK_PROVENANCE env)
│     • session_id + cwd    → Claude Code UserPromptSubmit
│     • prompt + turn_id    → Codex UserPromptSubmit
│     • prompt + history    → Gemini BeforeAgent
├── extract normalized (session_key, cwd) per CLI
└── shared core: dirty-marker check → Go/Python hot path → emit
    hookSpecificOutput envelope (identical across CLIs per R1/R2/R4a).
```

**2. Stdin fixtures** — `tests/fixtures/hook-stdin/{claude,codex,gemini}.json`. Canonical JSON input per CLI, captured from live hook invocation (preferred) or constructed from doc source if live-capture not available. These become the source-of-truth for each CLI's stdin schema — pinning them here means the scripts can be schema-checked against stable artifacts rather than chasing upstream doc drift.

**3. Install-script extension** — `scripts/install-global-protocol`. Currently writes `~/.claude/CLAUDE.md` only. Extend to also register hook configs:
- `~/.claude/hooks/` (existing target)
- `~/.codex/hooks.json` per R1 gamma (hooks co-located with active config layers)
- `~/.gemini/settings.json` per R4a gemini-peer (`.gemini/settings.json` at user scope, hooks section)

Detect installed CLIs via `command -v` and skip cleanly when absent. Failure to register one CLI's hook should not block the others.

**4. Go hook binary** — verify `go/cmd/hook/main.go` still works when invoked from Codex/Gemini hook contexts. The Go fast path reads env vars `AGENT_COLLAB_SESSION_KEY / CLAUDE_SESSION_ID / CODEX_SESSION_ID / GEMINI_SESSION_ID` (per `sessionKeyEnvCandidates` in `go/cmd/hook/main.go:76-81`), so the env-based identity discovery is already CLI-agnostic. Confirm during implementation that cwd-resolution via argv[1] still works when the bash hook passes the CLI's reported cwd.

**5. Docs** — two updates:
- `docs/PEER-INBOX-GUIDE.md` per-runtime cheatsheet: Codex + Gemini auto-inject columns flip from "Not yet" → "Yes." Describe the one-command install via `scripts/install-global-protocol` picking up any installed CLI.
- `docs/PEER-INBOX-ARCHITECTURE.md` (if it carries runtime topology): note three-way hook parity as current architecture, reference this scoping doc.

### Test-engineer lane

**Two deliverables per owner-expanded scope (2026-04-18):**

**1. `tests/hook-parity.sh` — CI regression gate** (shape-2 per test-engineer's Track C push):

```
for each cli in {claude, codex, gemini}:
  input = tests/fixtures/hook-stdin/$cli.json
  output = stdin-to-hook($input)
  assert output.hookSpecificOutput.hookEventName matches $cli's event name
  assert output.hookSpecificOutput.additionalContext matches the canonical
         <peer-inbox> block format produced by go/cmd/hook/main.go::formatHookBlock
  assert byte-identical additionalContext across CLIs given identical inbox state
```

Fallback to shape 3 (contract-shape parity without CLI-exact fixture accuracy) only if exact schemas prove hard to pin. Authored against coder-produced fixtures.

**2. End-to-end flow validation — owner-supervised one-off** (shape-1, upgraded from "v3.2+ optional" to "required for Option J closure" per mediator's owner-directive):

Protocol landing at `docs/HOOK-PARITY-VALIDATION.md` alongside the `tests/hook-parity.sh` commit. Probe sketch from test-engineer:
- **Probe A (Claude → Codex):** Claude broadcasts a known-text probe; verify it surfaces in Codex's injected context on next prompt.
- **Probe B (Claude → Gemini):** same shape; verifies R4a gemini-peer's "fully silent" → "surfacing" flip empirically.
- **Probe C (Codex → Gemini, Gemini → Codex):** non-Claude bidirectional sanity; catches stdin-schema mismatches the fixture-based test can't detect.
- **Probe D (idempotency):** verify dedup via mtime-marker + ack-file behavior behaves identically on all three CLIs.

Owner spins the non-Claude terminals; test-engineer drives the probes + verifies receipt. Validation is a smoke test at ship-time, not a persistent CI regression gate — the shape-2 parity test holds that role.

**Why both shapes:** shape-2 validates our adapter code against canonical schemas. Shape-1 validates that Codex/Gemini runtimes actually honor the `additionalContext` injection as documented. R4a gemini-peer's "fully silent" first-hand observation is the baseline this validation proves we've cleared.
