## Question

Can IDE-embedded AI coding tools (Cursor, Windsurf, Cline) participate in a peer-inbox room as Topic 3 target peers? Three sub-questions per tool:

- (a) **Headless invocation** — comparable to `codex exec` / `gemini -p`?
- (b) **Pre-turn / mid-turn injection mechanism** — comparable to Claude UserPromptSubmit, Codex UserPromptSubmit (behind `features.codex_hooks`), Gemini BeforeAgent?
- (c) **MCP server support** — viable as polling-workaround if no native hook?

## Findings

### Cursor

- **(a) Headless:** Yes. `cursor-agent -p "prompt" --output-format json [--model ...]`; auth via `CURSOR_API_KEY`. Sources: cursor.com/docs/cli/headless, cursor.com/blog/cli. Known issue: `-p` mode has reports of hanging and not releasing the terminal (forum.cursor.com/t/cursor-agent-p-print-headless-mode-hangs-indefinitely-and-never-returns/150246).
- **(b) Injection hook:** Cursor Hooks (v1.7+, Oct 2025). `sessionStart` accepts `additional_context` in its JSON output and injects into the conversation's initial system context. `beforeSubmitPrompt` fires after user submits but per docs/forum **cannot inject context** — only block/continue. Sources: cursor.com/docs/hooks, cursor.com/blog/hooks-partners, forum.cursor.com/t/hooks-allow-beforesubmitprompt-hook-to-inject-additional-context/150707.
- **(c) MCP support:** Yes, first-class MCP client. Sources: cursor.com/docs, semgrep.dev/blog/2025/cursor-hooks-mcp-server.
- **Peer-inbox viability:** Workable with caveat. `sessionStart` covers first-turn injection only; per-turn injection equivalent to Claude's UserPromptSubmit is absent. MCP-polled inbox fills the mid-session gap. Hanging `-p` mode is a daemon-mode risk (confirms idle-birch's lease/requeue requirement).

### Windsurf

- **(a) Headless:** No official CLI. Third-party `wsc` wrapper (github.com/staronelabs/windsurf-cli) and Docker+Xvfb harness (github.com/pfcoperez/windsurfinabox) exist but drive the GUI, not a first-class `--print` mode.
- **(b) Injection hook:** Cascade Hooks, including `pre_user_prompt` (plus `pre_read_code`, `pre_write_code`, `pre_run_command`, `pre_mcp_tool_use` and post-variants). Pre-hooks receive JSON on stdin and can block with exit 2; docs emphasize **block/log rather than context injection** into the prompt. Sources: docs.windsurf.com/windsurf/cascade/hooks, windsurf.com/changelog.
- **(c) MCP support:** Yes, documented MCP client incl. remote MCP + OAuth. Sources: docs.windsurf.com/windsurf/cascade, natoma.ai/blog/how-to-enabling-mcp-in-windsurf.
- **Peer-inbox viability:** Weak. No official headless CLI + `pre_user_prompt` is documented as block/audit, not inject. MCP-polled inbox is the only credible path, and that requires the user to prompt the agent to call the tool — no auto-wake.

### Cline

- **(a) Headless:** Yes. Cline CLI 2.0 ships `cline "task"` with `--yolo`/`--no-interactive` for autonomous runs and `--json` for structured streaming; CLI shares the extension core engine via gRPC. Sources: docs.cline.bot/getting-started/installing-cline, cline.bot/blog/cline-v3-36-hooks, github.com/cline/cline/discussions/2622.
- **(b) Injection hook:** Hooks system (v3.36, PR #6440) **modeled on Claude Code spec**. Includes `UserPromptSubmit` plus `TaskStart/Resume/Cancel/Complete`, `PreToolUse`, `PostToolUse`, `PreCompact`. Returns `{cancel, contextModification, errorMessage}` — `contextModification` injects text into the conversation before the model sees the turn. Sources: docs.cline.bot/customization/hooks, cline.bot/blog/cline-v3-36-hooks, github.com/cline/cline/pull/6440.
- **(c) MCP support:** Yes, native MCP client with marketplace and remote (Streamable HTTP / SSE) servers.
- **Peer-inbox viability:** Strong. Semantics map 1:1 to Claude's `UserPromptSubmit` + headless CLI + MCP. The existing `hooks/peer-inbox-inject.sh` unified script (Option J) should port with minimal changes — just add a fourth branch for Cline's hook-event-name convention.

## Comparative summary

**Cline is the clearest Topic 3 target peer candidate.** `UserPromptSubmit`+`contextModification` is a direct analogue of the existing baseline-trio turn-start injection, and `cline --yolo --json` gives a Codex-style exec path. The same injector hook + SQLite inbox design ports directly.

**Cursor is second**, with an asterisk. Headless via `cursor-agent -p` works (modulo flaky exit), but only `sessionStart` injects context — so per-turn peer-inbox replay falls back to MCP-polled inbox tool. That changes the delivery contract: MCP-polling requires the agent to opt-in every turn (via skill instructions telling it to call `inbox_receive`), rather than hook-driven automatic injection. Hybrid mode: first-turn catchup via `sessionStart`, per-turn via MCP tool.

**Windsurf is the weakest fit.** No first-party headless CLI, and `pre_user_prompt` is block/audit rather than injection. MCP-as-workaround is possible but requires both (a) a custom wrapper to drive the GUI headlessly and (b) MCP-polling-via-skill-instructions. Not a Topic 3 v0 candidate; revisit if Codeium ships a proper `windsurf -p` equivalent.

## Implications for Topic 3 scoping

1. **Worker-shape candidate list expands from 3 → 4.** Cline joins Claude / Codex / Gemini as a viable baseline-trio-equivalent. Reason's orchestrator/worker matrix (O1–O5 × worker-shape) now has a 4th column, with Cline slotting into the same row as Claude (hook-driven turn-start injection + headless exec).

2. **Cursor introduces a new worker-shape variant:** "partial-hook (first-turn only) + MCP-polled (per-turn)." This is a hybrid shape worth naming explicitly in the matrix — it's architecturally distinct from either full-hook (Cline/Claude) or full-polling (Windsurf). The 9-item convergent scope may need a 10th item covering hybrid delivery semantics.

3. **`cursor-agent -p` hanging issue reinforces lease/requeue.** Independent confirmation of idle-birch's concern: a daemon spawning `cursor-agent -p` that hangs indefinitely loses both the work AND shows as live. The lease/requeue-or-outbox recovery semantic is load-bearing not just for crash, but also for hang — which is harder to detect than crash.

4. **Windsurf's `pre_user_prompt` block-only contract is a datapoint** for Topic 3 scoping of the content-stop sentinel layer: at least one peer-CLI's native hook contract is block/continue, not inject. If Topic 3 targets Windsurf via MCP-polling, the room-closed event primitive must work without hook-driven delivery — i.e. the MCP tool itself must surface `envelope.state: closed` from the payload it returns.
