# v3.11 — cards Phase 1: standing drainer + durable runs

**Status:** DRAFT v1 · **Authored:** 2026-04-26 · **Author:** claude (lead-drafting at owner request) · **Trigger:** owner conversation 2026-04-26 — "asyncly assign cards for agents to drain tickets as they are available … cross-host worker pull eventually, gradual phasing." · **Positions as:** first phase of a six-phase migration from Shape B (one-shot drainer) to a cross-host worker pool. Phases laid out in §10.

> **Scope discipline:** Phase 1 only flips the dispatch model from *wave* to *standing*, makes runs durable, and reserves schema headroom for Phases 2-6. It does **not** introduce worker registration, role-based assignment, deferred auto-promote, or federation. Those are explicitly later phases.

## 1. Goal

Make the kanban drainer asynchronous and capacity-bounded on a single host, so newly-ready cards get picked up without a fresh Start click, and so the system survives a peer-web restart without orphaning state.

**Owner constraint verbatim (2026-04-26):** *"asyncly assign cards for agents to drain tickets as they are available … cross-host worker pull would be nice eventually. but i'm ok with moving to it gradually, testing each phase."*

**Success criteria.**

1. Setting `auto_drain=true` + `max_concurrent=N` on a board causes peer-web to dispatch ready+unclaimed cards continuously, capped at `N` concurrent in-flight runs.
2. Restarting peer-web reconstructs all running drainers from durable state. In-flight worker processes are reaped (one of: re-claimed, marked `lost`, marked `cancelled`) — no orphans masquerading as live.
3. Every dispatch produces a row in `card_runs` with `pid`, `started_at`, `host`, `worker_label`, `log_path`. Every terminal transition (success / failure / cancel / lost) updates `ended_at`, `status`, `exit_code`.
4. Schema reservations for Phases 2-6 land in the same migration so later phases are pure additions, not breaking changes.

**What Phase 1 does NOT do.**

- No cancellation UI (Phase 2).
- No auto-promote race fix (Phase 3).
- No worker registration / heartbeat / matchmaker (Phase 4).
- No federation visibility or remote claim (Phases 5-6).
- No change to worker prompt, context resolver, activity timeline, or `hasOpenBlockee` heuristic.

## 2. Architectural shift

```
v3.10 (now)                          Phase 1 (target)
───────────                          ────────────────
Click Start                          set auto_drain=true on board
  │                                    │
  ▼                                    ▼
Server.drainers[pk] = goroutine      board_settings row persists
  │  (in-memory, one-shot)             │
  │                                    ▼
  ▼                                  peer-web boot: spawn ticker per
SELECT ready cards, fan out          row with auto_drain=true
all of them now                        │
  │                                    ▼
  ▼                                  every 5s tick:
processes spawn untracked              ── count running rows in card_runs
  │                                    ── if running < max_concurrent:
  ▼                                       SELECT ready+unclaimed,
goroutine exits when wave done             dispatch up to deficit
                                       ── for each spawn: write card_runs row
                                       ── monitor child; on exit, update row
                                     restart-safe: state lives in DB
```

**Key invariant introduced in Phase 1:** *the DB is the source of truth for what is running.* Drainer goroutines are pure executors; they hold no state that isn't first written to `card_runs` / `board_settings`. This invariant is what unlocks Phases 4-6 — worker registries, federation views, and remote claims all read the same tables.

## 3. Schema migration: `0011_card_runs_and_board_settings.sql`

One migration, three phases of headroom.

```sql
-- +goose Up
-- +goose StatementBegin
CREATE TABLE card_runs (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  card_id       INTEGER NOT NULL REFERENCES cards(id) ON DELETE CASCADE,
  pair_key      TEXT    NOT NULL,
  host          TEXT    NOT NULL,            -- self_host() in Phase 1; non-local in Phase 6
  worker_label  TEXT    NOT NULL,            -- free-form in Phase 1; FK to workers.label in Phase 4
  pid           INTEGER,                     -- NULL once process exits
  started_at    TEXT    NOT NULL,
  ended_at      TEXT,
  status        TEXT    NOT NULL,            -- running|completed|failed|cancelled|lost
  exit_code     INTEGER,                     -- NULL while running
  log_path      TEXT,                        -- nulled by Phase 2 sweep
  trigger       TEXT    NOT NULL DEFAULT 'manual'  -- manual|drainer|matchmaker
);
CREATE INDEX card_runs_card_id_idx   ON card_runs(card_id);
CREATE INDEX card_runs_status_idx    ON card_runs(status);
CREATE INDEX card_runs_pair_key_idx  ON card_runs(pair_key);
-- +goose StatementEnd

-- +goose StatementBegin
CREATE TABLE board_settings (
  pair_key       TEXT PRIMARY KEY,
  auto_drain     INTEGER NOT NULL DEFAULT 0,    -- bool
  max_concurrent INTEGER NOT NULL DEFAULT 3,
  auto_promote   INTEGER NOT NULL DEFAULT 0,    -- replaces ?auto_promote=1 query param
  poll_interval_secs INTEGER NOT NULL DEFAULT 5,
  updated_at     TEXT NOT NULL,
  updated_by     TEXT NOT NULL
);
-- +goose StatementEnd

-- +goose StatementBegin
ALTER TABLE cards ADD COLUMN promotion_due_at TEXT;  -- reserved for Phase 3 deferred auto-promote
-- +goose StatementEnd

-- (no desired_role column — cards.needs_role from 0009 serves the same purpose;
--  Phase 4 matchmaker will reuse it.)

-- +goose Down — irreversible (forward-only migrations are the convention)
```

**Why these reservations now.**

| Column / table        | Phase 1 use            | Future-phase use                                    |
|-----------------------|------------------------|-----------------------------------------------------|
| `card_runs.host`      | Always = `self_host()` | Phase 6: remote host running the worker             |
| `card_runs.worker_label` | Free-form (`drainer-{pid}-{ts}`) | Phase 4: FK to `workers.label` |
| `card_runs.trigger`   | `manual` or `drainer`  | Phase 4 adds `matchmaker`                           |
| `cards.needs_role`    | Ignored (already in 0009) | Phase 4: matchmaker uses for skill matching   |
| `cards.promotion_due_at` | Ignored             | Phase 3: deferred auto-promote window               |
| `board_settings.auto_promote` | Reads, replaces `?auto_promote=1` query | Phase 3: deferred timer source |

Adding these in Phase 1 costs nothing in code (Phase 1 ignores them) and turns Phases 3-6 into additive code changes with no migration churn.

## 4. Drainer rewrite

Files affected: `go/cmd/peer-web/server/board_start.go`, `run.go`, `server.go`, plus a new `go/cmd/peer-web/server/board_settings.go`.

### 4.1 Lifecycle

```
peer-web boot
  │
  ├─ load board_settings WHERE auto_drain=1
  ├─ for each row: spawn drainer goroutine (ticker)
  │
  └─ also: scan card_runs WHERE status='running' → reap
        ├─ if pid alive (kill -0): leave it (re-adopt)
        ├─ if pid dead: status='lost', ended_at=now, exit_code=-1
        └─ append card_events run_complete with note "reaped on boot"
```

```
drainer tick (every poll_interval_secs)
  │
  ├─ deficit = max_concurrent − count(card_runs WHERE pair_key=? AND status='running')
  ├─ if deficit ≤ 0: return
  ├─ candidates = SELECT cards WHERE pair_key=? AND status='todo'
  │              AND ready=1 AND NOT EXISTS (running run for this card)
  │              ORDER BY priority DESC, id ASC
  │              LIMIT deficit
  ├─ for each candidate:
  │     ├─ INSERT card_runs (status='running', pid=NULL)
  │     ├─ spawn worker (claude -p), capture pid
  │     ├─ UPDATE card_runs SET pid=? WHERE id=?
  │     ├─ append card_events run_dispatch (with run_id)
  │     └─ goroutine waits on cmd.Wait(); on exit:
  │          ├─ UPDATE card_runs SET ended_at, status, exit_code
  │          └─ append card_events run_complete (with run_id)
  └─ apply auto_promote rule (unchanged hasOpenBlockee logic; will change in Phase 3)
```

### 4.2 In-memory state shape

```go
type Drainer struct {
    pairKey   string
    cancel    context.CancelFunc   // signals graceful stop
    settings  BoardSettings        // live copy, refreshed each tick
}

type Server struct {
    drainers   map[string]*Drainer  // keyed by pair_key
    drainersMu sync.RWMutex
    // ... existing fields
}
```

Drainer goroutine reads `BoardSettings` from DB at the top of each tick. Settings updates take effect within one tick interval — no in-memory cache to invalidate.

### 4.3 What stays identical

- `buildWorkerPrompt()` — same prompt, same context resolver, same BLOCKED escape hatch.
- `claude -p --permission-mode=bypassPermissions --max-turns=40` — same invocation.
- `card_events` event kinds — `run_dispatch` and `run_complete` keep recording, gain a `run_id` field in their event payload.
- `/api/cards/{id}/run` (manual run) — keeps working, also writes a `card_runs` row with `trigger='manual'`.
- `hasOpenBlockee` auto-promote heuristic — unchanged; just gated on `board_settings.auto_promote` instead of `?auto_promote=1` query param. Race fix is Phase 3.

### 4.4 Reaper at boot

Pseudocode for the `card_runs` reap step:

```go
func reapOrphanedRuns(db *sql.DB) error {
    rows, err := db.Query(`SELECT id, pid FROM card_runs
                           WHERE status='running' AND host=?`, selfHost())
    // ... iterate
    for each row:
        if pid == 0 || !pidAlive(pid):
            UPDATE card_runs SET status='lost', ended_at=now, exit_code=-1 WHERE id=?
            INSERT card_events (kind='run_complete', payload={run_id, status:'lost'})
        else:
            // process still alive — re-adopt by spawning a Wait() goroutine
            go waitAndUpdate(pid, run_id)
}
```

`pidAlive` = `kill(pid, 0)` returns no error; `ESRCH` means dead.

## 5. New endpoints

Three additions to peer-web.

| Method | Path                                  | Purpose                                                          |
|--------|---------------------------------------|------------------------------------------------------------------|
| GET    | `/api/boards/{pair_key}/settings`     | Returns `BoardSettings` row, defaulted if absent.                |
| POST   | `/api/boards/{pair_key}/settings`     | Upsert `auto_drain`, `max_concurrent`, `auto_promote`, `poll_interval_secs`. |
| GET    | `/api/cards/{id}/runs`                | Returns `card_runs` rows for that card, newest first.            |

`POST /api/boards/{pair_key}/start` and `/stop` keep working as **temporary overrides** for backward-compat — they toggle `auto_drain` in the DB and bring drainers up/down accordingly. UI's existing Start button still works; just routes through the new state.

`GET /api/cards/{id}/runs` is consumed by the drawer in Phase 2's run-history tab. Phase 1 ships the endpoint but the drawer change is Phase 2 — that way the data is being collected before the UI exists.

## 8. Test plan

**Per the verified-vs-assumed convention,** Phase 1 ships only after these are run end-to-end. None are optional.

### 8.1 Unit / integration

- `go test ./...` clean.
- New table-driven test in `go/cmd/peer-web/server/board_drainer_test.go` covering: deficit calculation, candidate ordering, settings refresh mid-run, cancel propagation through `context.CancelFunc`.
- New test in `go/pkg/store/sqlite/runs_test.go` covering: `card_runs` insert/update lifecycle, status transitions, orphan reap behavior.

### 8.2 End-to-end (`tests/cards-phase1-drainer.sh`)

1. **Standing dispatch.** Set `auto_drain=true, max_concurrent=2` on `phase1-test`. Create 5 cards. Assert: at most 2 `card_runs` rows in `running` at any sample. After ~2 minutes, assert: 5 total `card_runs` rows, all `completed` or `failed`, none `lost`.
2. **Mid-flight restart.** Same setup. After 3 cards have started, `kill -TERM` peer-web. Restart. Assert: `card_runs` rows for the 3 in-flight cards either show `lost` (workers were children of peer-web and died with it) or `completed` (workers detached and finished). No row remains `running` with a dead pid.
3. **Late arrival.** Standing drainer with 0 ready cards. After 1 minute idle, create card. Assert: dispatched within `poll_interval_secs + 1` seconds.
4. **Capacity ceiling.** `max_concurrent=1`, create 10 cards. Assert: at no point are 2 rows `running` simultaneously.
5. **Settings update mid-drain.** With drainer running, change `max_concurrent` from 1 to 3. Assert: within one tick, in-flight count grows toward 3.
6. **Manual run still works.** With `auto_drain=false`, click Run on a single card. Assert: `card_runs` row with `trigger='manual'` is created and reaches terminal state.
7. **Schema reservation sanity.** Create a card with `desired_role='test'`. Phase 1 drainer should ignore it (treat as null). Assert: card is dispatched normally.

### 8.3 Federation parity (orange + mac)

- Run `tests/cards-phase1-drainer.sh` on mac. Then sync the new binaries to orange (same dance §sync-orange just exercised). Run the same test on orange. Both must pass independently.
- Cross-host claim is **not** in Phase 1; this is purely a "does it work standalone on each host" check.

## 9. Migration + deploy

1. Land `0011_card_runs_and_board_settings.sql` and the Go code together. Goose forward-only; no down migration.
2. On peer-web start with the new binary, `board_settings` is empty → all boards default to `auto_drain=false`. Behavior is unchanged from v3.10 until a board is opted in.
3. The existing `start-button-test` / `context-demo-board` boards continue to work with the manual ▶ Run button. Migration is opt-in per-board.
4. Orange sync: same scp + migrate dance just exercised. The reaper at boot is the new failure mode to watch — confirm no spurious `lost` rows on a cold start with no prior `card_runs` data.

**Rollback:** if Phase 1 ships and we discover a defect, revert the binaries — the old wave-shaped drainer still works against the new schema (it just won't write `card_runs` rows). The migration is forward-only but the schema additions don't break v3.10 binaries.

## 12. Open questions

1. **Worker process group.** Phase 1 drainer is a goroutine in peer-web; spawned `claude -p` workers are children. On peer-web kill, child PGID may or may not propagate SIGTERM. The reaper handles "child died with parent" cleanly; the question is whether we want to `setpgid` so children survive a peer-web restart and the reaper re-adopts them, or always die with the parent. Default proposal: **die with parent** (simpler reap semantics; in-flight work is lost but the DB knows). Owner sign-off needed.
2. **`max_concurrent` global ceiling.** Per-board cap is the obvious knob. Should there also be a peer-web-global cap (sum across all boards) to prevent a single host running 50 claudes simultaneously across 10 boards? Default proposal: **no global cap in Phase 1**; revisit in Phase 4 when worker pool model lands. Owner sign-off needed.
3. **Default `auto_drain`.** Should newly-created boards default to `auto_drain=false` (current behavior) or `true` (anti-friction)? Default proposal: **false in Phase 1** — keeps all opt-in until the model is proven. Owner sign-off needed.

Once §12 is resolved I'll send the PR.
