## 1. What is being built

An always-on OS-local daemon process (one per managed label, or one-per-workspace with per-label worker-shells — see §2.2 for the shape choice) that:

1. Watches the peer-inbox for messages addressed to its managed label(s).
2. On new messages, atomically claims them into daemon-mode receive state (new column `claimed_at`, not existing `read_at`).
3. Spawns a fresh `claude -p` / `codex exec` / `gemini -p` invocation per batch, exporting the stable session-key env var so the spawned CLI's hook resolves to the same `(cwd, label)` session row.
4. Feeds the claimed message batch to the spawned CLI via the existing UserPromptSubmit / BeforeAgent hook path (Option J).
5. Waits for the CLI to emit a completion-ack signal.
6. On ack: marks the batch `completed_at`; on ack-absence-within-TTL: requeues (reverts `claimed_at`) for the sweeper to redeliver.
7. Respects the four-layer termination stack (content-stop sentinel + `envelope.state: closed` + quiescence primitives + heartbeat).

**Scope:** Codex + Gemini reactive parity with Claude for the empirically-confirmed trio, via O4 orchestrator + W3 worker-shape + hook-delivery. Canonical JSON envelope schema with deterministic hook serialization (item 10, code-quality scope). Completion-ack with TTL-fallback recovery semantics (item 9 revision). See §7.6 of the brainstorm doc for the full narrowed-default wording.

**Out of this doc's scope:** Cline / Cursor / Windsurf (owner directive, full horizon); API-backed workers (Topic 2); heartbeat/reaper implementation (v3.1 papercut at `712121c` — daemon-mode depends on it being in place but does not ship it).

## 2. Verb surface

### 2.1 New CLI verb(s)

Two proposed new verbs on `scripts/peer-inbox-db.py` (and Go parity where the hot path crosses into Go):

- **`peer receive --daemon-mode`** — atomic claim of unread rows addressed to `self` into daemon-mode state (`claimed_at = now, completed_at IS NULL, read_at IS NULL`). Returns the claimed batch as JSON (for daemon stdout consumption) and/or as a `<peer-inbox>` hook-envelope block (for the daemon to pipe into a spawned CLI's hook context — see §5).
- **`peer receive --complete <batch-id>`** — called by the daemon after the spawned CLI emits a completion-ack. Sets `completed_at = now` on the previously-claimed batch. Idempotent on re-invocation.
- **`peer receive --sweep`** (or a separate `peer-inbox sweep` subcommand) — runs the reaper pass: reverts `claimed_at = NULL` on rows where `completed_at IS NULL AND claimed_at < now() - TTL`. See §3.3.

Interactive receive (`peer receive --mark-read`) is unchanged — stays byte-identical with the existing `UPDATE inbox SET read_at = ? WHERE read_at IS NULL RETURNING ...` at `scripts/peer-inbox-db.py:1812-1820` and `go/pkg/store/sqlite/sqlite.go:277-281`. No shared code path.

### 2.2 Daemon process shape (O4 + W3)

**Single daemon binary (proposed):** `agent-collab-daemon` (Go, built at `go/cmd/daemon/`).

- One OS process per managed label per workspace. Example: a user running daemons for both `reviewer-codex` and `reviewer-gemini` in one repo gets two processes.
- Long-running shell (W3): holds the sessions-table row across spawns; avoids per-spawn re-registration.
- Ephemeral LLM per batch (item 1): each batch is served by a fresh `codex exec` / `gemini -p` / `claude -p` process that the shell spawns.
- Stable env-var propagation: the shell exports `CODEX_SESSION_ID` / `GEMINI_SESSION_ID` / `CLAUDE_SESSION_ID` (pick one per managed CLI kind) — or the CLI-neutral `AGENT_COLLAB_SESSION_KEY` — into the LLM's environment at each spawn. The ephemeral LLM never calls `session register`; daemon owns identity lifecycle.

**Config:** daemon reads a config file (YAML or TOML, coder's call — suggest TOML for parity with Codex config) listing managed labels, their cwd, their session-key, their CLI kind, their workspace_id (item 6, day-one forward-compat for v2 delivery-key migration), and their TTL + quiescence parameters. One file per daemon invocation; multiple daemons for multiple labels.

**Process model alternative (flagged):** single daemon binary managing multiple labels via N spawn-shell subprocesses. Tradeoff is simpler config distribution vs larger blast-radius on daemon crash. Recommending one-daemon-per-label for v0 per blast-radius isolation; revisit in v1+ if ops friction motivates consolidation.

### 2.3 Dispatch + path separation (alpha §1 + gamma elevation)

The path-separation invariant from §7.4(c) + alpha's cross-broadcast is structurally preserved:

- Interactive receive (`--mark-read`): unchanged SQL, unchanged columns, unchanged Python/Go runtimes. `tests/hook-parity.sh` stays green.
- Daemon receive (`--daemon-mode` + `--complete` + `--sweep`): new code paths, new column predicates (`claimed_at IS NOT NULL AND completed_at IS NULL` for in-flight; `claimed_at IS NULL AND read_at IS NULL` for unclaimed).
- Optional `sessions.receive_mode TEXT DEFAULT 'interactive'` sanity-check column (alpha §3 defense-in-depth): coder's v0 call. **Recommendation: include it.** The column-WHERE-clause partition is sufficient for correctness but adds no clarity to operators inspecting the sessions table. A single-character column addition (`'interactive' | 'daemon'`) costs ~nothing and makes state explicit. Un-ratified; challenge if this bloats scope.

## 3. Schema migration

### 3.1 New columns (additive, NULLable)

```sql
ALTER TABLE inbox ADD COLUMN claimed_at TEXT;        -- ISO-8601 UTC; NULL until daemon claims
ALTER TABLE inbox ADD COLUMN completed_at TEXT;      -- ISO-8601 UTC; NULL until daemon acks completion
ALTER TABLE inbox ADD COLUMN claim_owner TEXT;       -- optional: daemon-label that holds the claim, for observability
```

Plus (if §2.3 recommendation adopted):

```sql
ALTER TABLE sessions ADD COLUMN receive_mode TEXT NOT NULL DEFAULT 'interactive' CHECK (receive_mode IN ('interactive', 'daemon'));
```

### 3.2 Migration ordering (item 9 revision, W1 pattern)

One migration commit adds all four columns in both runtimes:

- Python: new goose migration file `migrations/XXXX_add_daemon_mode_columns.sql` (next number after whatever W1 ended on; coder checks at implementation time).
- Go: `go/pkg/store/sqlite/sqlite.go` schema-check path updated to match; migration applied via existing goose integration.
- Both runtimes in the same commit. Interactive-mode receive path does not reference the new columns, so `tests/hook-parity.sh` stays green throughout.

Daemon-mode runtime consumption (the `--daemon-mode` / `--complete` / `--sweep` verbs themselves) wires up in **subsequent commits**, Python first per W1 pattern. Go parity follows. Each runtime's daemon-mode verb commits against an already-migrated schema, so there is no window where one runtime references a column the other has not added.

### 3.3 Sweeper (third actor)

Periodic process — either a separate `peer-inbox sweep` subcommand invoked on cron/systemd-timer, or an in-daemon goroutine. Either approach works; **recommend subcommand + timer** for v0 so the sweeper can be audited and triggered independently of the daemon, and so a crashed daemon doesn't also take out its own recovery mechanism.

Sweeper query:

```sql
UPDATE inbox
SET claimed_at = NULL, claim_owner = NULL
WHERE claimed_at IS NOT NULL
  AND completed_at IS NULL
  AND claimed_at < ?  -- now - TTL
RETURNING id, to_cwd, to_label, claim_owner;
```

TTL default: **5 minutes**, configurable per-daemon. Rationale: order-of-magnitude wider than any legitimate fresh-invocation LLM turn (typical 10-90s); narrow enough that a crash-with-unacked-batch heals before a second batch queues behind it. Un-ratified — reviewers may have calibration arguments.

### 3.4 What stays unchanged

- Hook hot-path (`ReadUnread` in Go, `format_hook_block` in Python for interactive-mode receive). Not touched.
- `tests/hook-parity.sh` fixtures. Gate stays green across the migration (per alpha+gamma convergence in §7.4(c)).
- Send path (`peer send`). Senders don't need to know about daemon-mode; routing is still `(to_cwd, to_label)`.
- Idempotency / mtime marker (`mark_inbox_dirty`). Dirty-marker writes on send regardless of receive mode.

## 4. Daemon-harness requirements

The daemon shell (W3 long-running process) wraps every LLM spawn with the operational requirements catalogued in §7.6 Q#4 + §7.3. Violating any of these silently breaks peer-inbox delivery to the spawned worker, so the harness must handle them explicitly:

1. **Stable session-key env propagation (item 5; §7.3).** Shell exports one of `CLAUDE_SESSION_ID` / `CODEX_SESSION_ID` / `GEMINI_SESSION_ID` — or `AGENT_COLLAB_SESSION_KEY` if CLI-neutral — into every LLM spawn's environment. The value is resolved once at daemon startup (from config) and held stable for the daemon's lifetime. Ephemeral LLM never calls `session register`; daemon-registered session row owns identity.

2. **`codex exec` stdin-close** (test-engineer Option J carry-over). `codex exec` hangs on open stdin. Every spawn must pass `< /dev/null` (or equivalent) so the CLI sees EOF and exits cleanly after its turn.

3. **`codex exec --skip-git-repo-check`** (test-engineer carry-over). Codex refuses to run outside a git-tracked cwd without this flag. The daemon may run in a workspace that is not git-tracked — add the flag unconditionally so the spawn doesn't fail during e.g. a non-repo scratch workspace.

4. **Gemini hook timeout units** (shipped at `b5f9485`). Gemini's hook timeout is milliseconds; Claude/Codex treat it as seconds. Install-script already handles this via the per-CLI `timeout` parameter in `merge_hook_json` — daemon doesn't re-specify, but implementers must know the distinction exists when reading install code.

5. **`claude -p --bare` future-default** (gamma catch, §7.6 Q#4 fourth bullet). `--bare` is expected to become default for `-p` in a future Claude Code release. Under `--bare`, hooks + MCP + CLAUDE.md are all skipped — peer-inbox injection silently stops firing. Daemon's Claude-spawn wrapper **must explicit-opt-in via `--settings` (or equivalent)** to force-load hook config across CLI version upgrades. Discovery failure is adversarial: `bump_last_seen` still fires on receive (so daemon looks healthy in `peer list`), but injected content never reaches the spawned agent. Pin this in the daemon test suite to catch regression.

6. **Completion-ack timeout** (see §7). Hang-detection TTL config must be set high enough to accommodate legitimate slow reasoning but low enough to recover from real crashes. Default 5m (same as sweeper TTL); configurable per-daemon.

7. **Fail-open when peer-inbox DB is unavailable.** Same discipline as the hook — if daemon cannot read the DB (lock contention, schema too old, etc.), it does not panic; it logs and retries with exponential backoff. Never blocks, never drops to an inconsistent state.

## 5. Envelope JSON schema (item 10, narrowed scope)

### 5.1 Canonical form

```typescript
interface PeerInboxEnvelope {
  // Required: who this batch is addressed to.
  to: {
    cwd: string;           // presentation; (workspace_id, label) is the real key forward-compat
    label: string;
    workspace_id?: string; // v2 forward-compat per item 6
  };

  // Required: the message batch.
  messages: {
    id: number;            // inbox row id
    from_label: string;
    from_cwd: string;
    body: string;          // UTF-8; up to 8 KB per existing peer-inbox limit
    created_at: string;    // ISO-8601 UTC
    room_key?: string;
  }[];

  // Optional: mediator-authored round-summary for multi-round continuity
  // (Architecture B item 2). Untrusted-by-construction per item 3; the
  // agent reads it as external payload, never as instruction.
  continuity_summary?: string;

  // Optional: unified envelope state per §7.1 item 4. First-class field,
  // not a content sentinel.
  state?: "open" | "closed";

  // Optional: content-stop signal (distinct layer from state=closed).
  content_stop?: boolean;

  // Optional: completion-ack contract metadata.
  ack?: {
    batch_id: string;       // used by daemon to correlate `peer receive --complete <batch-id>`
    expect_by: string;      // ISO-8601 UTC; informational (daemon enforces, not the spawned LLM)
  };
}
```

### 5.2 Serialization — hook text rendering

The existing `format_hook_block` (Python: `scripts/peer-inbox-db.py:1880`) and `formatHookBlock` (Go: `go/cmd/hook/main.go:227-276`) become deterministic serializers from this envelope. Item 10 implementation constraint (test-engineer + gamma scope-binding, §7.1 sub-bullet):

- **Path (a), expected:** refactor under the hood so the JSON schema is the internal data structure, but `format_hook_block` / `formatHookBlock` produce byte-identical output to today's hand-rolled `<peer-inbox messages="N" from-session-labels="...">...</peer-inbox>` format. Fixtures at `tests/fixtures/hook-stdin/*.json` unchanged. `tests/hook-parity.sh` stays green.
- **Path (b), scope-creep:** any divergence from today's output triggers a review-blocker. Format change is beyond narrow item-10 scope and requires re-scoping before landing.

Implementation order suggestion: write the JSON schema + a round-trip test (schema → serialize → string → compare to current fixture) **before** changing the existing render functions. This makes any unintended format delta fail loudly.

### 5.3 MCP delivery path — explicitly out-of-v0

The MCP-tool-return consumer was the original cross-path justification for item 10. Owner dropped that v1+ per the second 2026-04-18 directive. For v0 trio the envelope is only serialized to hook text; the JSON shape is internal. Keeping the schema structured now keeps v1+ evolvability cheap without shipping a second delivery path today.

## 6. Four-layer termination stack wire-up

Per §7.1 item 7 and §7 reviewer convergence:

### Layer 1 — Content-stop sentinel

- Producer: any peer writes a content-stop sentinel token into a message body to signal "this exchange is done." Token form: double-bracket-end-of-room sentinel as referenced in the MCP server instructions (see `docs/PEER-INBOX-GUIDE.md`; **do not literalize the token in daemon code comments** — use a named constant).
- Consumer: the daemon checks for the sentinel in each received batch. If present, daemon completes the current batch (ack normally) and then transitions the managed label to a "drain then dormant" state — no new claims until state clears externally.

### Layer 2 — `envelope.state: closed`

- Producer: the mediator (or any authorized peer) sends a message whose envelope has `state: "closed"`. Semantically separate from Layer 1: "daemon go dormant" vs "this exchange is done."
- Consumer: on receive, daemon unclaims any in-flight batch it hasn't yet sent to the LLM (to avoid spending cycles on closed rooms), acks in-flight LLM calls it can't cancel, and transitions to dormant. Waking requires external intervention (config update + daemon restart, or a state-reset verb).

### Layer 3 — Quiescence primitives (gamma #4)

Three independent checks, all defaulting to conservative values and configurable:

- **Exponential backoff on same-pair turns within N seconds.** If daemon detects rapid back-and-forth with a single peer (default: > 3 turns in 60s), it lengthens its poll interval to prevent runaway. Counter resets on quiet period.
- **Empty-response terminates.** If the spawned LLM produces an empty or whitespace-only response, daemon does not re-spawn. Treated as an implicit "I have nothing to add."
- **Pause-on-idle timer.** If no new messages arrive for N minutes (default: 30), daemon reduces poll frequency and logs a pause event. User-overridable.

### Layer 4 — Heartbeat / liveness

Heartbeat is **scoped to the v3.1 papercut at commit `712121c`**, not to this doc. Topic 3 v0 depends on heartbeat landing so that `peer list` accurately distinguishes "daemon alive" from "daemon crashed mid-batch." If heartbeat hasn't shipped by the time daemon-mode implementation starts, coder should surface the ordering dependency before writing v0 daemon code.

## 7. Completion-ack signal shape

### 7.1 What the daemon needs to detect

Per item 9 revised (§7.1 + §7.4(c) convergence): **ack-absence, not wall-clock TTL**, is the primary trigger for requeue. TTL is fallback for ack-loss (CLI crash, network partition, spawned process killed by OS).

### 7.2 Three candidate mechanisms

Order of preference, all three viable, coder's call at implementation:

1. **Exit-code + stdout-sentinel.** Spawned CLI exits 0 with a recognizable stdout marker (e.g. `<peer-inbox-ack batch-id="...">ok</peer-inbox-ack>`). Daemon parses stdout; on marker, calls `peer receive --complete <batch-id>`. On missing marker or non-zero exit, daemon does not call `--complete` → sweeper requeues after TTL. **Pro:** works for all three trio CLIs uniformly; no new CLI-side protocol. **Con:** stdout-parsing is fragile (gamma flagged this as v0-shortcut earlier in the auto-reply discussion; same caveat applies here).

2. **Direct `peer send --ack` from inside the spawned LLM.** The spawned CLI's agent loop calls `agent-collab peer receive --complete <batch-id>` (via Bash tool or similar) as its final action. **Pro:** no stdout-parsing; agent is the source of truth for completion. **Con:** requires the agent to know about `batch-id`; envelope must surface it (§5.1 `ack.batch_id`); agent prompt template must instruct completion semantics (adds surface area to the daemon prompt template — keep it static per item 3).

3. **MCP-tool-backed ack.** Daemon provides a `peer_inbox_ack` MCP tool; spawned LLM calls it. Clean separation of protocol from stdout. Costs: each CLI needs MCP server setup configured, and Codex/Gemini MCP client behavior under `exec`/`-p` is not empirically validated for server-initiated calls. **Likely v1+** unless S3 research clears it first.

**Recommendation:** ship mechanism (1) + (2) together in v0 — daemon accepts either an exit-code stdout-marker OR a `peer receive --complete` call from the agent, whichever arrives first. Redundancy lets the agent use the direct-call path when tool permissions allow, and falls back to stdout-parsing when they don't. Mechanism (3) explicitly deferred.

### 7.3 Batch-id generation

Daemon generates batch-id at claim time (UUID or `{label}-{timestamp}-{random}`). Writes it to `inbox.claim_owner` (or a dedicated `claimed_batch_id` column — coder's call on column granularity). Batch-id appears in the envelope `ack.batch_id` field (§5.1) so the spawned LLM knows what to ack.

## 8. Test plan

### 8.1 Preserved gates

- **`tests/hook-parity.sh`** stays green throughout the migration. Interactive receive path is not touched; hook-block text format unchanged (path (a) per §5.2). Validate on every daemon-mode commit: `bash tests/hook-parity.sh`.
- **T1-T4 regression gates** (from W1). All continue to pass. If any gate requires updating, flag as review-blocker; investigate.
- **In-process hook latency bench** (`go/cmd/hook/hook_bench_test.go`). Must stay under the p99 <15ms budget. Daemon-mode code does not touch the hook hot-path, so expected to pass unchanged — verify on each hot-path-adjacent commit.

### 8.2 New v0 gates (coder authors first drafts; test-engineer owns)

- **Daemon-mode claim/ack/sweep lifecycle** (`tests/daemon-mode-lifecycle.sh` or Go integration test). Script:
  1. Seed inbox with N unread rows addressed to daemon label.
  2. Invoke `peer receive --daemon-mode` → assert N rows returned, `claimed_at` set, `read_at` still NULL, `completed_at` still NULL.
  3. Invoke `peer receive --complete <batch-id>` → assert `completed_at` now set; same rows no longer appear in any receive query.
  4. Second seed; claim but do NOT complete; sleep TTL+1s; run sweeper → assert `claimed_at` NULL again; rows re-appear in next `--daemon-mode` claim.

- **Interactive-mode path isolation** (daemon-mode gate): verify that an interactive `peer receive --mark-read` called during a daemon-mode claim window does NOT see rows that are currently `claimed_at IS NOT NULL`. Assert clean row-partition under concurrent receivers.

- **Daemon-harness caveats (regression coverage).** For each caveat in §4:
  - Spawned `codex exec` with open stdin → assert daemon timeout triggers and requeues.
  - `codex exec` in non-git cwd without `--skip-git-repo-check` → assert daemon-harness adds the flag (fixture-level).
  - Claude spawn without `--settings` → today passes; once `--bare` ships as default, this becomes a failure without the fix. Consider pinning the daemon-harness args as a fixture comparison so future drift fails loudly.

- **Envelope schema round-trip** (§5.2 suggestion): JSON schema → serialize → compare to hook-parity fixture output. Fails loudly on any format delta.

- **Completion-ack mechanism mixing.** Daemon spawns an LLM; LLM emits both stdout marker AND calls `peer receive --complete`. Assert idempotent completion (no double-completion side-effect).

### 8.3 E2E probe (owner-supervised, one-off per §7 pattern)

Same shape as Option J probes at `docs/HOOK-PARITY-VALIDATION.md`, adapted for daemon-mode:

- **Probe E1:** start daemon for Codex label; send three messages from Claude; verify Codex replies autonomously via daemon spawn.
- **Probe E2:** Gemini variant of E1.
- **Probe E3:** kill Codex mid-spawn (SIGKILL on the spawned process); verify sweeper requeues after TTL and next spawn handles the batch.
- **Probe E4:** send a `state: "closed"` envelope to daemon; verify dormant transition and no further spawns.

E2E probes are smoke tests at ship-time, not persistent CI gates. Shape-2 CI coverage is §8.2.

## 10. Open implementation-level questions

These are implementation-shape questions not yet answered in §7; surfacing for reviewer challenge **before** code lands.

1. **Single daemon-binary managing multiple labels vs one-per-label** (§2.2). Recommendation: one-per-label for v0 blast-radius isolation. Counter-argument: operational complexity of managing N daemons. Reviewer-call.
2. **Sweeper as subcommand+timer vs in-daemon goroutine** (§3.3). Recommendation: subcommand+timer (independent audit path; crashed daemon doesn't take out recovery). Counter-argument: more moving parts to install. Reviewer-call.
3. **TTL default** (§3.3, §4 bullet 6, §7.1). Recommendation: 5 minutes. Counter-arguments span narrow ranges; reviewers with calibration data (typical LLM reasoning turn duration under real workloads) should challenge if they have evidence.
4. **Completion-ack mechanism** (§7.2). Recommendation: ship (1) + (2) together, defer (3). Counter-argument: (1) stdout-parsing is fragile; (2) adds prompt-template surface area. Reviewer-call on whether to pick one instead of both.
5. **`sessions.receive_mode` column** (§2.3). Recommendation: include. Counter-argument: scope creep beyond what §7.1 strictly requires (alpha flagged as "coder's call"). Reviewer-call.
6. **Daemon-config format** (§2.2). Recommendation: TOML for Codex-config parity. Counter-argument: YAML is more common in daemon-config ecosystems (systemd drop-ins, Kubernetes-adjacent tooling). Low-stakes; reviewer sanity-check only.
7. **Go CLI surface gap** (§9 commit 4). `go/cmd/peer-inbox/` does not exist today. Do daemon-mode verbs live there (new cmd dir) or are they added to existing Go CLI scaffold? Alpha's audit flagged this as a scope-level note; now it's an implementation-level decision.
8. **Batch-id column granularity** (§7.3). `claim_owner` alone (daemon-label + ad-hoc batch-id parsing) or dedicated `claimed_batch_id` column? Lean `claimed_batch_id` for explicitness; `claim_owner` still useful for observability. Reviewer-call.

None of these reopen §7 ratified scope. All are implementation shape questions with tradeoffs coder considered but doesn't want to lock without reviewer input.
