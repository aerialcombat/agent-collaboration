## 1. What is being built

An always-on OS-local daemon process (one per managed label, or one-per-workspace with per-label worker-shells — see §2.2 for the shape choice) that:

1. Watches the peer-inbox for messages addressed to its managed label(s).
2. On new messages, atomically claims them into daemon-mode receive state (new column `claimed_at`, not existing `read_at`).
3. Spawns a fresh `claude -p` / `codex exec` / `gemini -p` invocation per batch, exporting the stable session-key env var so the spawned CLI's hook resolves to the same `(cwd, label)` session row.
4. **Injects the claimed batch as the spawned CLI's user prompt directly** — daemon serializes the envelope (§5) and passes it via argv/stdin to the CLI. Daemon does NOT rely on Option J's hook to deliver the peer-inbox content; the hook path stays interactive-only (see §2.4 for why this matters — idle-birch's blocker resolved).
5. Waits for the CLI to emit a completion-ack signal (§7).
6. On ack: marks the batch `completed_at`; on ack-absence-within-TTL: sweeper requeues (reverts `claimed_at`) for redelivery on the next claim cycle.
7. Respects the four-layer termination stack (content-stop sentinel + `envelope.state: closed` + quiescence primitives + heartbeat).

**Scope:** Codex + Gemini reactive parity with Claude for the empirically-confirmed trio, via O4 orchestrator + W3 worker-shape + hook-delivery. Canonical JSON envelope schema with deterministic hook serialization (item 10, code-quality scope). Completion-ack with TTL-fallback recovery semantics (item 9 revision). See §7.6 of the brainstorm doc for the full narrowed-default wording.

**Out of this doc's scope:** Cline / Cursor / Windsurf (owner directive, full horizon); API-backed workers (Topic 2); heartbeat/reaper implementation (v3.1 papercut at `712121c` — daemon-mode depends on it being in place but does not ship it).

## 2. Verb surface

### 2.1 New CLI verb(s)

Three new verbs. The daemon uses Go-direct SQL via `go/pkg/store/sqlite/sqlite.go` at runtime (matching Option J's hook-reader pattern at `go/cmd/hook/main.go`); the Python verbs are the operator surface + parity anchor, not the daemon's call path. This avoids a Python-subprocess dependency in daemon install (gamma #6).

- **`peer receive --daemon-mode --as <label>`** — atomic claim of unread rows addressed to `self` into daemon-mode state. Explicit SQL (alpha §C; anchors the path-separation invariant at scope-doc level):

  ```sql
  UPDATE inbox
  SET claimed_at = ?, claim_owner = ?          -- claim_owner = daemon label
  WHERE to_cwd = ? AND to_label = ?
    AND read_at IS NULL
    AND claimed_at IS NULL                     -- excludes already-claimed rows
  RETURNING id, from_cwd, from_label, body, created_at;
  ```

  The `claimed_at IS NULL` predicate is load-bearing: prevents two concurrent daemon-claim calls from double-claiming, and prevents a fresh claim from grabbing a row whose ack is in-flight from a prior batch. Returns claimed rows as JSON for the daemon (envelope serialization per §5) or as a `<peer-inbox>` hook-envelope block for piping into a spawned CLI's hook context.

- **`peer receive --complete --as <label>`** — called by the daemon after the spawned CLI emits completion. Fail-loud semantics (alpha §B + reason endorse; closes the at-most-once-processing invariant against the sweeper-reaped-mid-processing race):

  ```sql
  UPDATE inbox
  SET completed_at = ?
  WHERE to_cwd = ? AND to_label = ?
    AND claim_owner = ?
    AND claimed_at IS NOT NULL
    AND completed_at IS NULL
  RETURNING id;
  ```

  A 0-row UPDATE means the claim was reaped (sweeper reverted `claimed_at = NULL` because daemon missed its TTL). `--complete` exits non-zero and emits a structured error on stdout so the daemon logs the rejected work; the sweeper-then-reclaim cycle handles redelivery via a fresh daemon spawn. This is distinct from "idempotent": idempotent-on-same-claim calls still succeed (UPDATE matches zero rows on the second call because `completed_at` is already set — but it matches on `claim_owner + claimed_at IS NOT NULL`, not on `completed_at IS NULL` being the first writer). Implementation check: verify idempotent re-call of the same claim returns success (matches completed_at-already-set via the explicit NULL predicate only).

- **`peer receive --sweep`** (or a separate `peer-inbox sweep` subcommand) — reaper pass. Runs periodically (see §3.3 for cadence + TTL ordering).

Interactive receive (`peer receive --mark-read`) is unchanged — stays byte-identical with the existing `UPDATE inbox SET read_at = ? WHERE read_at IS NULL RETURNING ...` at `scripts/peer-inbox-db.py:1812-1820` and `go/pkg/store/sqlite/sqlite.go:277-281`. No shared code path with daemon-mode verbs.

**Batch identity (alpha #8 + reason endorse; simplification from v1 draft):** daemon enforces single-batch-at-a-time semantics — one outstanding claim per daemon at any time, claim→process→complete→next. "Current batch" is whatever rows match `claim_owner = self AND completed_at IS NULL AND claimed_at IS NOT NULL`. No separate batch-id column or envelope field needed in v0. If v1+ wants concurrent-batch-per-daemon semantics, adding a `claimed_batch_id` column is a clean additive change.

### 2.2 Daemon process shape (O4 + W3)

**Single daemon binary (proposed):** `agent-collab-daemon` (Go, built at `go/cmd/daemon/`).

- One OS process per managed label per workspace. Example: a user running daemons for both `reviewer-codex` and `reviewer-gemini` in one repo gets two processes.
- Long-running shell (W3): holds the sessions-table row across spawns; avoids per-spawn re-registration.
- Ephemeral LLM per batch (item 1): each batch is served by a fresh `codex exec` / `gemini -p` / `claude -p` process that the shell spawns.
- Stable env-var propagation: the shell exports `CODEX_SESSION_ID` / `GEMINI_SESSION_ID` / `CLAUDE_SESSION_ID` (pick one per managed CLI kind) — or the CLI-neutral `AGENT_COLLAB_SESSION_KEY` — into the LLM's environment at each spawn. The ephemeral LLM never calls `session register`; daemon owns identity lifecycle.

**Config:** daemon reads a config file (YAML or TOML, coder's call — suggest TOML for parity with Codex config) listing managed labels, their cwd, their session-key, their CLI kind, their workspace_id (item 6, day-one forward-compat for v2 delivery-key migration), and their TTL + quiescence parameters. One file per daemon invocation; multiple daemons for multiple labels.

**Process model alternative (flagged):** single daemon binary managing multiple labels via N spawn-shell subprocesses. Tradeoff is simpler config distribution vs larger blast-radius on daemon crash. Recommending one-daemon-per-label for v0 per blast-radius isolation; revisit in v1+ if ops friction motivates consolidation.

### 2.3 Dispatch + path separation (alpha §1 + gamma elevation + reviewer convergence)

The path-separation invariant from §7.4(c) is preserved by construction + verb-entry enforcement:

- Interactive receive (`--mark-read`): unchanged SQL, unchanged columns, unchanged Python/Go runtimes. `tests/hook-parity.sh` stays green.
- Daemon receive (`--daemon-mode` + `--complete` + `--sweep`): new code paths, new column predicates (`claimed_at IS NOT NULL AND completed_at IS NULL` for in-flight; `claimed_at IS NULL AND read_at IS NULL` for unclaimed).
- **`sessions.receive_mode TEXT NOT NULL DEFAULT 'interactive' CHECK (receive_mode IN ('interactive', 'daemon'))`** — included per converged reviewer vote (alpha elevation to correctness gate + gamma concede + reason fail-loud-frame consistency). Each verb checks the managed label's `receive_mode` at entry: interactive verbs refuse to run on daemon-mode labels (fail loud, not silent no-op); daemon verbs refuse on interactive-mode labels. Closes the misconfiguration footgun alpha named (interactive `--mark-read` double-consuming daemon-claimed rows if a label flips mode). Failure mode of the column-WHERE-clause partition alone: silent drop on misconfig. Failure mode with receive_mode preflight: loud error. Reason's frame: the scope-doc's other convergent semantics (fail-loudly on contract violation) all align on loud-over-silent; receive_mode joins the pattern.

### 2.4 Daemon delivery architecture (idle-birch blocker resolved)

**Load-bearing:** daemon does NOT rely on Option J's hook path for peer-inbox delivery to its spawned CLIs. The hook path is hard-wired to the interactive receive verb (`hooks/peer-inbox-inject.sh:149` calls `peer receive --format hook-json --mark-read`; `go/cmd/hook/main.go` runs the equivalent `ReadUnread`). If a daemon-spawned CLI ran the hook unchanged, the interactive verb would see the daemon-claimed rows (still `read_at IS NULL`) and consume them through the interactive path — collapsing the path-separation invariant and making `claimed_at` / `completed_at` recovery semantics non-authoritative (idle-birch review-blocker, `5e81b7c`).

**Resolution (alpha concrete shape + reason/idle-birch convergence):** daemon builds the envelope itself and injects it as the spawned CLI's user prompt via argv/stdin. Daemon spawns opt-out of hook-side DB receive via an env-flag short-circuit:

1. Daemon spawn-harness exports `AGENT_COLLAB_DAEMON_SPAWN=1` into the spawned CLI's environment (matches existing `AGENT_COLLAB_FORCE_PY=1` additive env-flag pattern at `hooks/peer-inbox-inject.sh:126`).
2. `hooks/peer-inbox-inject.sh` adds an early short-circuit near the top: if `AGENT_COLLAB_DAEMON_SPAWN=1` is set, exit 0 with no output. Additive; interactive-mode flow unchanged.
3. `go/cmd/hook/main.go` `run()` adds the same env check at the top, returns 0 immediately with no envelope.
4. Daemon calls the envelope serializer that backs item 10 (canonical JSON → `format_hook_block`/`formatHookBlock` text rendering) and pipes the resulting `<peer-inbox>...</peer-inbox>` block as the CLI's prompt input.

**Byte-equivalence note (alpha insight):** this puts the item 10 serializer on the critical path with two consumers — existing hook delivery (for interactive-mode labels) + daemon prompt-injection (for daemon-mode spawns). Both paths produce byte-identical envelope text. The "no MCP consumer in v0" framing from §5.3 no longer means "one consumer only"; the daemon prompt-injection path is the second consumer in v0. Item 10's structural-refactor-under-the-hood scope (path (a)) is exactly right because both consumers depend on byte-stable serialization from the same internal JSON schema.

**Hook-parity gate preservation:** `tests/hook-parity.sh` stays green because (a) interactive-mode hook path is unchanged, (b) the env-flag short-circuit is additive and doesn't touch the envelope-rendering code, and (c) the daemon prompt-injection path produces byte-identical text to what hook delivery produces (same serializer). New gate added in §8.2 to pin the env-flag short-circuit behavior.

## 3. Schema migration

### 3.1 New columns (additive)

```sql
-- Inbox: daemon-mode claim state. NULLable columns; interactive-mode rows never set them.
ALTER TABLE inbox ADD COLUMN claimed_at TEXT;           -- ISO-8601 UTC; NULL until daemon claims
ALTER TABLE inbox ADD COLUMN completed_at TEXT;         -- ISO-8601 UTC; NULL until daemon acks completion
ALTER TABLE inbox ADD COLUMN claim_owner TEXT;          -- daemon-label that holds the claim; set by --daemon-mode, survives sweep for audit

-- Sessions: receive-mode verb-entry gate + daemon-state for closed-at-claim-time check.
ALTER TABLE sessions ADD COLUMN receive_mode TEXT NOT NULL DEFAULT 'interactive' CHECK (receive_mode IN ('interactive', 'daemon'));
ALTER TABLE sessions ADD COLUMN daemon_state TEXT NOT NULL DEFAULT 'open' CHECK (daemon_state IN ('open', 'closed'));
```

- `claimed_at` / `completed_at` / `claim_owner` form the daemon-mode claim state machine (unclaimed → claimed → completed; sweeper reverts claimed → unclaimed on TTL-expired).
- `claim_owner` is NOT nullified by the sweeper (alpha §A); the next claim overwrites it. Audit trail by construction: `claim_owner` + (`completed_at IS NULL`, `claimed_at IS NULL`) reads as "this daemon's last claim on this row was reaped without completion."
- `receive_mode` enforces the verb-entry check (alpha §2 correctness gate + gamma concede + reason fail-loud consistency).
- `daemon_state` implements the closed-at-claim-time check for Layer 2 termination (gamma #2 + alpha DB-backed recommendation); queried by `--daemon-mode` before it claims so a closed daemon doesn't re-enter the claim cycle via sweeper-requeued rows.

### 3.2 Migration ordering (item 9 revision, W1 pattern)

One migration commit adds all five columns in both runtimes:

- Python: new goose migration file `migrations/XXXX_add_daemon_mode_columns.sql` (next number after whatever W1 ended on; coder checks at implementation time).
- Go: `go/pkg/store/sqlite/sqlite.go` schema-check path updated to match; migration applied via existing goose integration.
- Both runtimes in the same commit.

**Interactive-mode SQL does change.** Per gamma's "both fixes, not alternative" position: interactive receive WHERE clauses add `AND claimed_at IS NULL` in Python (`scripts/peer-inbox-db.py:1812-1820`) and Go (`go/pkg/store/sqlite/sqlite.go:277-281`). This is the **SQL-layer partition** that closes idle-birch's blocker at construction — daemon-claimed rows are invisible to the interactive path regardless of whether the daemon-spawn hook remembered to set `AGENT_COLLAB_DAEMON_SPAWN=1`. `tests/hook-parity.sh` fixtures have no `claimed_at IS NOT NULL` rows, so hook text output is byte-identical under fixture state; the preserved invariant is **hook/output parity for non-daemon rows, not SQL byte-identity** (idle-birch doc-precision).

Same-commit runtime parity on this predicate is mandatory per item 9 pattern: both Python and Go SQL gain the predicate simultaneously, else the asymmetric-visibility window reintroduces double-consumption risk the fix exists to prevent.

Daemon-mode runtime consumption (the `--daemon-mode` / `--complete` / `--sweep` verbs themselves) wires up in **subsequent commits**, Python first per W1 pattern. Go parity follows.

### 3.3 Sweeper (third actor)

Periodic process — either a separate `peer-inbox sweep` subcommand invoked on cron/systemd-timer, or an in-daemon goroutine. Either approach works; **recommend subcommand + timer** for v0 so the sweeper can be audited and triggered independently of the daemon, and so a crashed daemon doesn't also take out its own recovery mechanism.

Sweeper query (alpha §A fix — `claim_owner` is NOT nullified; surviving as last-claim audit):

```sql
UPDATE inbox
SET claimed_at = NULL                          -- note: claim_owner NOT nullified (alpha §A)
WHERE claimed_at IS NOT NULL
  AND completed_at IS NULL
  AND claimed_at < ?                           -- now - sweeper_ttl
RETURNING id, to_cwd, to_label, claim_owner;   -- claim_owner now returns the observability value, not NULL
```

**TTL ordering invariant (gamma #1 + alpha complementary frame).** Two distinct TTLs, must obey an ordering:

- `daemon_ack_timeout` (default **5 minutes**) — daemon's own per-batch clock; on expiry, daemon abandons the batch internally and does not call `--complete` for it.
- `sweeper_ttl` (default **10 minutes = 2× daemon_ack_timeout**) — sweeper's wall-clock check against `claimed_at`.

`sweeper_ttl` MUST be strictly greater than `daemon_ack_timeout`. Properly-configured daemon abandons before sweeper reaps, so `--complete` is never called for a would-be-reaped batch. The fail-loud `--complete` semantic (§2.1) catches the near-impossible edge cases (clock skew, daemon-process death between abandon-and-mark-failed, sweeper runs just as daemon receives ack). TTL config validator must reject `sweeper_ttl <= daemon_ack_timeout` at daemon startup — misconfiguration is a fail-loud, not a silent race.

**TTL override for CI (test-engineer).** Both TTL values must be overridable via CLI flag (`--ack-timeout`, `--sweep-ttl`) or env var (`AGENT_COLLAB_ACK_TIMEOUT`, `AGENT_COLLAB_SWEEP_TTL`) for test viability. `tests/daemon-mode-lifecycle.sh` sets `ack=1s, sweep=2s` (preserving the 2× ratio) for bounded CI duration.

### 3.4 Daemon-sweeper-ack contract: five guarantees + one optimization

Reviewer-consolidated from alpha + gamma + reason + idle-birch + test-engineer cluster observation. Anchors the concurrency surface explicitly rather than leaving it implementation-latent. Guarantees (a) through (e) are fail-loudly-on-contract-violation per reason's semantic-consistency frame. Item (f) is a performance optimization layered on top, not a correctness guarantee.

**(a) Hook-path SQL partition (gamma; primary structural correctness, broadest coverage).** Interactive receive WHERE clauses add `AND claimed_at IS NULL` (Python + Go, same commit). Daemon-claimed rows are invisible to interactive readers at the SQL layer. **Critically: this is the ONLY defense that covers the Go hook hot-path** (`go/cmd/hook/main.go:135` → `go/pkg/store/sqlite/sqlite.go:277-281` calls `ReadUnread` directly, bypassing verb dispatch; the Go binary is the production default fast path for all three trio CLIs per `hooks/peer-inbox-inject.sh:126-144`). Holds by construction under ALL storage access paths regardless of daemon discipline.

**(b) Receive_mode verb-entry gate (alpha elevation; gamma concede; narrower coverage than (a)).** Each Python verb checks `sessions.receive_mode` at entry. Interactive verbs refuse to run on daemon-mode labels; daemon verbs refuse on interactive-mode labels. Fail-loud on misconfig before any row-state is touched. **Coverage limitation (alpha):** only covers Python-verb callers — the Go hook hot-path bypasses verb dispatch, so (b) does not protect it. (a) SQL partition is the primary coverage fix; (b) is verb-layer defense-in-depth valuable for Python-hook fallback (`hooks/peer-inbox-inject.sh:149` calls `peer receive --format hook-json --mark-read`) and user-CLI invocations (human typing `agent-collab peer receive --mark-read`), both of which route through `cmd_peer_receive`.

**(c) TTL ordering invariant (gamma #1 + alpha complementary).** `sweeper_ttl > daemon_ack_timeout`, enforced at daemon startup. Bounds the race window to near-impossible.

**(d) Claim-identity fail-loud on `--complete` (alpha §B).** `peer receive --complete --as <label>` UPDATE requires `claim_owner = ? AND claimed_at IS NOT NULL AND completed_at IS NULL`; 0-row UPDATE means the claim was reaped → caller's work is rejected and logged. Closes the at-most-once-processing invariant even when (c) fails (clock skew, etc.).

**(e) Closed-state check at claim-time (gamma #2 + alpha DB-backed preference).** `--daemon-mode` preflights `SELECT daemon_state FROM sessions WHERE cwd = ? AND label = ?`; if `'closed'`, returns zero rows without claiming. Sweeper-requeued rows cannot trigger a new spawn on a closed daemon. Observability bonus: closed-state visible in `peer list` via the sessions table.

**(f) Daemon-spawn hook short-circuit (idle-birch + alpha mechanism; performance optimization, not correctness).** Daemon exports `AGENT_COLLAB_DAEMON_SPAWN=1` into spawned CLI environment. `hooks/peer-inbox-inject.sh` and `go/cmd/hook/main.go` add an early short-circuit at the top (matches existing `AGENT_COLLAB_FORCE_PY=1` additive pattern at `hooks/peer-inbox-inject.sh:126`): if the env flag is set, exit 0 with no output. Happy-path optimization that skips the hook's DB round-trip entirely for daemon spawns. **Not a correctness guarantee** — if the daemon forgets to set the flag, (a) SQL partition still prevents double-consumption at the cost of one wasted DB query per daemon spawn turn. Listed here as a sixth bullet for doc completeness; reviewers (alpha + reason) frame the canonical contract as **five guarantees (a-e) + one optimization (f)**.

### 3.5 What stays unchanged

- Hook hot-path text OUTPUT format (`format_hook_block` / `formatHookBlock`). Under item 10, refactored to serialize from canonical JSON schema (§5), but path (a) preserves byte-identical output. `tests/hook-parity.sh` fixtures unchanged, gate stays green.
- `tests/hook-parity.sh` fixtures themselves — no fixture regeneration under path (a).
- Send path (`peer send`). Senders don't need to know about daemon-mode; routing is still `(to_cwd, to_label)`.
- Idempotency / mtime marker (`mark_inbox_dirty`). Dirty-marker writes on send regardless of receive mode.

## 4. Daemon-harness requirements

The daemon shell (W3 long-running process) wraps every LLM spawn with the operational requirements catalogued in §7.6 Q#4 + §7.3. Violating any of these silently breaks peer-inbox delivery to the spawned worker, so the harness must handle them explicitly:

1. **Stable session-key env propagation (item 5; §7.3).** Shell exports one of `CLAUDE_SESSION_ID` / `CODEX_SESSION_ID` / `GEMINI_SESSION_ID` — or `AGENT_COLLAB_SESSION_KEY` if CLI-neutral — into every LLM spawn's environment. The value is resolved once at daemon startup (from config) and held stable for the daemon's lifetime. Ephemeral LLM never calls `session register`; daemon-registered session row owns identity.

2. **`codex exec` stdin-close** (test-engineer Option J carry-over). `codex exec` hangs on open stdin. Every spawn must pass `< /dev/null` (or equivalent) so the CLI sees EOF and exits cleanly after its turn.

3. **`codex exec --skip-git-repo-check`** (test-engineer carry-over). Codex refuses to run outside a git-tracked cwd without this flag. The daemon may run in a workspace that is not git-tracked — add the flag unconditionally so the spawn doesn't fail during e.g. a non-repo scratch workspace.

4. **Gemini hook timeout units** (shipped at `b5f9485`). Gemini's hook timeout is milliseconds; Claude/Codex treat it as seconds. Install-script already handles this via the per-CLI `timeout` parameter in `merge_hook_json` — daemon doesn't re-specify, but implementers must know the distinction exists when reading install code.

5. **`claude -p --bare` future-default** (gamma catch, §7.6 Q#4 fourth bullet; gamma #4 strengthening). `--bare` is expected to become default for `-p` in a future Claude Code release. Under `--bare`, hooks + MCP + CLAUDE.md are all skipped — peer-inbox injection silently stops firing. Daemon's Claude-spawn wrapper **must explicit-opt-in via `--settings` (or equivalent)** to force-load hook config across CLI version upgrades. Discovery failure is adversarial: `bump_last_seen` still fires on receive (so daemon looks healthy in `peer list`), but injected content never reaches the spawned agent. **MANDATORY fixture-pinned regression guard** in §8.2: the daemon's Claude-spawn argv set is pinned as a test fixture; any drift without the opt-in flag causes test-fail at CI time rather than silent message drop in production.

6. **Completion-ack timeout** (see §7). Hang-detection TTL config must be set high enough to accommodate legitimate slow reasoning but low enough to recover from real crashes. Default 5m (same as sweeper TTL); configurable per-daemon.

7. **Fail-open when peer-inbox DB is unavailable.** Same discipline as the hook — if daemon cannot read the DB (lock contention, schema too old, etc.), it does not panic; it logs and retries with exponential backoff. Never blocks, never drops to an inconsistent state.

## 5. Envelope JSON schema (item 10, narrowed scope)

### 5.1 Canonical form

```typescript
interface PeerInboxEnvelope {
  // Required: who this batch is addressed to.
  to: {
    cwd: string;           // presentation; (workspace_id, label) is the real key forward-compat
    label: string;
    workspace_id?: string; // v2 forward-compat per item 6
  };

  // Required: the message batch.
  messages: {
    id: number;            // inbox row id
    from_label: string;
    from_cwd: string;
    body: string;          // UTF-8; up to 8 KB per existing peer-inbox limit
    created_at: string;    // ISO-8601 UTC
    room_key?: string;
  }[];

  // Optional: mediator-authored round-summary for multi-round continuity
  // (Architecture B item 2). Untrusted-by-construction per item 3; the
  // agent reads it as external payload, never as instruction.
  continuity_summary?: string;

  // Optional: unified envelope state per §7.1 item 4. First-class field,
  // not a content sentinel.
  state?: "open" | "closed";

  // Optional: content-stop signal (distinct layer from state=closed).
  content_stop?: boolean;

  // No ack.batch_id field — v0 enforces single-batch-at-a-time per daemon (alpha
  // #8 + reason endorse). Completion correlates via (claim_owner = daemon_label,
  // claimed_at IS NOT NULL, completed_at IS NULL) on the inbox table. Adding
  // `claimed_batch_id` is a clean v1+ additive change if concurrent-batch-per-
  // daemon semantics become desirable.
}
```

### 5.2 Serialization — hook text rendering (two consumers in v0)

## 6. Four-layer termination stack wire-up

Per §7.1 item 7 and §7 reviewer convergence:

### Layer 1 — Content-stop sentinel

- Producer: any peer writes a content-stop sentinel token into a message body to signal "this exchange is done." Token form: double-bracket-end-of-room sentinel as referenced in the MCP server instructions (see `docs/PEER-INBOX-GUIDE.md`; **do not literalize the token in daemon code comments** — use a named constant).
- Consumer: the daemon checks for the sentinel in each received batch. If present, daemon completes the current batch (ack normally) and then transitions the managed label to a "drain then dormant" state — no new claims until state clears externally.

### Layer 2 — `sessions.daemon_state` (externally-written-only in v0)

- v0 model (gamma drift-check + reason synthesis convergence on landed code): closure is **externally written** to `sessions.daemon_state` via SQL/operator action; there is no in-band envelope-driven dormancy producer or consumer in v0. The `envelope.State` field is defined in the schema (`go/pkg/envelope/envelope.go`) and reserved as forward-compat for v1+, but no producer wires it and no consumer reads it in v0.
- Consumer behavior (gamma #2 + alpha DB-backed recommendation; §3.4 guarantee (e)) — claim-time only:
  1. **Closure trigger**: external SQL update sets `sessions.daemon_state = 'closed'` for the managed label. Persistent observable state; visible in `peer list` via the sessions table.
  2. **Claim-time enforcement** (prevents sweeper-requeue-then-respawn footgun gamma flagged): `--daemon-mode` preflights `SELECT daemon_state FROM sessions WHERE cwd = ? AND label = ?` and returns zero rows immediately if `'closed'`. Inbox rows are untouched (no `claimed_at` set). Even if sweeper requeues rows, they do not trigger new spawns on a closed daemon.
  3. **In-flight batches at the moment of closure**: not actively cancelled in v0. The daemon's current batch (if any) runs to completion; subsequent claim attempts return zero rows once `daemon_state = 'closed'` lands.
  4. **Waking**: external intervention — operator flips `sessions.daemon_state` back to `'open'` (no `--reopen` verb shipped in v0; a daemon restart against the open state always works).
- **v1+ extension** (deferred): in-band envelope-driven dormancy. Producer (mediator/authorized peer) sends a message whose envelope has `state: "closed"`; daemon consumer flips `sessions.daemon_state` and unclaims in-flight pre-LLM batches. The `envelope.State` field reserved in v0's schema is the forward-compat anchor for this extension.

### Layer 3 — Quiescence primitives (gamma #4)

Three independent checks, all defaulting to conservative values and configurable:

- **Exponential backoff on same-pair turns within N seconds.** If daemon detects rapid back-and-forth with a single peer (default: > 3 turns in 60s), it lengthens its poll interval to prevent runaway. Counter resets on quiet period.
- **Empty-response terminates.** If the spawned LLM produces an empty or whitespace-only response, daemon does not re-spawn. Treated as an implicit "I have nothing to add."
- **Pause-on-idle timer.** If no new messages arrive for N minutes (default: 30), daemon reduces poll frequency and logs a pause event. User-overridable.

### Layer 4 — Heartbeat / liveness

Heartbeat is **scoped to the v3.1 papercut at commit `712121c`**, not to this doc. Topic 3 v0 depends on heartbeat landing so that `peer list` accurately distinguishes "daemon alive" from "daemon crashed mid-batch." If heartbeat hasn't shipped by the time daemon-mode implementation starts, coder should surface the ordering dependency before writing v0 daemon code.

## 7. Completion-ack signal shape

### 7.1 What the daemon needs to detect

Per item 9 revised (§7.1 + §7.4(c) convergence): **ack-absence, not wall-clock TTL**, is the primary trigger for requeue. TTL is fallback for ack-loss (CLI crash, network partition, spawned process killed by OS).

### 7.2 Two mechanisms, ordered by convergence

Preference order reviewer-converged (gamma recommendation flip + alpha endorse + reason endorse; mechanism previously marked "Exit-code + stdout-sentinel" as option 1 was flipped to fallback after stdout-parsing fragility concerns surfaced):

1. **Direct completion verb from inside the spawned LLM (PRIMARY) — two equivalent forms, Go fast-path preferred.** The spawned CLI's agent loop calls one of:
   - **`peer-inbox daemon-complete --cwd <cwd> --as <label>` (preferred, Go fast-path)** — no Python subprocess in the daemon-call path; extends gamma #6 Go-fast-path pattern to the agent side. Landed daemon `buildStaticPrompt` (`go/cmd/daemon/main.go`) prompts toward this form by default.
   - **`agent-collab peer receive --complete --as <label>` (Python-equivalent fallback)** — same semantics, equivalent UPDATE; available where the Go `peer-inbox` binary isn't on PATH.

   Both verbs are rendered equivalent on the database side: the UPDATE matches against `claim_owner = ? AND claimed_at IS NOT NULL AND completed_at IS NULL` — no batch-id parameter needed because `claim_owner` already holds the daemon's label. Daemon static prompt selects the Go form by default; either form completes the same claim.

   **Pro:** no stdout-parsing; agent is the source of truth for completion; aligns with Architecture B's "agent calls peer send directly" pattern converged in the auto-reply-driver round. **Con:** agent prompt template must instruct completion semantics — keeping the instruction static and scoped to "call this verb as your last action" preserves the §7.1 item 3 trusted-template constraint (no peer content interpolated into the template).

2. **Exit-code + structured-stdout fallback (FALLBACK, for cases where tool-call permission isn't available).** Spawned CLI emits a structurally-distinct ack marker as a JSONL line (e.g. final line of stdout: `{"peer_inbox_ack": true}` with optional metadata). Daemon scans stdout for the JSONL line after the spawn exits. **Marker form (gamma #3 + alpha concurrence on robustness):** JSONL is the structural fence — HTML-comment form (`<!-- peer-inbox-ack -->`) is vulnerable to agent prose that strips or reformats comments; raw XML-like tags collide with agent-prose discussions of peer-inbox itself (this very conversation contains many `<peer-inbox>` strings). JSONL with a specific `peer_inbox_ack: true` key is the least-collision-prone convention for ack detection.

**On "works for all three CLIs uniformly" (reason #4 + test-engineer + gamma precision):** stdout formats differ across the trio (Claude `--output-format text|json|stream-json`; Codex stderr-progress-then-stdout-final or `--json` JSONL; Gemini not yet detailed). Earlier draft's "uniformly" claim over-specifies. **Practical scope:** mechanism (1) is the uniform path (works everywhere); mechanism (2) requires per-CLI stdout-format handling in daemon spawn-harness (configure each CLI to emit `--json` or `--output-format json` where available; parse accordingly).

3. **MCP-tool-backed ack — DEFERRED (v1+).** Daemon provides a `peer_inbox_ack` MCP tool; spawned LLM calls it. Clean separation of protocol from stdout. Costs: each CLI needs MCP server setup configured, and Codex/Gemini MCP client behavior under `exec`/`-p` is not empirically validated for server-initiated calls. Explicitly deferred per Topic 3 v0 scope.

**v0 recommendation:** daemon accepts either path — mechanism (1) is PRIMARY and what the daemon prompt template instructs the agent to call; mechanism (2) is FALLBACK that catches the case where the agent couldn't complete the tool call (tool-permission revoked, timeout, interrupt). Whichever arrives first wins; `--complete`'s single-row-update + idempotent-on-already-completed means redundant signals are benign. Daemon's ack-timeout fires if neither arrives within `daemon_ack_timeout` (§3.3) → daemon abandons internally; sweeper eventually requeues per §3.4 (c)/(d).

### 7.3 Batch identity — single-batch-at-a-time semantics (per alpha #8 + reason + gamma converged)

Daemon enforces single-batch-at-a-time: one outstanding claim per daemon at any time. "Current batch" is whatever rows match `claim_owner = self AND claimed_at IS NOT NULL AND completed_at IS NULL`. No `claimed_batch_id` column in v0; no `batch_id` field in the envelope. Completion via mechanism (1) or (2) above correlates through `claim_owner` alone. Concurrent-batch-per-daemon semantics become a clean v1+ additive schema change if/when they're needed.

- **Daemon-mode claim/ack/sweep lifecycle** (`tests/daemon-mode-lifecycle.sh` or Go integration test). Script:
  1. Seed inbox with N unread rows addressed to daemon label.
  2. Invoke `peer receive --daemon-mode --as <label>` → assert N rows returned, `claimed_at` set, `read_at` still NULL, `completed_at` still NULL, `claim_owner = <label>`.
  3. Invoke `peer receive --complete --as <label>` → assert `completed_at` now set; same rows no longer appear in any receive query; **fail-loud on stale-claim re-call (per §3.4(d)): a second `--complete` against the same already-completed claim returns `EXIT_CONTRACT_VIOLATION=65`** (landed code in Python `cmd_peer_receive` daemon-mode `--complete` and Go `peer-inbox daemon-complete`; covered by `tests/daemon-mode-lifecycle.sh` step 4a + Go parity test).
  4. Second seed; claim but do NOT complete; sleep `sweep-ttl+1s`; run sweeper → assert `claimed_at` NULL again, `claim_owner` preserved (audit trail per §3.1 + alpha §A), rows re-appear in next `--daemon-mode` claim.

Invoke interactive `peer receive --mark-read` → assert only the unclaimed row returns; daemon-claimed row stays invisible. Run via BOTH Python CLI (`cmd_peer_receive`) AND Go hook binary (`ReadUnread`). This is the load-bearing test for the primary structural fix.

Consumer 1: invoke Go hook binary, capture `hookSpecificOutput.additionalContext` string. Consumer 2: invoke daemon's envelope-serializer on the same batch, capture output string. Assert byte-equal. Prevents divergence between hook-delivered and daemon-injected envelope text.

- **Receive-mode verb-entry rejection (§3.4 guarantee (b)).** Register a session with `receive_mode = 'daemon'`. Invoke `peer receive --mark-read --as <label>` → assert non-zero exit + "mode mismatch" error (or similar); no row-state change. Inverse: register interactive-mode session; invoke `peer receive --daemon-mode --as <label>` → same rejection.

- **TTL ordering invariant (§3.4 guarantee (c); gamma #1).** Start daemon with `--ack-timeout 2s --sweep-ttl 1s` (violating the invariant). Assert daemon startup fails with clear error. Start with `--ack-timeout 1s --sweep-ttl 2s` (correct ratio). Assert daemon starts normally.

- **Reap-race / stale-batch `--complete` fail-loud (§3.4 guarantee (d); alpha §B + gamma #5 consolidated).** Claim batch; sleep past `sweep-ttl`; run sweeper (reverts `claimed_at`); then invoke `peer receive --complete --as <label>` → assert non-zero exit + "claim no longer held" error (0-row UPDATE).
