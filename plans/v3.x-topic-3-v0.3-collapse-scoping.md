## 1. Goal + motivation

**Retire codex-direct and gemini-direct** as daemon CLI choices. Operators who want codex or gemini behavior configure `--cli=pi --pi-provider={openai-codex | google-antigravity} --pi-model=<model>` and get the exact same (or better) functionality through the existing spawnPi helper.

Four motivating observations:

1. **Empirical redundancy (2026-04-18 dogfood).** Three daemon-backed peers ran side-by-side in the same room with different pi providers (zai-glm, openai-codex, google-antigravity). All three demonstrated cross-spawn context retention via pi's operator-owned session files. The direct codex + gemini code paths (with their capture/translate machinery) produced functionally identical peer-visible behavior while carrying substantially more code complexity.

2. **Architectural simplification.** codex-direct currently requires: regex-based banner capture (scanning stdout + stderr per v0.1.2 fix), stale-UUID-fallback, UUID persistence, resume-argv branch. gemini-direct requires: `--list-sessions` delta-snapshot, UUID→index translation at resume time, race-warning on multi-new-UUID, 15-second timeout with env override. pi-routed replaces both with **path-as-identity**: daemon mints the session file path, persists it once, reuses forever; reset is a straightforward file-delete. Collapsing removes ~400 LoC of capture machinery from `go/cmd/daemon/main.go`.

3. **v3.1 dogfood degradation signal.** The v3.1 [PROMPT] patch E9 probe 2026-04-19 showed 4/5 daemon peers (including room-pi-codex) deliver peer-send replies correctly; only room-gemini (gemini-direct) hit `ack_timeout_abandoned` consistently — the same behavioral quirk v0.1.2 dogfood surfaced. gemini-direct is actively degrading, not just architecturally complex. pi-routed Google-provider peers (room-pi-gemini) have not exhibited this quirk.

4. **Operator ergonomics win.** Current ladder has four distinct daemon launch shapes (claude / codex / gemini / pi) with different config semantics (claude-settings-path vs nothing vs nothing vs pi.provider+pi.model+pi.session_dir). Post-v0.3: claude (asymmetric Arch B) + pi (universal Arch D). Operator mental model: two shapes.

**Architectural framing:** v0.3 is **subtractive + substitutive**. Subtracts: capture machinery + per-CLI spawn helpers for codex/gemini. Substitutes: pi-routed equivalents for any operator who previously used codex-direct or gemini-direct. claude-direct is unchanged; v0.1 §4.3 asymmetry stays documented. v0.4+ may revisit claude-collapse once claude-via-pi-anthropic auth provenance is owner-evaluated.

**Out of this doc's scope:**

- claude-direct retirement — deferred to v0.4+ scoping round; requires separate auth-provenance decision (see §4.3).
- v3.1 papercut-queue residual items (reason #3, smoke.sh macOS, hook-latency-exec p99, reason's v0 §6 L2 annotation) — owner-curated; not bundled.
- 100-turn pair-cap (AGENT_COLLAB_MAX_PAIR_TURNS) scope — envelope-layer concern; orthogonal to spawn collapse (§10 flag-for-awareness only).

## 3. Architecture contract

### 3.1 Trust boundary — what collapses, what stays

Pre-v0.3 trust-boundary ladder:
- claude → Arch B (no Arch D; asymmetry)
- codex → Arch D (opt-in capture + resume)
- gemini → Arch D (opt-in capture + resume; more complex than codex)
- pi → Arch D (path-as-identity; simplest)

Post-v0.3:
- claude → Arch B (unchanged)
- pi → Arch D (sole Arch D path; routes to any provider via pi's `--provider` flag)
- codex / gemini → **retired as --cli choices** (see §3.2 for mechanism)

The trust-boundary trade for pi-routed workflows is v0.2's — operator-owned JSONL file, opt-in via `--cli-session-resume`, file-delete reset. The v0.1 trade for codex + gemini (daemon owns opaque UUID pointer; cannot delete vendor session file) disappears because that capture machinery disappears.

**Multi-provider trust axis (v0.2 §3.1 inheritance):** pi-routed workflows already carry the "daemon is a gateway to ANY provider pi supports" trust surface. v0.3 does not expand this axis; it consolidates existing codex + gemini flows under the same acknowledgment lever (`--cli-session-resume` opt-in + operator-chosen `pi.provider`).

### 3.2 Retirement mechanism — three options for reviewer challenge

Owner's language ("retire them and just use pi") frames this as a deliberate removal, not a soft deprecation. Three implementation shapes:

**OPTION 3.2.a — HARD RETIRE (v0.3 = full removal):**
- Remove `spawnCodex`, `spawnGemini`, and their capture/resume helpers from `go/cmd/daemon/main.go`.
- `parseCLIKind("codex")` + `parseCLIKind("gemini")` return an error directing operators to the pi-routed equivalent.
- All codex/gemini test gates retired or rewritten for pi-routed equivalents.
- Migration: operators see a fail-loud startup error with a copy-pasteable replacement config snippet.
- Biggest code-delta; cleanest result; steepest operator cliff.

**OPTION 3.2.b — SOFT SHIM (v0.3 = preserve flag, reroute internally):**
- `--cli=codex` / `--cli=gemini` flags remain accepted.
- `spawnCodex` and `spawnGemini` become thin shims that construct a `piConfig` with the appropriate provider and call `spawnPi` internally.
- Legacy capture code (banner regex, `--list-sessions`) deleted; legacy argv shape (`codex exec resume ...` / `gemini --resume <N>`) disappears in favor of pi's argv.
- Migration: operators see a deprecation warning at startup ("`--cli=codex` is now routed through pi; set `--cli=pi --pi-provider=openai-codex` for the canonical form"). Existing configs keep working.
- Code-removal deferred to v0.4+; shim maintenance cost.

**OPTION 3.2.c — GRADUAL (v0.3 = warn-only, retire in v0.4+):**
- codex-direct + gemini-direct code paths unchanged.
- Add startup deprecation warning: "codex-direct and gemini-direct are scheduled for retirement in v0.4. Please migrate to `--cli=pi --pi-provider=<provider>`."
- v0.4+ does the HARD RETIRE.
- Smallest v0.3 delta; longest operator migration window; defers the complexity-reduction win.

**Coder recommendation: OPTION 3.2.b (SOFT SHIM) for v0.3 + HARD RETIRE in v0.4.** Rationale:
- SOFT SHIM lands the architectural win (capture machinery deletion) immediately while giving operators a 1-release window to migrate configs without breaking their daemons.
- The shim is cheap (a dozen LoC mapping `cli=codex` → `pi-provider=openai-codex` in parseFlags, then spawnCLI dispatches through spawnPi).
- v3.1 E9 probe just proved room-pi-codex behaves identically to room-codex for the operator — the shim preserves that equivalence during migration.
- HARD RETIRE at v0.3 (3.2.a) risks leaving operators with broken room-codex / room-gemini configs at an unexpected moment; the DROOD-style dogfood surfaced exactly that fragility.
- GRADUAL (3.2.c) fails to harvest the complexity-reduction win and carries the capture machinery forward indefinitely.

Flagged for reviewer challenge — especially test-engineer and reason, since the shim adds a small maintenance surface in exchange for the migration window. Owner decides.

### 3.3 Reset primitives (inherited from v0.2 §3.3)

All reset primitives in v0.2 §3.3 carry forward unchanged, now applying to ALL pi-routed flows (which is every non-claude flow under OPTION 3.2.b or post-3.2.a):

- **PRIMARY (a)** — `peer-inbox daemon-reset-session --as <label>` (Go) + `peer receive --reset-session --as <label>` (Python). NULL column + delete session file for pi-managed rows (gated on `sessions.agent == 'pi'`, see §8.1 plugin note below).
- **SECONDARY (d)** — auto-GC on L1 content-stop via `transitionClosed`. File-delete after Clear.
- **TERTIARY (c)** — SQL escape hatch + `rm` (operator-guide documented).
- **REJECTED (b)** — envelope-based reset (same trust-boundary reasoning as v0.1).

**§8.1 `sessions.agent` gate under OPTION 3.2.b:** currently the gate tests for literal `'pi'`. Under SOFT SHIM, a label registered as `--agent codex` or `--agent gemini` is ALSO backed by a pi-routed spawn. Two shapes to choose:

- **Shape α (NARROW):** reset file-delete only when `sessions.agent == 'pi'`. Labels registered as codex/gemini under SOFT SHIM don't get the file-delete even though the daemon wrote a session file at `$pi.session_dir/$label.jsonl`. Result: orphan files accumulate on reset of shim-backed labels.
- **Shape β (WIDE):** reset file-delete when `sessions.agent IN ('pi', 'codex', 'gemini')`. Covers shim-backed labels. Requires the reset verb to know about the shim mapping. Under HARD RETIRE in v0.4 this widening is harmless (no new rows with `agent='codex'`).
- **Shape γ (REGISTER-TIME):** SOFT SHIM silently writes `sessions.agent = 'pi'` even when operator passed `--agent codex` at register time. Cleanest runtime story but lossy at audit (the audit record says "pi" even though operator intended "codex").

**Coder recommendation: Shape β (WIDE).** Operator audit intent is preserved (`sessions.agent` reflects the `--agent` flag passed); file-delete covers shim-backed labels correctly; post-HARD-RETIRE in v0.4 the widening is vestigial but harmless. Reviewer challenge pending.

### 3.4 Normative constraints — pi-reading inherited

All 5 invariants from v0.1 §3.4 (as re-read in v0.2 §3.4) apply unchanged. Post-collapse, the invariants no longer have per-CLI specializations because codex + gemini collapse into the pi flow:

1. Envelope payload load-bearing per batch — spawnPi always passes full prompt.
2. Daemon MUST NOT introspect session-store contents — daemon owns the path, not the JSONL contents.
3. Reset MUST be idempotent — `os.Remove` is NotExist-tolerant.
4. Claude + flag non-fatal — claude asymmetry preserved (unchanged from v0.1).
5. Capture-failure non-fatal — pi persist failure → orphan-until-next-persist; next batch retries.

No new invariants for v0.3. The collapse simplifies enforcement (one code path) without changing contract surface.

## 4. CLI specifics

### 4.1 codex — RETIRED (OPTION 3.2.b mechanism)

Pre-v0.3: `--cli=codex` dispatched to `spawnCodex` with regex-based banner capture + stale-UUID fallback + `codex exec resume --skip-git-repo-check <UUID> <prompt>` argv.

Post-v0.3 under SOFT SHIM: `--cli=codex` is internally rewritten to `--cli=pi --pi-provider=openai-codex --pi-model=<inferred or operator-supplied>`. `spawnCodex` is deleted. `spawnCLI` dispatch loses the `case cliCodex` branch.

**Model-resolution under shim:** the shim needs a `pi.model` to pass to pi. Two options:
- **Operator-supplied:** the shim accepts `--pi-model` alongside `--cli=codex` and fails fast if absent.
- **Default-to-latest:** daemon picks a reasonable openai-codex default (e.g., `gpt-5.3-codex`). Operators who want a different model override via `--pi-model`.

Coder lean: operator-supplied (no default model). Forces operators to make the provider+model coupling explicit during migration. Aligns with v0.2's MANDATORY-both §4.4 coupling check.

**Deprecation warning:** daemon startup emits:
```
--cli=codex is routed through pi as of v0.3 (codex-direct retired per v3.x-topic-3-v0.3-scoping §4.1). Equivalent explicit config: --cli=pi --pi-provider=openai-codex --pi-model=<your-choice>. --cli=codex will be removed in v0.4.
```

### 4.2 gemini — RETIRED (OPTION 3.2.b mechanism)

Pre-v0.3: `--cli=gemini` dispatched to `spawnGemini` with `--list-sessions` delta-snapshot + UUID→index translation + race-warn-and-NULL + 15-second timeout with env override.

Post-v0.3 under SOFT SHIM: `--cli=gemini` rewritten to `--cli=pi --pi-provider=google-antigravity --pi-model=<operator-supplied>`. `spawnGemini` + `runGeminiListSessions` + `parseGeminiListSessions` + `lookupGeminiSessionIndex` + related helpers deleted. `spawnCLI` dispatch loses the `case cliGemini` branch.

**Provider mapping caveat:** google-antigravity is pi-mono's routing name for the provider that serves Google Gemini models (and also routes Anthropic Claude models via Google's hosting per the earlier `pi --list-models` output showing `google-antigravity claude-opus-4-5-thinking`). An operator running `--cli=gemini` pre-v0.3 was using the Google Gemini CLI directly; mapping to `google-antigravity` preserves the provider identity but the argv shape changes completely (pi-routed, not direct-CLI).

**Deprecation warning:** daemon startup emits:
```
--cli=gemini is routed through pi as of v0.3 (gemini-direct retired per v3.x-topic-3-v0.3-scoping §4.2). Equivalent explicit config: --cli=pi --pi-provider=google-antigravity --pi-model=<your-choice>. --cli=gemini will be removed in v0.4.
```

### 4.3 claude — UNCHANGED (preserves v0.1 §4.3 asymmetry)

`--cli=claude` continues to dispatch to `spawnClaude` with its existing `claude -p --settings <path> <prompt>` argv and claude-CLI-native session management. The v0.1 asymmetry (no cross-process session-resume) stays documented.

**Rationale for preservation (per v0.3 pre-draft spot-probe findings):**
- `pi --provider anthropic` requires separate ANTHROPIC_API_KEY env or `pi /login` OAuth — operators currently using claude-direct have claude-CLI-native credential management (subscription tokens, work/personal account switching, MCP server bindings).
- Collapsing claude-direct to pi-anthropic would force operators into a credential-provenance migration that isn't part of the codex/gemini collapse story.
- The v0.1 asymmetry framing ("Claude has no cross-process session-resume") stops being a property of the claude CLI once claude routes through pi (pi owns --session PATH, gaining cross-spawn context for claude). That's an upgrade worth scoping — but it belongs in v0.4+, not v0.3.

**claude-via-pi-anthropic is NOT retired; it was never launched.** Operators who want pi-anthropic can still configure `--cli=pi --pi-provider=anthropic --pi-model=claude-sonnet-4-5` directly; v0.3 documents that path in the operator guide as an alternative to claude-direct.

### 4.4 pi — PROMOTED TO SOLE NON-CLAUDE PATH (inherits v0.2 §4.4 unchanged)

pi's spawn forms, required-field validation, model-provider coupling check, session-file mint/reuse/reset behavior all carry forward from v0.2 §4.4 with zero changes. v0.3 expands pi's role (now the sole non-claude path) without modifying pi's behavior.

## 5. Schema

**No migration needed.** v0.2's `daemon_cli_session_id TEXT` column (from migration 0003) continues to serve both pi-routed workflows (path values) and — if `sessions.agent` for legacy codex/gemini labels still exists — UUID values. Under SOFT SHIM new rows will be written with path values; existing UUID-shape rows from pre-v0.3 codex/gemini labels stay on disk until operator resets them.

**UUID-shape-value cleanup:** optional one-shot SQL documented in operator guide:
```sql
UPDATE sessions SET daemon_cli_session_id = NULL WHERE agent IN ('codex', 'gemini') AND daemon_cli_session_id IS NOT NULL AND daemon_cli_session_id NOT LIKE '%/%';
```
Shape-heuristic (UUIDs lack `/`) cleanly distinguishes pre-v0.3 values from post-v0.3 paths. Runs once during operator migration; not required (shim tolerates stale UUIDs in the column by simply re-minting on next spawn).

GOOSE_VERSION_REQUIRED stays at 3. No new tables.
