# Peer-Inbox Push Asymmetry — Research Scaffold

**Status:** research / investigation doc · **Owner:** reason (synthesizer) · **Track C of round 2 post-W1 closure** · **Contributors needed:** gemini-peer, idle-birch (codex) — see §4 · **Convergence bar:** none yet — scaffold first, synthesis after non-Claude input

> This is a scaffold, not a recommendation. Mediator's owner-directive: "we can't solve the gemini/codex asymmetry with only Claude reasoning about it." Claude-side analysis here is explicitly provisional — any claim gemini-peer or idle-birch contradicts is wrong, not right.

---

## 1. Problem statement

Claude peers have turn-boundary hook delivery and (optionally) mid-turn channel push. Codex and Gemini peers have neither. Non-Claude sessions are effectively polling-only, and the polling is human-mediated.

This breaks the heterogeneous-agent value proposition: agent-collab exists so that different model families can review each other. If only Claude-to-Claude rounds converge in real time, the challenger-from-a-different-model safeguard degrades to "eventually, if the human remembers."

## 2. Evidence of the gap

- `idle-birch` (codex) went silent during round 2 of the v3.0 W1 review — no notification path fired on broadcast pushes.
- `gemini-peer` shows `(no-channel)` on every push; `agent-collab peer broadcast` explicitly logs it every time.
- The `dusty-island-8114` pair DB has 7+ stale gemini sessions accumulated across recent days — re-registration pattern consistent with sessions going dark and users starting fresh rather than reviving.
- Mediator's round-2 synthesis proceeded without idle-birch when timing forced a ruling; the review round's independence bar was measurably lower as a result.

## 3. Known architecture across all three CLIs (post-R1 + R2 evidence)

**Turn-boundary delivery: parity exists today across Claude, Codex, and Gemini.**

- **Claude Code** — `hooks/peer-inbox-inject.sh` runs as the `UserPromptSubmit` hook. On turn boundary, the bash script invokes `peer-inbox-hook` (Go, v3.0+) or Python fallback, which emits `{"hookSpecificOutput": {"additionalContext": "<peer-inbox>..."}}`. Claude Code injects into the turn's context.
- **Codex CLI** — `UserPromptSubmit` hook published in Codex CLI hooks docs (R1 gamma). Same `additionalContext` contract as Claude's. Five hook events total: `SessionStart`, `PreToolUse`, `PostToolUse`, `UserPromptSubmit`, `Stop`. Source: [Codex CLI Hooks](https://developers.openai.com/codex/hooks).
- **Gemini CLI** — `BeforeAgent` hook shipped in **v0.26.0+** (R2 alpha; the earlier "planned for v1.6" framing was outdated). Fires "after a user submits a prompt, but before the agent begins planning." `hookSpecificOutput.additionalContext` is "Text appended to the prompt for this turn only" — identical contract to Claude's `UserPromptSubmit`. Configured in `.gemini/settings.json` or `~/.gemini/settings.json`. Source: [Gemini CLI Hooks](https://geminicli.com/docs/hooks/).

**Mid-turn push: Claude-only today; known gap on Codex + Gemini.**

- **Claude Code** — MCP server `peer-inbox` loaded via `claude --dangerously-load-development-channels server:peer-inbox` (preview feature). Enables server-initiated push into the active context mid-turn.
- **Codex CLI** — MCP `notifications/message` arrives but is routed to Rust tracing logs in `codex-rs/rmcp-client/src/logging_client_handler.rs` (stderr/debug only, model never sees them). Open feature request [issue #18056](https://github.com/openai/codex/issues/18056) proposes injecting the notification data into active conversation (~50 LOC, ~3-4 files). No maintainer response yet; patch surface is small. Third-party precedent [agent-peers-mcp](https://github.com/Co-Messi/agent-peers-mcp) cited in the issue.
- **Gemini CLI** — MCP server notifications support requested in [issue #3052](https://github.com/google-gemini/gemini-cli/issues/3052) (2025-07-02). **Closed as Stale**, labeled `priority/p3 (Backlog)`. No implementation merged; no roadmap commitment.

**Implication — the real gap is narrower than the scaffold's opening framing suggested.** Turn-boundary parity is a today-problem, not a future-problem. Mid-turn push is the actual asymmetry, and even there the Codex gap has a known bounded patch.

### 4.3 Cloud-hosted integration — partially resolved (R3 test-engineer, 2026-04-18)

**MCP spec does support server-initiated push.** Streamable HTTP transport (MCP spec 2025-06-18) defines two paths for unsolicited server→client:
- **Client POST → SSE response stream** — server MAY send unprompted JSON-RPC messages before the POST response.
- **Client GET → dedicated SSE listener** — server MAY send requests/notifications "unrelated to any concurrently-running JSON-RPC request." This is the unprompted-push channel.
- Spec-defined notification types: `notifications/tools/list_changed`, `notifications/resources/list_changed`, `notifications/resources/updated` (subscribe-based), `notifications/prompts/list_changed`, `notifications/progress`. Resumability via `Last-Event-ID`.

Source: [MCP spec 2025-06-18 Transports](https://modelcontextprotocol.io/specification/2025-06-18/basic/transports)

**The constraint isn't the wire format — it's client-side surfacing.** Codex client drops notifications to logs (R1 gamma); Gemini client has no surfacing path (R2 alpha, #3052 closed). A cloud-hosted MCP server with push semantics works for Claude today via `notifications/claude/channel`, but non-Claude clients won't surface it regardless of transport.

**Closest prior art: [mcp_agent_mail](https://github.com/Dicklesworthstone/mcp_agent_mail)** (R3 test-engineer). Near-identical problem space (AI coding agents + inbox + threads + leases, supports 7 CLIs including Claude Code / Codex / Gemini / Factory Droid / Cursor / Cline / Windsurf). Delivery is "HTTP-only FastMCP server (Streamable HTTP). No SSE, no STDIO" — agents poll via `fetch_inbox()` or read `resource://inbox/{agent}`. **A project that spans 7 CLIs deliberately chose polling over push**, corroborating the client-surfacing-is-unreliable-across-CLIs finding.

**Still unknown — gemini-peer + idle-birch's ground-truth lane:**
- Integration shape preference from each CLI's side (HTTP SSE / WebSocket / webhook / REST polling)?
- CLI-side limits on long-lived connections, background tasks, outbound ports?
- Auth model fit from each CLI's perspective?

## 6. Crosscuts and constraints

> **Note (post-synthesis supersession per alpha m1):** the latency-budget and cloud-option bullets below are the scaffold's original framing pre-evidence. §8.5 dissolves cloud as a current path (three-way hook parity + client-surfacing-is-the-blocker); §8.4 reframes Option C as best-effort Gemini workaround not a latency-pick. Keep §6 for the scaffold's structural view; cite §8 for the evidence-backed current reading.

- **Security / privacy.** Peer-inbox messages routinely contain code fragments, file paths, in-progress review content. Cloud options need auth + transit encryption + redaction before they are safe — creates a dependency on v3.1 write-time redaction, the same gate v3.2 dashboard sits behind.
- **Reliability.** Whatever channel we pick must fail-open: the non-Claude CLI's startup or turn must never block on peer-inbox being reachable. Hook fail-open discipline (v3.0 ratified) extends to whatever replaces or supplements it.
- **Operational cost.** Cloud options introduce ongoing hosting burden we don't have today. Order-of-magnitude: a minimal relay on a small cloud VM or serverless is ~$5–20/month; a real service with monitoring is more.
- **Latency budget.** Turn-boundary delivery tolerates seconds. Mid-turn delivery wants sub-second (otherwise the agent replies before seeing the message). Different options have different floors here — option F over SSE is plausibly ~200ms; option C over tool-call-pull is one-tool-call-of-latency. *(Superseded by §8 — Options F/E dissolved, Option C reframed as workaround.)*
- **Upstream coupling.** D is pure "wait"; A/B/C/E/F/G/H/I are "own it." Hybrid paths (e.g., C now, migrate to D when upstream lands) reduce the risk of locking in a permanent workaround.
- **Heterogeneity is the goal.** Any option that ends up de-facto Claude-only (or requires non-Claude peers to emulate Claude's hook behavior) fails the core value test. Cloud options are attractive partly because they're natively model-agnostic.

## 8. Synthesis (R4b deferred; R1 + R2 + R3 + R4a complete)

### 8.1 Current-state diagnosis (evidence-backed, not inferred)

All three CLIs — Claude Code, Codex CLI, Gemini CLI — expose an equivalent **turn-boundary hook** with an **identical `hookSpecificOutput.additionalContext` contract** (R1 gamma, R2 alpha, R4a gemini-peer). Claude's `UserPromptSubmit`, Codex's `UserPromptSubmit`, and Gemini's `BeforeAgent` all accept the same JSON envelope shape and inject the result into the next turn. This is a shipped, documented capability on every target CLI today.

**Pre-turn asymmetry is self-inflicted** — peer-inbox ships only Claude's hook script today; Codex and Gemini have never had the equivalent installed. When they polled, they polled human-mediated; when they didn't, they went dark. Installing per-CLI hooks closes this lane.

**Mid-turn asymmetry is architectural, not self-inflicted** (gamma C1 correction). Codex is model-blind to MCP `notifications/message` by design — its `rmcp-client` routes them to tracing macros per R1 gamma's finding #2. Gemini has no surfacing path at all — #3052 closed-stale. Installing hooks doesn't fix this layer; fixing it requires either upstream patches (Option K for Codex) or accepting that non-Claude CLIs lag mid-turn.

The operational delta the room observed (idle-birch going silent during round 2; gemini-peer always `(no-channel)` on pushes; 7+ stale gemini sessions) is dominantly a pre-turn (install-state) problem with a smaller mid-turn (architectural) residual. Option J closes the big one.

**Mid-turn push is the only genuinely asymmetric capability.** Claude has it via the `notifications/claude/channel` MCP feature. Codex has the wire capability but routes notifications to logs — [#18056](https://github.com/openai/codex/issues/18056) proposes a ~50 LOC fix, open + unmerged. Gemini has neither surfacing — [#3052](https://github.com/google-gemini/gemini-cli/issues/3052) closed as Stale, priority/p3. R3 test-engineer's industry precedent ([mcp_agent_mail](https://github.com/Dicklesworthstone/mcp_agent_mail) across 7 CLIs) independently corroborates: the dominant pattern is polling + hooks, and projects that span multiple CLIs deliberately design around unreliable client-side notification surfacing.

### 8.2 Short-term path (v3.1 candidate): Option J — port hooks to Codex + Gemini

**Recommendation: ship in v3.1.**

Scope (estimated hours to low-single-digit days per gamma refinement; ~95% script portability per R4a + alpha R2 per-CLI estimate):

1. **Unified hook script with stdin-schema detection — NOT three separate files** (alpha C1, endorsed by gamma + test-engineer). Three parallel scripts would duplicate ~95% of the inbox fetch/format logic and recreate exactly the constants-drift class W1 just fixed on UTF-8 bytes (B1, B8). One script branching on stdin-schema — or a thin per-CLI adapter sourcing a shared core — is the obviously-right shape given we've seen this drift class bite twice. Taking the position rather than punting to coder. Sketch:
   - Top of script detects the CLI from stdin JSON shape (`session_id`+`cwd` → Claude; `prompt`+`turn_id` → Codex; `prompt`+`history` → Gemini) or an explicit env var set by the hook registration (simpler).
   - Shared core handles DB lookup + block formatting + stdout envelope.
   - Single source of truth for `SESSION_KEY_ENV_CANDIDATES` ordering, `HOOK_BLOCK_BUDGET`, and the fetch path.
2. **Install-script extension.** `scripts/install-global-protocol` currently writes only `~/.claude/CLAUDE.md`. Extend to also write hook config snippets into `~/.codex/hooks.json` and `~/.gemini/settings.json` when the respective CLIs are detected. Alpha m2 flag: verify the existing CLI-detection block in the installer is the seam it appears to be before banking the "small" scope estimate — if absent, adding detection is part of the work (still bounded; just not free).
3. **Canonical stdin fixtures (Option J prerequisite).** As part of the J port, whoever adapts the hooks produces `tests/fixtures/hook-stdin/{claude,codex,gemini}.json` — one fixture per CLI capturing the hook-event JSON each CLI passes on stdin (Claude: `session_id`+`cwd` shape; Codex: `prompt`+`turn_id` per R1 gamma, requires live-checkout verification; Gemini: `prompt`+`history` per R4a gemini-peer). These fixtures couple the J implementation to its own regression gate and unblock the test-authoring in (4).
4. **Test surface.** Test-engineer authors `tests/hook-parity.sh` at **shape 2 — hook-script parity**: feed each CLI's stdin fixture to its per-CLI hook script and assert byte-identical `hookSpecificOutput.additionalContext` on stdout across the three. Shape 3 (contract-shape parity — asserting `hookEventName` + `additionalContext`-matches-`formatHookBlock` without CLI-real fixture accuracy) is the fallback if stdin schemas prove hard to pin. Shape 1 (end-to-end CLI boot + model-call scaffolding) is explicitly out of scope for v3.1 — costs more than it returns and CI would need all three CLIs installed with login tokens. Per test-engineer's lane framing.
5. **Documentation.** Update `docs/PEER-INBOX-GUIDE.md` and `CLAUDE.md` equivalents for each CLI to document hook installation as standard setup.

**Why Option J over the alternatives:**
- **A (accept asymmetry):** obsolete once we know hooks work identically; we'd be accepting a self-inflicted gap.
- **B (sidecar poll):** hooks do this better with less code and no human-in-the-loop.
- **C (`peer_inbox_receive` MCP tool):** useful as a complement for on-demand inbox check, not as the primary delivery channel. Still a v3.1+ candidate for its own narrow purpose.
- **E/F/G (cloud-hosted):** don't improve on local hooks for turn-boundary delivery and don't help with mid-turn on non-Claude CLIs (client surfacing, not transport, is the blocker).
- **H (local sidecar daemon):** subsumed by hook approach.

Gemini-peer explicitly endorsed Option J as "the 80/20 win that solves the 'agent goes dark' problem." No reviewer argued against.

### 8.4 Long-term path (v3.3 / v4): Gemini mid-turn — best-effort workaround, not Claude-parity

**Recommendation: accept the gap explicitly; ship Option C as a best-effort workaround; name its known UX failure mode honestly.**

Gemini #3052 closed as Stale, priority/p3. Even an upstream PR here is unlikely to land without maintainer alignment that doesn't exist today. The practical workaround for Gemini mid-turn is Option C (`peer_inbox_receive` MCP tool) as an opt-in: a Gemini agent that knows it's in a mediated round calls the tool between tool-calls to pick up mid-turn messages.

**Concrete cueing mechanism** (substance per alpha C2 + gamma Challenge 2):

- **`SessionStart` hook** primes the agent's mental model on session entry: injects text like "You are in a peer-inbox mediated round. If you're working through a multi-step plan, call `peer_inbox_receive` between tool calls so mid-turn messages don't stall the room." Uses Gemini's SessionStart lifecycle hook per R2 alpha.
- **Tool description copy** on `peer_inbox_receive` explicitly instructs frequency: "Call this every ~3-5 tool calls if you're in a mediated round, or after any tool that takes >30s." Makes the frequency visible to the agent's tool-selection heuristic.
- **Session-register-time opt-in:** `agent-collab session register --mid-turn-cue` adds the prompt insertion and tool description only when mid-turn delivery is desired — avoids noise for non-mediated pair sessions.

**Honest framing — this is a polling workaround, not Claude-parity** (gamma C2, alpha C2 withdrew to this):

- Under deep-focus turns (Gemini working through a 10-step plan with visible progress), the "check inbox" tool call is a context-switch tax the agent's own task-prioritization will deprioritize. R3's industry-prior-art finding (mcp_agent_mail chose polling specifically because client-side notification surfacing is unreliable) applies in the mirror-image here — *self-polling* is unreliable when it competes with task progress.
- Under shallow turns or natural break points (between major tool calls, after a user question), cueing reliably fires. So the mitigation is **best-effort on Gemini, not full mid-turn parity**.
- Document this as a known limitation, not a silent feature gap.

Flag for owner visibility: if mid-turn parity becomes a hard requirement on Gemini (e.g., high-throughput review rounds where laggy Gemini peers reliably block convergence), the only paths are (a) wait for upstream (unbounded timeline), or (b) cloud-hosted MCP server with a Gemini-side proxy — which depends on redaction gate (v3.1) and has operational cost.
