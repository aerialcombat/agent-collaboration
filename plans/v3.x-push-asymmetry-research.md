## 1. Problem statement

Claude peers have turn-boundary hook delivery and (optionally) mid-turn channel push. Codex and Gemini peers have neither. Non-Claude sessions are effectively polling-only, and the polling is human-mediated.

This breaks the heterogeneous-agent value proposition: agent-collab exists so that different model families can review each other. If only Claude-to-Claude rounds converge in real time, the challenger-from-a-different-model safeguard degrades to "eventually, if the human remembers."

## 2. Evidence of the gap

- `idle-birch` (codex) went silent during round 2 of the v3.0 W1 review — no notification path fired on broadcast pushes.
- `gemini-peer` shows `(no-channel)` on every push; `agent-collab peer broadcast` explicitly logs it every time.
- The `dusty-island-8114` pair DB has 7+ stale gemini sessions accumulated across recent days — re-registration pattern consistent with sessions going dark and users starting fresh rather than reviving.
- Mediator's round-2 synthesis proceeded without idle-birch when timing forced a ruling; the review round's independence bar was measurably lower as a result.

## 3. Known Claude-side architecture (provisional — may be incomplete)

- `hooks/peer-inbox-inject.sh` runs as the `UserPromptSubmit` hook. On turn boundary, the bash script invokes `peer-inbox-hook` (Go, v3.0+) or the Python fallback, which emits a `{"hookSpecificOutput": {"additionalContext": "<peer-inbox>..."}}` envelope that Claude Code injects into the turn's context.
- MCP server `peer-inbox` is loaded via `claude --dangerously-load-development-channels server:peer-inbox`. This is a Claude-Code-specific preview feature ("channels") that lets an MCP server push messages into the active context mid-turn, without waiting for a prompt boundary.
- Both mechanisms are Claude-Code-runtime-specific. Neither is portable to Codex or Gemini as-is.

## 5. Options surface (seed — expand as research lands)

### 5.1 Local-only options

- **A.** Accept asymmetry. Formalize protocol: "Claude peers converge first; non-Claude peers catch up when they re-engage." Zero work, codifies the current degradation.
- **B.** Sidecar `peer watch --only-new` in Codex/Gemini terminals — output surfaces to the human operator, who shuttles context to the agent on next prompt. ~half-day. Band-aid; still human-in-the-loop.
- **C.** Add a `peer_inbox_receive` MCP tool to the existing peer-inbox server. Codex/Gemini agents call it when their loop decides to check. ~2–3 days. Works today on both; requires system-prompt guidance to keep the agent actually calling it.
- **D.** Wait for upstream hooks (Gemini `BeforeAgent`, Codex TBD). No work by us; timing unknown; current gap persists.

### 5.2 Cloud-hosted options (newly in scope per mediator scope update)

- **E.** Cloud-hosted MCP server with push semantics. Non-Claude CLIs subscribe via whatever push their MCP client supports. Contingent on non-Claude MCP clients supporting server push at all — currently unclear.
- **F.** Cloud-hosted HTTP/SSE relay. Non-Claude agents either poll a REST endpoint or hold an SSE stream open. Works regardless of MCP push support, because it's transport-agnostic.
- **G.** Hosted agent-collab daemon as a service. Exposes a tool API the non-Claude agent calls to receive; push happens daemon-side via OS notifications or tool-call injection.

### 5.3 Hybrid options

- **H.** Local sidecar daemon on the dev machine (per-user) that bridges peer-inbox to whatever push the local CLI supports. Can be cloud-extended later for cross-machine use. Lowest privacy exposure of the push-capable options.
- **I.** Redacted-cloud hybrid: local daemon handles pair state; cloud relay carries redacted-only payloads (dependent on v3.1 write-time redaction landing first — same gate as v3.2 dashboard).

## 6. Crosscuts and constraints

- **Security / privacy.** Peer-inbox messages routinely contain code fragments, file paths, in-progress review content. Cloud options need auth + transit encryption + redaction before they are safe — creates a dependency on v3.1 write-time redaction, the same gate v3.2 dashboard sits behind.
- **Reliability.** Whatever channel we pick must fail-open: the non-Claude CLI's startup or turn must never block on peer-inbox being reachable. Hook fail-open discipline (v3.0 ratified) extends to whatever replaces or supplements it.
- **Operational cost.** Cloud options introduce ongoing hosting burden we don't have today. Order-of-magnitude: a minimal relay on a small cloud VM or serverless is ~$5–20/month; a real service with monitoring is more.
- **Latency budget.** Turn-boundary delivery tolerates seconds. Mid-turn delivery wants sub-second (otherwise the agent replies before seeing the message). Different options have different floors here — option F over SSE is plausibly ~200ms; option C over tool-call-pull is one-tool-call-of-latency.
- **Upstream coupling.** D is pure "wait"; A/B/C/E/F/G/H/I are "own it." Hybrid paths (e.g., C now, migrate to D when upstream lands) reduce the risk of locking in a permanent workaround.
- **Heterogeneity is the goal.** Any option that ends up de-facto Claude-only (or requires non-Claude peers to emulate Claude's hook behavior) fails the core value test. Cloud options are attractive partly because they're natively model-agnostic.
