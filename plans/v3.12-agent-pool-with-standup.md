# v3.12 — agent pool + standup room

**Status:** DRAFT v1 · **Authored:** 2026-04-26 · **Author:** claude (lead-drafting at owner request) · **Trigger:** owner conversation 2026-04-26 — *"i want to be able to add agents to a pool of agents. an agent can be a certain agent based on either claude or pi. the pool can contain any number of agents, and the kanban can be drained by a pool of agents. and the assignment can be either designated or auto-selected. … there can be a standup-like chat session for selecting agents for cards. there can be a running chatroom where agents can gather to select cards."* · **Positions as:** the user-facing realization of "Phase 4 — matchmaker" from `v3.11-cards-phase1-standing-drainer.md` §10, expanded to include chat-mediated selection and persistent agent sessions. v3.12 is **server-local**: workers live on the same host as the kanban DB. Cross-host (original Phase 5/6) is explicitly deferred.

> **Scope discipline:** v3.12 introduces an agent registry, per-board pools, persistent agent sessions, and chat-mediated dispatch. It does **not** introduce cross-host workers, federated card-list aggregation, deferred auto-promote, or cancellation UI. Those remain future work.

## 1. Goal

Let the operator define a roster of agents (each backed by claude or pi), assign rosters to kanban boards, run those agents as persistent server-resident sessions, and have them coordinate card pickup via a dedicated standup chat room — falling back to algorithmic auto-dispatch when chat is silent. The two mediums (kanban for durable work, peer-inbox for volatile coordination) stay deliberately separate; the standup room is where agents *talk about* kanban, not where the kanban lives.

**Owner constraint verbatim (2026-04-26):**
> *"i want to be able to add agents to a pool of agents. an agent can be a certain agent based on either claude or pi. the pool can contain any number of agents, and the kanban can be drained by a pool of agents. and the assignment can be either designated or auto-selected."*
>
> *"there should be a session dedicated for the kanban board, but is abled to gain more context from cards from references."*
>
> *"there can be a standup-like chat session for selecting agents for cards. there can be a running chatroom where agents can gather to select cards."*
>
> *"each board different pool."*

**Success criteria.**

1. Operator can manage a global agent registry (CRUD) and per-board pools (membership with replication count + priority).
2. Each enabled (board, agent) membership corresponds to a long-lived headless session running on the server, joined to the board's standup room.
3. When a card becomes ready, an event posts into the standup room with the card body + resolved `context_refs`. Any agent in the room can self-claim via the existing `card-claim` verb.
4. If no agent claims within a configurable claim-window, the drainer auto-dispatches to the agent that best matches `needs_role` and has the lowest in-flight count, with `priority` as tiebreak.
5. Every dispatch records `agent_id` on `card_runs`, alongside the existing `host` and `worker_label`.
6. The standup room and the kanban board remain distinct mediums: card events broadcast into the standup, but the standup never *replaces* card state.

**What v3.12 does NOT do.**

- No cross-host workers (deferred — was original Phase 5/6).
- No federated card-list across multiple servers (deferred).
- No cancellation UI (deferred).
- No deferred auto-promote race fix (deferred).
- No agent-to-agent direct DM workflows beyond the standup room (deferred).
- No automated review / lead-challenger handoff at `in_review` — deferred to v3.12.6 sub-phase, designed but optional.

## 2. Architectural shape

```
v3.11 Phase 1 (now)                v3.12 (target)
──────────────────                 ──────────────────
auto_drain ticker                  agent registry  ← global
  │                                  │
  ▼                                  ▼
SELECT ready+unclaimed             pool_members   ← per-board roster
  │  cards                           │
  ▼                                  ▼
spawn one worker via              for each enabled (pair_key, agent):
AGENT_COLLAB_WORKER_CMD             systemd template unit
  │                                   running long-lived peer-inbox session
  ▼                                   joined to {pair_key}-standup room
goroutine waits for exit             │
                                     ▼
                                   ticker (per board):
                                   1. card → ready
                                      → post into {pair_key}-standup
                                      "📋 #N ready: [title] · ctx: [N refs]"
                                   2. wait claim_window_secs
                                   3. if any agent self-claimed via chat:
                                      drainer DM's that agent the context bundle
                                   4. else: auto-dispatch
                                      filter pool by needs_role
                                      pick agent: lowest in-flight, priority tiebreak
                                      DM that agent the context bundle
                                   5. record card_runs.agent_id
                                   6. monitor as Phase 1; on terminal status
                                      post outcome back to standup
```

**Key invariant introduced in v3.12:** *the registry/pool tables describe the desired roster; systemd reconciles the actual running sessions; the drainer routes work by reading both pool config and chat claims.* Adding/removing pool members is a config change that triggers a unit start/stop, not a code change.

## 3. Schema migration: `0012_agents_and_pool.sql`

One migration, two tables, two columns.

```sql
-- +goose Up

-- Global agent registry. An "agent" is a config row, not a process.
-- Processes are systemd units that consume rows from this table.
CREATE TABLE agents (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  label           TEXT NOT NULL UNIQUE,        -- e.g. claude-impl, pi-review-1
  runtime         TEXT NOT NULL,                -- 'claude' | 'pi'
  worker_cmd      TEXT,                         -- argv override; empty = runtime default
  role            TEXT,                         -- matches cards.needs_role
  model_config    TEXT,                         -- JSON: model, max-turns, env additions
  enabled         INTEGER NOT NULL DEFAULT 1,
  created_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  updated_at      TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  CHECK (runtime IN ('claude','pi'))
);

CREATE INDEX idx_agents_role    ON agents(role)    WHERE role IS NOT NULL;
CREATE INDEX idx_agents_enabled ON agents(enabled) WHERE enabled = 1;

-- Per-board pool membership. (pair_key, agent_id) is unique;
-- count > 1 lets a single agent definition fill N parallel slots.
CREATE TABLE pool_members (
  pair_key   TEXT    NOT NULL,
  agent_id   INTEGER NOT NULL REFERENCES agents(id) ON DELETE CASCADE,
  count      INTEGER NOT NULL DEFAULT 1,
  priority   INTEGER NOT NULL DEFAULT 0,
  added_by   TEXT,
  added_at   TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
  PRIMARY KEY (pair_key, agent_id),
  CHECK (count >= 1)
);

CREATE INDEX idx_pool_members_pair_key ON pool_members(pair_key);

-- Designated assignment. NULL = use auto-select.
ALTER TABLE cards ADD COLUMN assigned_to_agent_id INTEGER
  REFERENCES agents(id) ON DELETE SET NULL;

-- Each run links back to the agent that ran it.
ALTER TABLE card_runs ADD COLUMN agent_id INTEGER
  REFERENCES agents(id) ON DELETE SET NULL;

CREATE INDEX idx_card_runs_agent_id ON card_runs(agent_id) WHERE agent_id IS NOT NULL;

-- +goose Down

DROP INDEX IF EXISTS idx_card_runs_agent_id;
ALTER TABLE card_runs DROP COLUMN agent_id;
ALTER TABLE cards DROP COLUMN assigned_to_agent_id;
DROP TABLE pool_members;
DROP TABLE agents;
```

## 4. Standup room

The standup room is a **dedicated peer-inbox room** named `{pair_key}-standup`. Distinct from any human-facing pair-room — agents coordinate there, humans optionally observe.

**Members.** All long-lived agent sessions for that board. Optionally the operator's own session for visibility.

**Message types posted by the system into the room:**
- **Card-ready announcement.** Posted when a card transitions to ready (no open blockers).
  ```
  📋 #42 ready · pair_key=foo · needs_role=impl
  Title: Add gravatar to user profile
  Context: 3 refs (2 files, 1 url)
  Claim with: peer-inbox card-claim --card 42 --label <your-label>
  ```
- **Claim acknowledgement.** Posted when an agent self-claims.
  ```
  ✅ #42 claimed by claude-impl-1
  ```
- **Outcome posted on terminal status.**
  ```
  ✓ #42 done by claude-impl-1 (rc=0, 2m31s)
  ```
- **Auto-dispatch fallback.** If no claim before the window expires:
  ```
  ⏱  #42 unclaimed after 30s — auto-dispatching to pi-impl-2 (lowest load)
  ```

**Why a separate room.** Memory `project_two_mediums_chat_and_kanban.md`: kanban + chat are deliberately separate. Reusing the board's pair-room would conflate human-design conversation with agent coordination chatter. Distinct rooms keep the discipline.

**Configuration.** Per-board: `claim_window_secs` (default 30) on `board_settings` (additive column or JSON blob — see §6.) `0` disables auto-dispatch fallback (chat-only mode).

## 5. Long-lived agent sessions

Each enabled `(pair_key, agent_id)` membership translates to a running headless session on the server.

**Process model.** A systemd template unit:
```
agent-collab-pool@<pair_key>-<agent_label>-<slot>.service
```
For a pool member with `count=3`, three units run as `slot=0,1,2`. Slots distinguish replicas of the same agent definition.

**Session bootstrap.** Each unit:
1. Reads its agent row from the registry.
2. Constructs argv from `worker_cmd` or runtime default.
3. Registers a peer-inbox session: `agent-collab session register --label <agent_label>-<slot> --agent <runtime> --role <role>`.
4. Joins the board's standup room: `agent-collab peer join --room {pair_key}-standup`.
5. Runs the agent runtime in a loop, processing inbound peer-inbox messages. Cards arrive as DMs from the system; standup posts arrive as room messages.

**Reconciliation.** A small Go program `agent-collab-pool-reconciler` runs on a 5s tick:
- Reads `agents` × `pool_members WHERE enabled=1`.
- Compares to systemctl unit state.
- Starts missing units, stops orphaned units.
- Logs additions/removals.

**Resource budget.** Each session is one Node.js process holding one runtime + one peer-inbox channel socket. Practical cap on a single server: ~10–20 idle agents, scaling down with active card load.

**Idle behavior.** Agents listen for peer messages; they don't poll the kanban. The system pushes via DM. So idle token cost is 0 — sessions only spend tokens when a card or chat message activates them.

## 6. CLI and MCP surfaces

### Agent registry (v3.12.1)
- `peer-inbox agent-create --label L --runtime {claude|pi} [--role R] [--worker-cmd "..."] [--config '{...}']`
- `peer-inbox agent-list [--enabled-only] [--format json|plain]`
- `peer-inbox agent-get --label L [--format json|plain]`
- `peer-inbox agent-update --label L [--enabled on|off] [--role R] [--worker-cmd "..."] [--config '{...}']`
- `peer-inbox agent-delete --label L`

MCP tools: `agent_create`, `agent_list`, `agent_get`, `agent_update`, `agent_delete`.

### Pool composition (v3.12.2)
- `peer-inbox pool-add --pair-key K --agent L [--count N] [--priority P] [--as LABEL]`
- `peer-inbox pool-remove --pair-key K --agent L`
- `peer-inbox pool-list --pair-key K [--format json|plain]`

MCP tools: `pool_add`, `pool_remove`, `pool_list`.

### Card assignment override
- `peer-inbox card-assign --card N --agent L` (sets `cards.assigned_to_agent_id`)
- `peer-inbox card-unassign --card N` (clears it)

MCP tools: `card_assign`, `card_unassign`.

### Settings extension
`board_settings` gains:
- `claim_window_secs` INTEGER NOT NULL DEFAULT 30 (or pack into a `pool_config` JSON column)
- `pool_dispatch_mode` TEXT NOT NULL DEFAULT 'auto' — one of `chat_only`, `auto`, `chat_then_auto`. Default `chat_then_auto` matches the design.

CLI: `peer-inbox board-set` gains `--claim-window-secs` and `--dispatch-mode`. Web UI gear modal grows two fields.

## 7. Web UI surfaces

- **`/agents`** — agent registry table; create/edit/delete; toggle enabled.

## 10. Backward-compat with Phase 1

- When a board's pool is empty, drainer falls back to `AGENT_COLLAB_WORKER_CMD` env (today's behavior). Existing harness `cards-phase1-drainer.sh` keeps passing without change.
- `card_runs.worker_label` (Phase 1) stays as a free-form display string. `card_runs.agent_id` (v3.12) is the structured FK. Both populated on every dispatch.
- `card_runs.host` (Phase 1) remains `self_host()` for v3.12 — all workers are server-local. The reservation continues to point toward future cross-host work.
- Existing `cards.needs_role` (0009) becomes the role filter for auto-dispatch. No new column.
- Phase 1's standing drainer remains the per-board ticker; v3.12 changes only what the ticker *does* when picking a worker.

## 11. Out of scope (explicit)

- **Cross-host workers.** Original Phase 5/6. v3.12 is server-local only.
- **Federated card-list aggregation.** A multi-server operator console.
- **Cancellation UI.** Original Phase 2. Carried in backlog.
- **Deferred auto-promote race fix.** Original Phase 3. Carried in backlog.
- **Standup room moderation / human override.** Operator can SSH and post into the room manually using existing `peer send`, but no purpose-built operator intervention UI.
- **Agent capability discovery.** Each agent's `role` is owner-declared, not introspected. If pi grows MCP servers for, say, image generation, the operator declares "image-gen" as a role; the system doesn't probe.
- **Multi-tenancy.** No per-user isolation; all agents share the agent-collab service user's home, env, and credentials.

## 12. Open questions (to revisit before v3.12.4)

- **Q12.A — claim_window_secs default.** 30s is plausible. Too short = chat never gets a turn; too long = queues stall. Worth tuning after v3.12.5 first run.
- **Q12.B — chat-claim format.** Today `peer-inbox card-claim` is a shell verb. For chat-mediated selection, the agent's persistent session needs to invoke this when it decides to take a card. Either: (a) the agent runtime calls the local CLI via shell, (b) the agent posts a structured chat message that a server-side bridge interprets. (a) is simpler, (b) decouples agent runtimes from system internals. Lean (a).
- **Q12.C — Per-agent context budget.** When the system DMs a card to an agent, it includes the resolved context bundle (card body + context_refs). Existing resolver caps are 200 KB/file, 2 MB total. With multiple cards in flight, do we want a per-agent context budget? Probably yes — Phase 1's resolver caps are for *one* card; the agent's session over time may accumulate many. Defer to v3.12.5.
- **Q12.D — How do agents discover their own pair_key?** Each session is started by a systemd unit that knows the (pair_key, agent_label, slot). The unit passes pair_key as an env var or CLI flag at session register. Agents don't auto-discover; the operator's pool-add is what creates the binding.
- **Q12.E — Operator's role in the standup.** Does the operator session join the standup by default? Useful for visibility; but adds noise. Default off; operator opts in via `agent-collab peer join`.
