> **Authorship note:** owner-requested lead-drafted first pass capturing a live panel debate (system-architect + backend-architect + business-panel-experts, 2026-04-21). Panel output drove the shape; see §7 for the provenance trail. Not yet ratified.

## 1. Goal + motivation

Today each host has its own SQLite `sessions.db` with no cross-host awareness. Owner wants two things, both without committing to a centralized hub:

1. **Bring a local session remotely** — when traveling, continue a room that was started on the laptop. Don't force all rooms to live on orange.
2. **Local agents talk to remote agents in one room** — a claude session on laptop and a claude session on orange coexist as peers.

**Owner motivation verbatim (2026-04-21):** *"i can make everything work on remote, which would be simpler. but i have projects locally, and local computer is faster to prototype with. i'll probably only use remote if i need to work remotely."*

**Why symmetric, not hub-and-satellite.** First framing pushed orange-as-always-on-hub to satisfy oozoo integration. Business panel (unanimous, six experts) flagged this as premature product-integration driving architecture. Stripping oozoo out of the decision, neither stated need requires a canonical hub — symmetric peer-to-peer routing covers both, survives either host being offline for the rooms that host doesn't own, and preserves optionality for how oozoo eventually integrates.

## 2. Core architecture decision

**Rooms declare home host at creation; home host is encoded in the room ID. No central registry.**

- Room IDs become `<host>:<slug>` — e.g. `orange:perky-dove-be3a`, `laptop:fluffy-beaver-bc6f`. Bare slug is display-only.
- CLI parses prefix, routes writes to home host's peer-web over tailnet HTTP.
- Both hosts expose peer-web on tailnet (laptop via `tailscale serve` when awake; orange permanently).
- Laptop-rooms survive orange outages; orange-rooms survive laptop sleeping. No split-brain — unreachable home host = unreachable room, full stop.

**What this forecloses, honestly:** offline symmetry on shared rooms. If you're on a flight with no tailnet, you can't send to any room whose home host you can't reach. Acceptable at 2–4 hosts; would push toward gossip (shape B) at more.

**What this preserves:** clean path to Postgres (E) later — swap orange's SQLite for Postgres behind the same `/api/send` and laptop doesn't know. The HTTP seam is the right seam.

## 4. Out of scope

- **Oozoo integration.** Business panel unanimous: defer. When ready to test "chat with my agents from mobile" as a job-to-be-done, build a throwaway bridge pointed at whichever host hosts the target agents. Don't let product integration drive federation architecture.
- **Offline-symmetric federation (shape B).** Gossip + vector clocks is weeks of real DB engineering; unjustified at 2–4 hosts.
- **Postgres migration (shape E).** HTTP seam preserves this path for free; adopt when actual multi-client contention appears, not before.
- **Room promotion / migration** (move a room from laptop-hosted to orange-hosted after creation). Flagged as nice-to-have; handle as immutable home for v3.3 and design promotion in a follow-on if real need emerges.
- **P2P direct laptop↔phone without either host as intermediary.** Different topology; separate cluster if ever.
- **Public-internet auth.** Tailnet remains the trust boundary. TLS-termination + auth for non-tailnet access is follow-on.
- **Federation across >2 hosts operationally hardened.** Design supports N, but v3.3 ships + smoke-tests with laptop + orange only.

## 5. Per-item impl surfaces

**Item 1 — Room ID host-qualification.**

- Schema migration: add `home_host TEXT NOT NULL` column to `rooms` table. Backfill existing rows with the host they were created on (`hostname`-based heuristic; owner confirms during migration).
- Display layer: `<host>:<slug>` when rendering room IDs at CLI/peer-web boundaries. Internal storage can keep slug + home_host separate.
- Parse helper: `parse_room_id(s) -> (host, slug)` with bare-slug fallback defaulting to local host for backwards compat during migration window.
- Invariant: once created, `home_host` is immutable. Room promotion (out-of-scope v3.3) would create a new room with aliased old ID.

**Item 2 — Laptop peer-web on tailnet.**

- Owner runs `tailscale serve --bg --https=443 http://localhost:8787` on laptop. Laptop becomes reachable at `https://laptop-name.tail48d298.ts.net/`.
- No code change; doc-only addition to `docs/PEER-WEB-GUIDE.md` under a new "Tailnet exposure" section mirroring the orange writeup.
- Operator caveat: laptop peer-web only responds when laptop is awake. Documented behavior, not a bug.

**Item 3 — Host-aware CLI routing.**

- `~/.agent-collab/remotes.json` format:
  ```json
  {
    "orange": {
      "base_url": "https://orange-1.tail48d298.ts.net",
      "auth_token_env": "AGENT_COLLAB_ORANGE_TOKEN"
    },
    "laptop": {
      "base_url": "https://laptop-name.tail48d298.ts.net",
      "auth_token_env": "AGENT_COLLAB_LAPTOP_TOKEN"
    }
  }
  ```
- CLI flow for `peer send --to X --message Y`:
  1. Resolve room ID from current context. Parse `<host>:<slug>`.
  2. If `host == $(hostname-self-label)`, write to local SQLite.
  3. Else, look up `host` in `remotes.json`, HTTP POST to `{base_url}/api/send` with bearer token.
  4. On network failure, enqueue to `pending_outbound` (item 5).
- Self-label per host: derive from `AGENT_COLLAB_SELF_HOST` env or hostname. Set by `install-global-protocol`.

**Item 4 — Server-seq + ULID.**

- Schema: `ALTER TABLE messages ADD COLUMN server_seq INTEGER; ALTER TABLE messages ADD COLUMN message_id TEXT UNIQUE;`
- `server_seq` assigned per-room via `(SELECT COALESCE(MAX(server_seq), 0) + 1 FROM messages WHERE room_id = ?)` inside the same transaction as insert. Sqlite's per-DB write serialization gives monotonicity for free.
- POST `/api/send` response: `{ "message_id": "<ulid>", "server_seq": N, "accepted_at": "<ts>" }`. Duplicate `message_id` on retry returns 200 with the original `server_seq`, not 409.
- Poll: `GET /api/messages?room=X&since_seq=N` — client sends `max(seen_seq, own_last_write_seq)` to guarantee read-your-writes.

**Item 5 — Outbound queue.**

- New table: `pending_outbound(id INTEGER PK, message_id TEXT UNIQUE, room_id TEXT, payload BLOB, created_at TIMESTAMP, attempts INTEGER DEFAULT 0, last_attempt_at TIMESTAMP)`.
- Flush trigger: every `peer send` attempt retries the queue first (FIFO per-room) before sending the new message. On first POST error, enqueue the new message and exit 0 with a stderr warning.
- TTL: rows > 24h old are dropped with a stderr warning on next flush. Don't silently replay ancient messages into rooms whose state has moved on.
- CLI visibility: `agent-collab peer queue --show` prints queue depth + oldest entry. Warning banner in normal `peer send` output when queue is non-empty.

**Item 6 — Host-qualified labels.**

- Messages table: `from_label` stored as `<host>:<label>`. Membership table: same.
- Registration (`session register --label beta`): reject duplicate `(host, label)` pairs locally. Cross-host duplicates allowed.
- Display: `render_label(host_qualified)` returns bare `label` unless the current room has members from multiple hosts with the same bare label, in which case `host:label`.
- Self-send: when laptop's `beta` sends to an orange-hosted room where orange also has a `beta`, both are first-class members with qualified labels; no identity merge attempted.

**Item 7 — Per-session bearer tokens.**

- Schema: `sessions` table already has `session_key`. Add `auth_token TEXT` — generated at `session register` time (256-bit random, base64url).
- Client stores token in `~/.agent-collab/sessions/<session_key>.json`. CLI loads on send, includes as `Authorization: Bearer <token>`.
- Server validates: `token → session_key → label`. Mismatch between claimed `from_label` and token's bound label = 403.
- Rotation: `session register` regenerates. Old token invalidated immediately.
- Threat model explicitly documented in `docs/PEER-WEB-GUIDE.md`: tailnet = transport trust, not identity; tokens defend against a compromised-but-authorized tailnet device.

**Item 8 — Federation smoke test.**

- `tests/federation-smoke.sh` runs on laptop, assumes orange reachable at its tailnet URL.
- Scenarios:
  1. Create `orange:smoke-A`, add laptop's `test-alpha` as member, laptop sends → assert orange `/api/messages` returns it.
  2. Orange agent sends to same room, laptop polls → assert received.
  3. `systemctl stop agent-collab-peer-web` on orange, laptop sends → assert queued (exit 0, stderr warning, queue depth +1).
  4. Restart orange peer-web, laptop sends again → assert queued message flushes first, new message follows.
  5. `laptop:smoke-B` room, orange member, laptop sleeps (simulated via `tailscale down`) → assert orange's attempts queue; laptop returns → assert flush.

## 6. Open questions + sequencing

- **Migration fallout for existing rooms.** Owner has rooms in both laptop.db and orange.db today. Item 1's backfill is straightforward (each host stamps its own rooms with its own hostname) but cross-host rooms that should've been shared have to stay separate — there's no magic merge. Document this in the migration runbook.
- **Self-label canonicalization.** Hostnames like `Dee-iMac` vs labels like `laptop` — pick a convention early. Recommend: `AGENT_COLLAB_SELF_HOST` env var, explicitly set at install time, short label (`laptop`, `orange`, `phone`). Hostname is fallback only.
- **Token bootstrap ergonomics.** First `session register` after upgrade needs to surface the token once (write to config, print to stderr). Not secret like an API key — can be recovered by re-register. Document.
- **Who drafts the migration script?** Python CLI is the canonical admin surface. `scripts/peer-inbox-db.py migrate --to v3.3` is the shape.
