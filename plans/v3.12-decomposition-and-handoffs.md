# v3.12.4.5 + v3.12.4.6 — card decomposition and handoffs

**Status:** DRAFT v1 · **Authored:** 2026-04-30 · **Author:** claude (lead-drafting at owner request) · **Trigger:** owner conversation 2026-04-30 — *"a card should keep track of handoffs if the card size is large, but generally cards should be broken into bite-size specs, preferably, and i'm hoping can be done well by claude code at the inception of jobs into cards … i would even go as far to say mid-session card decomposition could be possible."* · **Positions as:** the decomposition + handoff layer that sits between v3.12.4 (pool-aware drainer, shipped) and v3.12.5 (standup-mediated selection, pending). Reduces context-bloat failures by encouraging small cards three different ways: at inception, mid-session, and via context-saturation handoffs.

> **Scope discipline:** This plan covers two sub-phases — v3.12.4.5 (decomposition) and v3.12.4.6 (handoffs). Each is independently shippable and revertible. Together they make cards self-managing for size: most stay small because the decomposer sized them right, the residual gets split mid-run when work reveals hidden complexity, and the rare card too big to even split cleanly emits a handoff to preserve continuity across sessions.

## 1. Goal

Make card size a self-managing property. Three mechanisms, each catching a different failure mode:

1. **Inception-time decomposition** — when a job arrives as an "epic" card, a decomposer agent breaks it into N bite-sized children with dependencies before any work begins.
2. **Mid-session decomposition** — a worker mid-run that discovers hidden complexity can split its card into children, mark itself blocked-on-them, and exit. The drainer picks up the children.
3. **Context-saturation handoff** — a worker hitting its turn budget without finishing emits a structured handoff event. Next dispatch reads the handoff and resumes from where the previous worker left off.

**Owner constraint verbatim (2026-04-30):**
> *"a card should keep track of handoffs if the card size is large, but generally cards should be broken into bite-size specs, preferably, and i'm hoping can be done well by claude code at the inception of jobs into cards"*
>
> *"i would even go as far to say mid-session card decomposition could be possible"*

**Success criteria.**

1. Operator can create a card with `kind=epic`. The drainer recognizes epic cards and dispatches a decomposer instead of a regular worker; the decomposer emits N child cards with appropriate dependencies; the epic flips to `done` when all children are emitted.
2. A card with `splittable=true` can be split mid-run by its worker: the worker emits child cards via MCP, depends on them, comments rationale, releases its claim, and exits. The original card waits in `todo` (with open blockers) until children complete.
3. A card with `track_handoffs=true` running near its turn budget emits a structured handoff event. The next dispatch of that card prepends the latest handoff to the worker's prompt and resumes work.
4. Backward compat: cards without `kind`/`splittable`/`track_handoffs` set behave identically to v3.12.4 (existing harnesses pass unchanged).

**What this plan does NOT do.**

- No persistent agent sessions (still per-card spawn — that's v3.12.3).
- No standup room broadcast or chat-mediated claim (v3.12.5).
- No automated review/lead-challenger flow (v3.12.6 in the parent v3.12 plan).
- No cross-board decomposition — children land on the same `pair_key` as the parent.
- No fancy decomposer model selection — implicit hardcoded prompt; future commit can promote to explicit registry-based decomposer agent.

## 2. Three decomposition moments — architectural shape

```
┌────────────────────────────────────────────────────────────────────┐
│  Inception (v3.12.4.5)                                             │
│                                                                    │
│  Operator pastes job   →  card{kind=epic, body=full job text}      │
│                              │                                     │
│                              ▼                                     │
│  Drainer sees kind=epic → dispatches DECOMPOSER (claude with       │
│                            decomposition prompt)                   │
│                              │                                     │
│                              ▼                                     │
│  Decomposer emits N children via MCP                               │
│                            (card_create + card_add_dependency)     │
│                              │                                     │
│                              ▼                                     │
│  Epic auto-flips to done; pool drains the children normally        │
└────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────┐
│  Mid-session (v3.12.4.5)                                           │
│                                                                    │
│  Worker reads card body → starts work → discovers hidden complexity│
│                              │                                     │
│                              ▼                                     │
│  Worker emits split: N child cards (MCP) + dependencies            │
│                       + 'split' card_event with rationale          │
│                       + card-update-status back to todo            │
│                              │                                     │
│                              ▼                                     │
│  Original card now has open blockers → drainer drains children     │
│  first → original re-becomes ready when children terminate         │
└────────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────────┐
│  Context-saturation handoff (v3.12.4.6)                            │
│                                                                    │
│  Worker prompt instructs: "at turn 25 of 30, write a handoff"      │
│                              │                                     │
│                              ▼                                     │
│  Worker emits 'handoff' card_event (structured JSON in body) and   │
│  exits cleanly                                                     │
│                              │                                     │
│                              ▼                                     │
│  Next dispatch: buildWorkerPrompt sees the handoff event and       │
│  prepends "## Previous handoff" to the prompt; worker resumes      │
└────────────────────────────────────────────────────────────────────┘
```

**Composition.** The three mechanisms compose. Inception decomposition makes most cards small enough to never need the others. Mid-session split catches the cards that turn out to be too big once work begins. Handoffs are the safety net for the residual that even a properly-sized card can hit due to deep context.

## 3. Confirmed design questions

| ID | Question | Decision |
|---|---|---|
| Q-A | Where do decomposer children land? | Same `pair_key` as the parent epic. |
| Q-B | What happens to a card's claim when its worker emits a handoff and exits? | Claim released. Handoff event carries all continuity. Any pool member can resume. |
| Q-C | Is the decomposer a registered agent or an implicit hardcoded prompt? | Implicit hardcoded prompt for v3.12.4.5; future commit can promote to explicit registry-based decomposer if model swaps become useful. |
| Q-D | Default value of `cards.splittable` for human-authored cards vs. decomposer-emitted children? | `false` for human-authored (operator picked their granularity); `true` for decomposer-emitted children (decomposer expects further refinement). |
| Q-E | Cap on child cards per single split? | 5. Beyond that, the worker is required to emit a handoff describing the proposed restructuring instead — human-triggered restructuring for big reorgs. |
| Q-F | Status of the original card after a mid-session split? | Reuse `todo` (with new blocker dependencies). Distinct status (`awaiting_decomposition`) is unnecessary noise; the open-blocker count + `'split'` event in the timeline make the state obvious. |

## 5. Worker prompt additions

Three prompt-template additions in `go/cmd/peer-web/server/run.go:buildWorkerPrompt`. All conditional on the card's flags so unflagged cards see no extra prompt bloat.

### 5.1 Decomposer prompt (when `kind=epic`)

Replaces the regular worker prompt entirely. Output contract:

```
You are a job decomposer for the agent-collaboration kanban.

Read the job below and produce N child cards (1 ≤ N ≤ 12). Each child is
30-90 minutes of focused work for one agent, with one clear acceptance
criterion. If two cards share more than 50% of their context, merge.
Wire dependencies — don't fan everything out flat unless they truly are
independent.

For each child, you MUST call the `card_create` MCP tool with:
  pair_key:    "{parent.pair_key}"
  title:       short imperative (e.g. "Add JWT verification middleware")
  body:        markdown spec with acceptance criterion
  needs_role:  one of {impl, review, qa, design, docs} — pick the closest
  priority:    -1 | 0 | 1
  splittable:  true   (children of decomposers default to splittable=true)

After all children are created, for each blocker → blockee dependency,
call `card_add_dependency`.

When done, exit. The drainer will mark this epic done automatically once
you exit with at least one child created.
```

### 5.2 Mid-session split addendum (when `splittable=true`)

Appended to the regular worker prompt:

```
## You may split this card mid-session

If during work you discover this card's scope is wrong (e.g. it actually
requires three sub-pieces, or has a hidden dependency), you may split:

  1. Create child cards via `card_create` (max 5 children per split).
  2. Wire dependencies via `card_add_dependency` so this card waits on them.
  3. Call `card_comment` with a clear rationale: what you learned that
     made splitting necessary.
  4. Call `card_update_status --status todo` to release your claim.
  5. Exit cleanly.

You MUST declare what changed about your understanding. Splitting to
avoid hard work is not allowed. The audit trail is visible to operators.

If the proposed restructuring requires more than 5 children, do NOT
split — instead, write a handoff describing the proposed restructuring
so a human can approve.
```

### 5.3 Handoff discipline (when `track_handoffs=true`)

Appended to the regular worker prompt + a max-turn ceiling lowered (e.g. `--max-turns=30`):

```
## Hand off if you can't finish

You have approximately 30 turns. If by turn 25 you have not finished,
you MUST write a structured handoff and exit cleanly:

  1. Call `card_handoff` with body containing:
       summary:           one-paragraph status
       decisions:         list of choices made and why
       open_questions:    list of unresolved items
       next_steps:        ordered list for the next session
       files_touched:     list of paths
       context_to_preserve: any constraint or detail the next worker
                            absolutely needs to know

  2. Exit cleanly. Your claim will be released. The next dispatch will
     read your handoff and resume.

If a previous handoff exists, it will be prepended to your prompt.
Read it first and continue from where the previous session left off.
```

### 5.4 Latest-handoff prepend (when `track_handoffs=true`)

`buildWorkerPrompt` queries the latest `kind=handoff` event for this card and prepends:

```
## Previous handoff (from session {ts}, by {author})

{structured body verbatim}

---

(Original card body follows.)
```

## 6. CLI + MCP + HTTP surfaces

### 6.1 Card create gains `--kind` and `--splittable`

CLI:
```
peer-inbox card-create --pair-key K --title T [--kind task|epic]
                       [--splittable on|off] [other existing flags]
```

MCP `card_create`: schema gains `kind` (enum: task, epic) and `splittable` (bool). Defaults preserve current behavior.

HTTP `POST /api/cards` body: gains `kind` + `splittable` fields.

### 6.2 New `card-handoff` verb

CLI:
```
peer-inbox card-handoff --card N [--body STRUCTURED_JSON | --body-file PATH]
                        [--as AUTHOR] [--format plain|json]
```

MCP `card_handoff`: same shape. Body is required; either inline JSON or file.

HTTP `POST /api/cards/{id}/handoff` body: `{summary, decisions, open_questions, next_steps, files_touched, context_to_preserve, blocked_on}`. Server emits a `kind=handoff` event with the body as JSON in the event's body field.

### 6.3 Card update gains `--track-handoffs` toggle

CLI:
```
peer-inbox card-update --card N --track-handoffs on|off [other flags]
```

MCP `card_update`: schema gains `track_handoffs` bool.

HTTP `PATCH /api/cards/{id}/update`: gains `track_handoffs` field.

### 6.4 No new HTTP routes for split

Mid-session split is a sequence of existing MCP calls (`card_create`, `card_add_dependency`, `card_comment`, `card_update_status`) plus emitting a `kind=split` event. The worker prompt instructs the sequence; no new server endpoint required. The `'split'` event kind comes from the migration's extended CHECK.

Optional convenience verb (defer to v3.12.4.7 if needed): `peer-inbox card-split --parent N --children-json PATH` that wraps the sequence atomically.

## 7. Web UI surfaces

### 7.1 Create-card modal: kind + splittable toggles

Add a "Kind" radio (task / epic) and a "splittable" checkbox to the existing create-card modal in `cards.html`. Defaults match the schema: kind=task, splittable=false.

### 7.2 Drawer renders `kind=split` and `kind=handoff` events distinctly

- **Split events:** chevron icon + indented child-card-list with click-through links to the children. Body shows the rationale.
- **Handoff events:** scroll icon + collapsed-by-default expandable card showing structured fields (summary, decisions, etc.).

Both render in the existing activity timeline; no new pane.

### 7.3 Card drawer header shows current state flags

If `kind=epic`: badge "Epic". If `splittable`: small icon. If `track_handoffs`: small icon. Click to toggle (operator-side).

### 7.4 Epic-progress indicator

When viewing an epic, show "N of M children done" derived from the existing `blocker_ids` / status query. Just decoration; no new endpoint.

## 9. Backward compat invariants

- Cards without `kind` / `splittable` / `track_handoffs` set keep current behavior. Defaults: `kind='task'`, `splittable=false`, `track_handoffs=false`.
- All five existing harnesses (cards-poc, cards-phase1-drainer, cards-v312-agents-crud, cards-v312-pool-crud, cards-v312-pool-dispatch) keep passing — their fixture cards never set the new flags, so the old code paths fire.
- `AGENT_COLLAB_WORKER_CMD` env override (used by harnesses to swap claude for a fake worker) continues to work for both decomposer and split flows; the fake just needs to emit the appropriate MCP calls instead of doing real work.
- `card_runs.agent_id`, `card_runs.host`, `card_runs.trigger` reservations from earlier phases are unchanged.
- Existing card_event kinds (comment, status_change, claim, body_update, run_dispatch, run_complete, assigned, unassigned) remain valid. Only adding 'split' and 'handoff' to the CHECK.

## 10. Out of scope (explicit)

- **Persistent agent sessions.** Workers are still per-card spawn in v3.12.4.5/.6. Long-lived sessions are v3.12.3.
- **Standup-mediated card claim.** v3.12.5 — agents see card-ready broadcasts in a peer-inbox room and self-claim via chat.
- **Cross-board decomposition.** Children always land on the same `pair_key` as the parent.
- **Cancellation UI.** Operator-triggered card cancel + worker termination remains a separate item (originally Phase 2).
- **Decomposer model selection.** v3.12.4.5 uses an implicit hardcoded decomposer prompt; an explicit `decomposer` agent role for swapping models is a future commit.
- **Token-budget probing.** v3.12.4.6's handoff discipline is turn-count-based (count by the model itself per its prompt instruction). True token-utilization probing requires runtime instrumentation that doesn't exist in the per-card spawn model — defer to v3.12.3 when long-lived sessions can be observed.
- **Lead/challenger review handoff at in_review.** v3.12.6 in the parent v3.12 plan.

## 11. Open follow-ups (to revisit during smoke or after first ship)

- **Q11.A — Decomposer's MCP authority.** The decomposer needs `card_create` and `card_add_dependency` permissions. Today MCP tools are unrestricted; if a future authz layer lands, the decomposer needs explicit grant.
- **Q11.B — Decomposer turn budget.** Should be lower than worker default (e.g. `--max-turns=15`) since decomposition is a planning task, not implementation. Tune after first runs.
- **Q11.C — Mid-session split idempotence.** What if a worker tries to split a card whose ID it received but the card has already been split by a previous attempt? The dependency check should catch the cycle, but explicit guard ("card already has open blockers from a prior split") is friendlier.
- **Q11.D — Handoff event size cap.** Structured handoffs could grow unbounded over many sessions of one card. Cap event body at e.g. 64 KB; reject larger with clear error.
- **Q11.E — Should mid-session split require `splittable=true`?** Yes (decision Q-D). But operator may want a "force-allow split for this run" override that doesn't promote the card permanently. Defer.
- **Q11.F — Self-promotion of `track_handoffs`.** Plan §8 commit 3 sets `track_handoffs=true` after a turn-budget exhaustion. But the worker exits cleanly only when it understands the discipline — a prompt-naive worker might just hang or fail. Self-promotion should set the flag *and* the next dispatch teaches the discipline.
