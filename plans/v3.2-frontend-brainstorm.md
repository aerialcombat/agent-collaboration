## Framing

Owner framing: *"the frontend is very primitive, although it's very helpful."* Working, but below its potential. Brainstorm opened while coder was heads-down on v3.0 A3 (goose rewrite + remaining queued fixes) so the review panel could produce improvement ideas without idle time.

## Converged identity (3-peer concurrence)

**The dashboard is a LIVE REVIEW-COORDINATION SURFACE.**

Reason's framing, sharper than alpha's "synthesis workbench" and gamma's "orchestration console." Adopted by alpha + gamma + gemini-peer. Defined as much by what it ISN'T as what it is:

- **NOT a transcript browser** — v3.1 `room export` handles durable history better.
- **NOT an orchestration console** — mediator plays that role in-room, not on a screen.
- **NOT a code viewer** — editor + LSP wins there, file:line copy-paste friction is already low.
- **IS the meta-layer:** *who said what, to whom, when, is it still their position, who's seen it.*

Every reviewer's grounded pain from this session lives at that layer:
- "Where are we in this round?" (alpha, re-reading 50+ messages to orient)
- Manually rebuilding reviewer-findings tables in prose (mediator 4×, every reviewer summary re-tabulated)
- Reason's retraction crossing with in-flight drafts (alpha characterized reason's stance stale; mediator's first N1 flip built on stale read)
- Turn counter tripped 100/100 mid-welcome (no early warning)
- 10-15 registered labels, only 5-7 live (no way to see at-a-glance)

## Load-bearing design invariant

**Data model must be mediator-agnostic.** Alpha's catch: "mediated rounds" framing embedded a structural assumption that doesn't generalize. Non-mediated 2-peer lead/challenger rooms are the *more common* shape globally. The findings data model works identically with or without a mediator role — fewer columns fire in the no-mediator case. Gamma retracted "mediated rounds" scoping from their original framing. Don't bake mediator-required fields into the schema.

## Safety gate (mediator ruling)

**Ruling: v3.1 write-time redaction is a hard gate on v3.2 dashboard expansion. v3.2-α, β, γ, δ all ship only after v3.1 redaction lands.**

The exposure shift that forced the ruling:
- **v3.1-scoped problem:** secrets in `~/.agent-collab/sessions.db` on disk — compromise requires local filesystem access.
- **v3.2 dashboard expansion amplifies:** same bodies rendered on `localhost:8787` — accessible to **any process with localhost network access**, including spawned subprocesses, other users on shared machines, anything sandboxed-but-on-loopback. Workbench additions (findings panel, stance-stripe, lean-history, convergence replay) expose MORE structured data in a MORE accessible format.
- **Clean fix — sequence redaction first:** v3.1 write-time redaction sanitizes the underlying rows → workbench inherits safety for free. No duplicated redaction at render-path needed (gemini-peer's specific call, endorsed).

**Reviewer input consolidated into this ruling:**
- reason flagged the dashboard-exposure surface as a v3.1-adjacent safety question.
- alpha elevated it as "probably sequence redaction first; that's gemini-peer's call."
- gemini-peer provided the direct safety recommendation.
- gamma endorsed the sequencing constraint.

Four reviewers concurred on the substance. The mediator ratifies as a hard gate based on the strength of the concurrence + the soundness of the localhost-exposure reasoning. This is a mediator ruling, not a reviewer declaration — preserves the arbitration step for future cases where reviewer concurrence might want to override the mediator's judgment.

## Cross-roadmap sequencing (gamma observation)

**Workbench before v3.1 room-digest carryover.** The structured-state extractor the workbench needs (parses prose → findings / leans / convergences / blockers / roster / turn budget via regex heuristics today, typed payloads post-v4) is the **same machinery** room-digest carryover needs. If digest ships first with its own summarization pipeline, workbench rebuilds equivalent extraction. If workbench ships first, digest becomes a cheap checkpoint serializer (~few hundred LOC of "dump sidebar state to disk") rather than its own pipeline.

**Combined sequencing (safety + architectural):**

1. **v3.1:** export + redaction + papercuts (already converged per `plans/v3.1-roadmap-brainstorm.md`)
2. **v3.2-α:** Synthesis Workbench — gated on v3.1 redaction; produces the structured-state extractor as load-bearing artifact
3. **v3.1.5** (or v3.2-adjacent): room-digest carryover — consumes the extractor as checkpoint serializer
4. **v3.2-β/γ:** convergence replay, stance-stripe seen-by extension
5. **v4:** typed payloads — extractor migrates from heuristic to typed-payload consumer; view layer unchanged

## Ranked roadmap

### v3.2-α — Synthesis Workbench (killer)

Core: **structured findings panel with position-lean history per finding**, embedded in a live-room rendering shell. Three integrated views over one shared structured-state extractor:

- **Findings panel (sidebar):** sortable/filterable table — `{finding_id, reviewer, severity, file:line, claim, state (open/accepted/deferred/routed-to-coder), lean_history, updated_at}`. Groups by severity / file / reviewer. Position-lean history per finding makes flip-thrash visible at a glance (e.g., "alpha: weak (a) → (b) → (a)+cutover; gamma: (b) → (a) → (a)+cutover → abstain; reason: (b) → (a) → (b)"). Retires the highest-recurring manual labor this round — the mediator rebuilt that table 4+ times; every reviewer summary re-tabulated.
- **Stance-at-time stripe (inline, per-message):** compact pills on each message row — `[N1:(b) | A3:(a) | R1:(b)]` — showing the author's concurrent stance across open items at the time the message was sent. Answers "where was each speaker when they spoke?" vs. the sidebar's "where is the room now?" Reason's own retraction-crossing-drafts pain is the canonical repro — a stance snapshot on the outgoing retraction would have made it unmissable.
- **Room health (sidebar):** turn budget meter with color-coded ≥80% warning, active-in-last-5-min member filter (absorbs gamma's turn-cap warning + stale-session filter HMs).

**Extraction strategy (alpha's "don't structure unseen patterns" principle applied):**
- **Today (v3.2):** regex/heuristic pass over prose messages — parses `[[converged]]`, `lean (a)`, `blocker:`, `file:line`, position-update phrasings. 80%-accurate on common cases; edge-case phrasings slip through. Honest ship.
- **Post-v4 (typed payloads):** extractor retires heuristics, consumes `ReviewFinding` / `PositionUpdate` / `RoundConvergence` types directly. Same view layer above; trivial migration.

**Cost (reason's calibration, alpha + gamma accepted):** **5-8 days realistic**, possibly more. The initial 3-5d estimate was optimistic for manual-SSE in Python stdlib + regex state extraction + file:line parsing + iteration on extraction heuristics.

**Retires this round's specific recurring costs:**
- Manual findings-table rebuild (mediator 4+ times, reviewers 2-3×).
- Reason's three crossed-retractions pain.
- Turn-cap surprise at 100/100 (would show ≥80% yellow warning).
- Stale-member clutter in `peer list` output.

### v3.2 papercuts (ship independently, 0.5d each)

**Fold into workbench sidebar, but also land standalone if workbench slips:**

- **Turn-cap warning at ~80% of max** — 3-peer confluence (gamma, alpha, test-engineer as "turn-counter gauge"; unified). Prevents the mid-broadcast surprise this room hit twice.

### v3.2-γ — Test-fixture inspector (orthogonal, test-engineer killer)

**`?db=<path>&home=<path>` query-param plumbing on `cmd_peer_web`.** Lifts the current single-DB-singleton assumption so the dashboard can render any SQLite file — typically a test's `$TMP/sessions.db`.

- **Identity impact:** reason flagged the narrow-vs-broad identity question. Ruling: **narrow identity holds** ("live review-coordination surface"); inspector is a documented side extension, same rendering pipeline over different data source. Doesn't dilute what the dashboard IS.
- **Primary pain:** test-engineer wrote T1/T3/T4 shelling to `python3 -c "import sqlite3..."` a dozen times to verify fixture state — dashboard already renders it all, just single-DB-locked.
- **Second-order win (gemini-peer-endorsed, alpha-endorsed):** **post-mortem mode for cap hits.** When `AGENT_COLLAB_MAX_PAIR_TURNS` fires mid-round (happened this session), owner `cp sessions.db /tmp/frozen.db`, resets, points dashboard at frozen.db → **preserves pre-reset transcript while live room continues.** Partial substitute for v3.1 room-digest carryover at near-zero cost.
- **Safety guard (alpha, non-optional):** path-safelist. Only paths under `/tmp` or `AGENT_COLLAB_INSPECT_ALLOWED_PREFIXES` env allowlist. Do NOT ship "render arbitrary file from disk" attack surface.
- **Cost:** 0.5-1d standalone, or ~2-3h when added on top of the workbench (param plumbing + safety gate; rendering is already there).

### v3.2-β — Three-pane workbench frame + Convergence Replay (additive, if/when owner wants bigger surface)

**Alpha's full workbench:** main transcript pane (threaded rendering of `to`/`mention`/`cohort`, sender pills, timestamps, per-message anchor links) + sidebar + optional live-tail SSE. +2-3d on top of v3.2-α findings panel.

## Meta-observations

Three preserved for the record.

### The room's convergence pattern, again

Same shape as the v3.1 brainstorm: each contributor missed something the others caught, synthesis produced a bundle none proposed in isolation.
- alpha had the workbench frame but missed findings-panel-as-MVP discipline (gamma caught).
- gamma had the findings panel but missed that dashboard expansion amplifies secret-exposure (reason flagged, gemini-peer ruled).
- reason had the stance-stripe but the three-layer composition only emerged after seeing alpha's and gamma's ideas stacked.
- test-engineer had the inspector orthogonal to everyone's frame — the cheap win nobody saw coming.
- gemini-peer had the safety sequencing nobody else quite articulated as a gate.

The v3.1 killer (**typed structured payloads → automated synthesis dashboards**) is the destination for exactly this pattern. v3.2's workbench + extractor is the first concrete step toward it.

### The manual mediator-synthesis labor is the live demonstration

Gamma noted in the v3.1 brainstorm: *"we're the manual SQL query right now."* v3.2's findings panel is the first attempt to automate a slice of that labor. The mediator rebuilt the findings table 4+ times this round from prose; the panel is the "render that table from extractor state" substitution. Same underlying argument for v4 typed payloads (formalize what the extractor already extracts); the panel ships first to prove the pattern works on prose.

### Narrow identity keeps scope contained

Reason's "dashboard is a live review-coordination surface, NOT X Y Z" negative-space definition prevented scope creep twice this round: first when alpha's workbench risked expanding to full transcript-browser + orchestration-console + code-viewer; second when test-engineer's inspector could have reframed the whole tool as a generic SQLite viewer. The narrow identity held; both expansions shipped as compose-on-top features without diluting what the dashboard IS.

### Procedural hygiene catch (reason's drift-watch, at its best)

Both gemini-peer ("Ruling: v3.1 Redaction MUST...") and alpha ("LOCKED... governance-locked, not just preference") used mediator-authority language for reviewer contributions during the brainstorm. Reason's drift-check flagged the pattern: four-reviewer concurrence on substance is strong signal, but "locked" / "MUST" as framing collapses the mediator's arbitration step. Both contributors accepted the correction on broadcast; alpha articulated the underlying principle sharply: *"procedural hygiene matters precisely because it preserves the mediator's authority for cases where reviewers disagree. If reviewer concurrence could declare itself binding, then the next round where four reviewers concur on something the mediator should override loses its arbitration channel."*

Worth preserving — the role boundaries between reviewer concurrence and mediator ratification have to stay intact even when the underlying content is unambiguous, because the boundary only matters on the edge cases where it's load-bearing. This brainstorm's boundary was easy (ratified). A future one might not be.
