## 1. Goal + motivation

Add cross-spawn context continuity for `codex` and `gemini` daemon-spawned CLIs by passing each CLI's native session-ID into subsequent invocations. Result: a daemon-managed `reviewer-codex` (or `-gemini`) label can hold conversation continuity across batches without long-running shells (Architecture A) and without daemon-controlled `ContinuitySummary` envelope-bridging (deferred v1+).

**Architectural framing (mediator broadcast):** Architecture D = keep Arch-B fresh-invocation model BUT pass a stable per-label session-ID into the CLI's native resume flag. Three properties:

1. **No long-running CLI process.** Each batch still spawns a fresh CLI invocation; daemon lifecycle stays unchanged.
2. **Cross-spawn context carry-over.** The CLI vendor's own session-store provides continuity; daemon orchestrates which session-ID to pass.
3. **CLI-vendor-controlled context bound.** Unlike v1+ ContinuitySummary (daemon-controlled, bounded, auditable), Arch D delegates context-bound to the CLI's own management. Trust-boundary trade explicit (see §3).

**Claude asymmetry stated up front:** `claude -p` (the daemon-spawn invocation form) does not expose a stable cross-process session-resume mechanism analogous to `codex exec resume <UUID>` or `gemini --resume <UUID>`. Claude-managed daemons remain Architecture B fresh-invocation; this asymmetry is documented and warned-on at runtime, not failed-on (per Q9 ratification).

**Out of this doc's scope:**
- Long-running CLI shell models (Arch A) — already deferred at Topic 3 v0 scoping.
- Daemon-controlled ContinuitySummary envelope bridging — v1+ scope.
- Per-CLI session-ID granularity in the opt-in flag (e.g., `cli_session_resume: {codex: true, gemini: false}`) — reserved field for v1+; v0.1 ships single global boolean.
- Claude session-resume — out of scope per asymmetry above.
- CLI-side session-store GC (orphaned/stale CLI sessions accumulating server-side) — separate janitor concern; daemon only manages its own pointer to the session-ID.

## 3. Architecture contract

### 3.1 Trust-boundary carveout

Arch D introduces cross-spawn untrusted-state carryover: one batch's spawn output (and any prompt-injection / context contamination it accepts) flows into the next batch's spawn via the CLI vendor's session-store. v0 Architecture B is immune-by-construction; Arch D is not.

**Ratification lever (Q1 ratified):** the opt-in flag itself is sufficient ratification. Default `false` preserves v0 immunity. Operator opt-in to `true` is the explicit acceptance of cross-spawn carryover trade-off. **No §7.1 amendment to the v0 implementation-scope doc is required.**

### 3.2 SSOT opacity (acknowledged)

v0's session-state SSOT is `sessions` table in the inbox DB. Arch D extends this with `sessions.daemon_cli_session_id` (see §5) — the *pointer* to the CLI's session is daemon-owned. The CLI session's *contents* (prompt history, conversation state) live opaque inside the CLI vendor's store (`~/.codex/sessions/<UUID>` or equivalent for gemini). Daemon does not introspect those; cannot enforce content bounds.

This is a deliberate trade vs v1+ ContinuitySummary, which would put context contents inside the daemon-controlled envelope. Q3 reset primitives (§8) are how an operator reasserts control when CLI-side state goes wrong.

### 3.3 Reset primitives (ranked per Q3)

Three primitives, ranked PRIMARY → SECONDARY → TERTIARY:

- **PRIMARY (a) — operator verb.** New verb `peer-inbox daemon-reset-session --as <label>` and Python equivalent `peer receive --reset-session --as <label>`, both shipped in the same commit per the W1 Python+Go parity pattern (Python is the operator surface + parity anchor; Go is the daemon-call fast-path). NULLs `sessions.daemon_cli_session_id` for the named label. Next spawn does not pass `--resume`; CLI vendor allocates a fresh session-ID; daemon captures and persists per §6. **No CLI-side GC attempt** — daemon does not call `gemini --delete-session` or equivalent because orphaned CLI-side sessions are a separate janitor concern (operator's responsibility or CLI-vendor server-side GC).
- **SECONDARY (d) — auto-GC on L1 content-stop.** Wires into `transitionClosed` at `go/cmd/daemon/main.go:787-799`. When the daemon transitions to `daemon_state='closed'` via L1 content-stop (`§7.2 of Topic 3 v0 scope`), simultaneously NULL `sessions.daemon_cli_session_id`. Reopening an L1-closed daemon gets a fresh CLI session-ID by construction.
- **TERTIARY (c) — SQL escape hatch.** Documented in `docs/DAEMON-OPERATOR-GUIDE.md`: operator can `UPDATE sessions SET daemon_cli_session_id = NULL WHERE cwd = ? AND label = ?` directly. Always-available recovery for cases where the daemon is wedged or the verb path is broken.
- **REJECTED (b) — envelope-based reset.** Same trust-boundary concern as v0's rejection of in-band `envelope.state: closed`: cross-peer-controlled in-band signals widen the attack/footgun surface. Operator must initiate reset out-of-band.

### 3.4 Normative constraints (5 invariants)

These are MUST-hold properties — any implementation that violates them is non-conformant and must be fixed before ship-closure. Frames the trust-boundary contract at code-level so reviewers can map findings against them.

1. **Envelope payload remains load-bearing per batch.** Even when `--resume` is in use, the daemon still serializes the full claimed-batch envelope and passes it as the spawn's prompt input. The CLI's resumed session is a *context-augmenter*, not a *content-source*. Rationale: prevents the "CLI remembers, daemon skips delivery" bug class — if the daemon ever delegated message delivery to "the resumed session already saw it," any session-store eviction or vendor-side GC would silently drop messages the inbox marks as completed. Most-critical invariant. Enforced by code: daemon's `processBatch` always builds and passes the envelope; resume only affects the CLI's argv, not the prompt-rendering path.

2. **Daemon MUST NOT introspect CLI session-store contents.** Daemon owns the *pointer* (`sessions.daemon_cli_session_id`); the *contents* are opaque. No reading `~/.codex/sessions/<UUID>/*`, no parsing of gemini's session-history files. Rationale: §3.2 SSOT-opacity stated as architectural fact; this invariant is the operationalization. Violations include any code path that opens, reads, or parses CLI-vendor-internal session storage.

3. **Reset MUST be idempotent.** Calling `daemon-reset-session --as <label>` against a label whose `daemon_cli_session_id` is already NULL must succeed (exit 0) with a clear "already reset" message — NOT an error. Rationale: operators recovering from confused state shouldn't need to know the prior state; reset is safe-to-spam. Same for the auto-GC hook on L1 content-stop — second call after the first has already cleared the column is a no-op, not a failure.

4. **`--cli claude --cli-session-resume=true` combo MUST be non-fatal.** Daemon emits the warning at startup (§4.3) and continues in Arch B fresh-invocation mode. Does NOT exit-78 (EX_CONFIG); does NOT skip the spawn; does NOT alter the prompt. Rationale: shared operator config (one JSON applied across multiple daemon labels with different CLI kinds) must remain ergonomic; failing-hard would force operator to maintain N config files instead of one. Q9 ratification.

5. **Capture-failure MUST be non-fatal.** If §6.1 codex regex fails to match the banner, OR §6.2 gemini delta-snapshot returns zero or multiple new IDs, the daemon logs a warning and leaves `daemon_cli_session_id` NULL. Next batch proceeds Arch B fresh-invocation (equivalent behavior to capture-was-never-attempted). Daemon does NOT crash, retry-spam, or block the batch. Rationale: capture is best-effort enrichment; CLI-vendor UX drift is recoverable via fixture-pin + operator-driven regex update; v0 Arch B is always a safe fallback.

These invariants are the contract reviewers (gamma drift-check, reason synthesis, test-engineer empirical) verify against landed code. Code that compiles, tests that pass, but invariant violation = block-class finding.

## 4. CLI specifics

Parallel to Topic 3 v0 scope §4 (daemon-harness requirements). Each CLI's spawn helper in `go/cmd/daemon/main.go` gets resume-aware variants.

### 4.1 codex

- **Spawn form (resume disabled):** `codex exec --cd <CWD> --skip-git-repo-check < <(printf '%s' "$prompt")` (current Topic 3 v0).
- **Spawn form (resume enabled, with cached session-ID):** `codex exec resume <UUID> --cd <CWD> --skip-git-repo-check < <(printf '%s' "$prompt")`.
- **Spawn form (resume enabled, no cached session-ID — first batch):** identical to resume-disabled form above; daemon captures the new session-ID from stdout banner per §6 and persists for subsequent batches.
- **Workdir caveat:** `--cd` already in v0; unchanged.
- **Stdin caveat:** `< /dev/null` (or piped prompt) already in v0 to prevent codex hang on open stdin; unchanged.

### 4.2 gemini

**Index-vs-UUID note (v3 amendment, Checkpoint 1 finding):** gemini 0.38.2's `--resume` flag is documented as accepting `"latest"` or an `<index>` integer (per `gemini --help`: `Use "latest" for most recent or index number (e.g. --resume 5)`). Direct UUID-resume works empirically but is undocumented. Per mediator ratification (Option A), Arch D builds on the documented index-addressing API while preserving UUID as the stable persisted identity. Capture stores UUID; resume translates UUID → current-index at invocation time.

- **Spawn form (resume disabled):** `gemini -p '<prompt>'` (current Topic 3 v0; cwd inherited from spawn process).
- **Spawn form (resume enabled, with cached UUID):** at each resume, daemon runs `gemini --list-sessions` to enumerate sessions, locates the cached UUID's CURRENT index `N` (indices shift as new sessions push older ones up the list), then invokes `gemini --resume <N> -p '<prompt>'`. The cached UUID is the stable identity; the index is computed-at-resume.
- **Spawn form (resume enabled, no cached UUID — first batch):** daemon snapshots `gemini --list-sessions` BEFORE the first spawn, then runs the spawn without `--resume`, then snapshots `gemini --list-sessions` AFTER. The set-difference yields the new session UUID, persisted to `sessions.daemon_cli_session_id`. See §6.2 for race-tolerance rationale.
- **Workdir caveat:** cwd inheritance unchanged from v0.
- **Timeout-in-ms vs seconds caveat:** unchanged from v0 (handled in install dispatch per Option J fix `b5f9485`).
- **v1+ collapse path:** if gemini documents direct-UUID-resume in a future release, v1+ can collapse the UUID→index translation step (the column shape stays UUID-based for forward-compat).

### 4.3 claude (asymmetry)

- **Spawn form:** unchanged from Topic 3 v0 (`claude -p --settings <SETTINGS> '<prompt>'`).
- **Resume behavior:** none. `cli_session_resume: true` combined with `cli: claude` triggers a runtime warning at daemon startup (per Q9): `"Claude has no cross-process session-resume; --cli-session-resume is a no-op for this daemon (see Arch B asymmetry note in operator guide)."` Daemon does not exit; spawn proceeds in fresh-invocation Arch B mode.
- **Doc surface:** `docs/DAEMON-OPERATOR-GUIDE.md` gets a "Known asymmetry: Claude" subsection explaining the why (no documented stable cross-process session-resume in `claude -p` form), the what (Arch D opt-in is silently ignored for Claude daemons), and the recovery (operator can use Arch B confidently; v1+ ContinuitySummary will be the path forward when needed).

## 6. Capture strategies per CLI

### 6.1 codex — stdout banner regex

- Daemon captures spawn stdout into a buffer.
- After spawn exits (regardless of exit code), scan buffer for the regex `(?i)session id:\s*([0-9a-f-]{36})` (UUIDv4 shape).
- On match: `SetDaemonCLISessionID(session, captured)`.
- On no-match: log warning (banner format may have drifted); leave session-ID NULL (next spawn will be fresh again, equivalent to no-op).
- **Drift-detect test:** `tests/daemon-codex-session-capture.sh` runs against a recorded fixture of the codex banner. Fails on banner-format change → operator must update the regex + bump fixture pin. Mirrors gamma #4 strengthening pattern (fixture-pin MANDATORY for regression coverage of CLI-version-coupled invariants).
- **Future upgrade path:** if `codex exec` exposes structured JSON output (e.g., `--json` flag emitting JSONL with a `session_id` key), upgrade to that for robustness. Note in §10.

### 6.2 gemini — `--list-sessions` delta snapshot (race-tolerant, conservative-on-race)

**Capture (first batch only):**

- BEFORE first spawn: run `gemini --list-sessions`, parse session-UUID set into `before`.
- Run first spawn normally (no `--resume`).
- AFTER spawn exits: run `gemini --list-sessions`, parse into `after`.
- New session-UUID = `after - before` (set difference).
- If exactly one new UUID: `SetDaemonCLISessionID(session, new_uuid)`.
- If zero new IDs: log warning (gemini may have suppressed session creation on this invocation); leave NULL. Per §3.4 invariant 5, capture-failure is non-fatal — next batch proceeds Arch B.
- If more than one new ID (concurrent-gemini race): **CONSERVATIVE default — log warning loudly and leave NULL** (§10 Q2 locked). Daemon does NOT silently pick an mtime-based winner because:
  - silent mtime-pick risks daemon grabbing the operator's interactive gemini session as the daemon's resume target, contaminating both
  - warn-and-NULL is loud-recoverable (operator sees the log, fixes the cause: e.g., quiesce other gemini invocations, or set per-daemon `GEMINI_CONFIG_DIR` for isolation per below)
  - per §3.4 invariant 5, falling through to Arch B is always safe
**Resume (every subsequent batch):**

- Re-query `gemini --list-sessions`; locate the row whose UUID matches the cached `daemon_cli_session_id`.
- Extract the row's CURRENT index `N` (1-based; gemini orders newest-first, so indices shift as new sessions are added).
- If UUID found: invoke `gemini --resume <N> -p '<prompt>'`. Index passed to `--resume`; UUID stays as the persisted identity.
- If UUID NOT found in current `--list-sessions` (CLI-side eviction, vendor GC, operator deletion): NULL the column per §3.4 invariant 5; this batch proceeds Arch B fresh-invocation; next batch will re-capture per the first-batch flow above.

- **Cost:** one extra `gemini --list-sessions` call before and after the first batch (capture); one extra `gemini --list-sessions` call per resume (translation). All cheap enumeration queries; no model invocation.
- **Drift-detect test (MANDATORY pin, parity with §6.1 codex regex):** `tests/daemon-gemini-session-capture.sh` against recorded fixture at `tests/fixtures/cli-session/gemini-list-sessions.txt`. Fails on serialization change. Operator must update fixture + parse logic on gemini-CLI version bumps. Same fixture-pin discipline as gamma #4 strengthening for v0's `claude --settings` MANDATORY assertion.

**Operator-setup paragraph — `GEMINI_CONFIG_DIR` for race avoidance:** gemini-CLI honors `GEMINI_CONFIG_DIR` env var to point at a per-daemon session-store location. For operators running daemon-mode gemini concurrently with interactive gemini sessions in other shells, set `GEMINI_CONFIG_DIR=$HOME/.gemini-daemon-<LABEL>` in the daemon's environment so its `--list-sessions` view is isolated from the operator's interactive gemini sessions. This eliminates the race-by-construction. Documented in `docs/DAEMON-OPERATOR-GUIDE.md` Arch D section. Daemon does NOT auto-set `GEMINI_CONFIG_DIR` — operator opt-in via daemon config, since defaulting it would mask which session-store gemini is using.

### 6.3 claude — N/A

No capture; spawn proceeds Arch B. See §4.3.

## 8. Reset primitives (detail)

Per §3.3 ranking. Implementation specifics:

### 8.1 PRIMARY — `peer-inbox daemon-reset-session --as <label>`

- Go binary subcommand in `go/cmd/peer-inbox/main.go`. Mirrors `peer-inbox daemon-complete` pattern from Topic 3 v0.
- Calls `ClearDaemonCLISessionID(session)`.
- Exit codes: 0 on success; 64 (EX_USAGE) on missing/invalid `--as`; 65 (EX_DATAERR) on session not found.
- Python parity: `peer receive --reset-session --as <label>` in `scripts/peer-inbox-db.py`.
- **No CLI-side GC.** Daemon does NOT call `gemini --delete-session <UUID>` or equivalent. The orphan in the CLI vendor's session store is the operator's or CLI vendor's GC concern.

### 8.2 SECONDARY — auto-GC on L1 content-stop

Wires into `transitionClosed` at `go/cmd/daemon/main.go:787-799`. After `SetDaemonState(closed)` succeeds, additionally call `ClearDaemonCLISessionID(session)`. Reopening (operator flips `daemon_state` back to `'open'`) gets a fresh CLI session by construction.

**L2 has no daemon-side trigger in v0 OR v0.1.** Topic 3 v0 amendment 1 deferred the in-band `envelope.state: closed` consumer to v1+; closure flows through external SQL writes to `sessions.daemon_state`. Therefore an external `UPDATE sessions SET daemon_state = 'closed' WHERE ...` (the §3.3 TERTIARY SQL escape hatch for v0's daemon-state-flip OR a parallel operator action for Arch D) does **NOT** auto-GC the captured CLI session-ID. Operator must additionally run `peer-inbox daemon-reset-session --as <label>` (or the SQL equivalent in §8.3) if both pieces of state should be cleared together. This is intentional separation: `daemon_state` controls whether the daemon claims new batches; `daemon_cli_session_id` controls whether the next claim resumes or starts fresh. Distinct concerns; distinct reset paths.

Auto-GC on L1 is the only daemon-side trigger because L1 (content-stop sentinel in a delivered batch) is the only termination signal v0/v0.1 daemons act on without external intervention. L3 quiescence (poll-cadence backoff, empty-response, pause-on-idle) is "pause" not "terminate" — preserves the resume invariant for waking traffic.

### 8.3 TERTIARY — SQL escape hatch (doc-only)

Documented in `docs/DAEMON-OPERATOR-GUIDE.md` Arch D section:

```sql
-- Drop the captured CLI session-ID for a daemon-managed label.
-- Next spawn will allocate a fresh session.
UPDATE sessions
SET daemon_cli_session_id = NULL
WHERE cwd = '/path/to/cwd' AND label = 'reviewer-codex';
```

No new code. Always-available fallback if Go binary or Python verb is unreachable.

## 10. Challenge-round resolutions (locked)

Original challenge-round open-question list with answers ratified by mediator clustering + owner pass-through.

1. **Codex `--json` upgrade path priority — LOCKED: regex-only in v0.1.** Defer auto-promote to v1+ follow-up. Rationale: regex + MANDATORY fixture-pin (§6.1) is sufficient for v0.1; auto-promote requires confirming current codex-CLI version's structured-output state across the supported version range, which is v1+ scope. `tests/daemon-codex-session-capture.sh` regression-detects banner drift; that's the safety net.

2. **Concurrent-gemini-race UX — LOCKED: conservative warn-and-NULL + `GEMINI_CONFIG_DIR` opt-in.** Per §6.2 and §3.4 invariant 5. Daemon never silently picks an mtime winner because that risks contaminating the operator's interactive gemini session. Loud warning + Arch B fallback is recoverable; per-daemon `GEMINI_CONFIG_DIR` is the operator's race-by-construction prevention.

3. **Reset verb authentication / authorization — LOCKED: any-DB-access (matches v0 `daemon-complete` pattern).** The `peer-inbox daemon-complete` verb landed in Topic 3 v0 with no auth check beyond DB access — v0.1 reset matches the same posture. Operators with DB write access can already corrupt session state directly; adding a verb-level check would be defense-in-depth without a real attacker model. Flag-for-future: if v1+ introduces multi-operator scenarios with bounded permissions, revisit.

4. **Auto-GC scope expansion — LOCKED: defer to v1+.** L1 content-stop is the only daemon-side trigger in v0.1 (per §8.2). L3b pause-on-idle timer-based auto-GC remains a v1+ candidate with its own config field for the threshold. Reason: dropping a still-useful CLI session right before traffic resumes is a real footgun; needs operator-tunable threshold, not a hard-coded default.

5. **Per-CLI granularity v1+ shape — RESERVED FLAG-FOR-AWARENESS.** §7.3 reserves the object-form config syntax for v1+; v0.1 ships single global boolean. The proposed v1+ shape (`cli_session_resume: {codex: true, gemini: false}`) is the working assumption but not locked — v1+ scoping round can revisit per-CLI vs per-label granularity. Not a v0.1 blocker.

5b. **Gemini direct-UUID-resume v1+ collapse — RESERVED FOLLOW-UP** (added per v3 amendment, Checkpoint 1 finding). v0.1 implements the index-translation step (UUID identity in column, `--list-sessions` re-query at resume time, index passed to `--resume`) because gemini 0.38.2's `--help` documents only `"latest"` and index-number semantics for `--resume`. Direct-UUID-resume works empirically in 0.38.2 but is undocumented; v0.1 does not build on undocumented behavior. If a future gemini release documents direct-UUID-resume, v1+ can collapse the translation step in §6.2 (the column shape stays UUID-based; only the resume invocation argv changes). No v0.1 blocker.

6. **Tag name — LOCKED: `v3.x-topic-3-v0.1-shipped`.** Per Q10 ratification + 3-reviewer + coder convergence + owner ratification. `v3.x-topic-3-arch-d-shipped` rejected because it buries the version axis under the architectural axis. Predecessor `v3.x-topic-3-v0-shipped` at `ab09f81` remains the v0 closure anchor; v0.1 layers cleanly.

7. **GOOSE migration ordering re-check — LOCKED: co-commit 1.** Migration 0003 (sqlite + postgres) + GOOSE_VERSION_REQUIRED bump 2→3 in `scripts/peer-inbox-db.py` + `tests/fail-open-fallback.sh` drift-sim update ALL land in commit 1.
