# Topic 3 v0.1 — Architecture D (CLI-Native Session-ID Pass-Through)

**Status:** DRAFT v1 (reviewer-challenge-pending) · **Authored:** 2026-04-18 · **Author:** coder · **Predecessor closure:** Topic 3 v0 at `ab09f81` (tag pending: `v3.x-topic-3-v0-shipped`) · **Pattern:** layered v0.x extension over a shipped v0; mirrors `plans/v3.x-option-j-scoping.md` shape at smaller scale · **Owner ratification source:** mediator broadcast 2026-04-18 (CONSULT #2 + planning round Q1-Q10) — convergent answers locked

> Context: Topic 3 v0 ships fresh-invocation-per-batch (Architecture B). Owner observed both `codex` and `gemini` expose stable cross-process session-resume primitives that v0 deliberately did not consume; PoC confirmed both work end-to-end. This doc scopes a v0.1 additive extension that pipes the CLIs' native session-IDs into spawn invocations behind an opt-in flag, preserving v0 as the safe default.

## 1. Goal + motivation

Add cross-spawn context continuity for `codex` and `gemini` daemon-spawned CLIs by passing each CLI's native session-ID into subsequent invocations. Result: a daemon-managed `reviewer-codex` (or `-gemini`) label can hold conversation continuity across batches without long-running shells (Architecture A) and without daemon-controlled `ContinuitySummary` envelope-bridging (deferred v1+).

**Architectural framing (mediator broadcast):** Architecture D = keep Arch-B fresh-invocation model BUT pass a stable per-label session-ID into the CLI's native resume flag. Three properties:

1. **No long-running CLI process.** Each batch still spawns a fresh CLI invocation; daemon lifecycle stays unchanged.
2. **Cross-spawn context carry-over.** The CLI vendor's own session-store provides continuity; daemon orchestrates which session-ID to pass.
3. **CLI-vendor-controlled context bound.** Unlike v1+ ContinuitySummary (daemon-controlled, bounded, auditable), Arch D delegates context-bound to the CLI's own management. Trust-boundary trade explicit (see §3).

**Claude asymmetry stated up front:** `claude -p` (the daemon-spawn invocation form) does not expose a stable cross-process session-resume mechanism analogous to `codex exec resume <UUID>` or `gemini --resume <UUID>`. Claude-managed daemons remain Architecture B fresh-invocation; this asymmetry is documented and warned-on at runtime, not failed-on (per Q9 ratification).

**Out of this doc's scope:**
- Long-running CLI shell models (Arch A) — already deferred at Topic 3 v0 scoping.
- Daemon-controlled ContinuitySummary envelope bridging — v1+ scope.
- Per-CLI session-ID granularity in the opt-in flag (e.g., `cli_session_resume: {codex: true, gemini: false}`) — reserved field for v1+; v0.1 ships single global boolean.
- Claude session-resume — out of scope per asymmetry above.
- CLI-side session-store GC (orphaned/stale CLI sessions accumulating server-side) — separate janitor concern; daemon only manages its own pointer to the session-ID.

## 2. PoC evidence

Both CLIs confirmed working end-to-end during owner-supervised PoC (room dogfood, 2026-04-18):

**Codex:**
- Workdir assignment: `codex exec --cd <DIR>` honored, scoped per invocation.
- Session capture: stdout banner emits `session id: <UUID>` on first spawn (regex-extractable).
- Resume: `codex exec resume <UUID>` resumes that session in subsequent invocation; cross-spawn context preserved.

**Gemini:**
- Workdir: cwd inherited from shell; daemon already sets the spawn cwd correctly (Topic 3 v0 spawn-harness).
- Session enumeration: `gemini --list-sessions` returns the set of known session-IDs.
- Resume: `gemini --resume <UUID> -p '<prompt>'` resumes the named session with a fresh prompt; cross-spawn context preserved.

**Claude:**
- `claude -p` (the daemon-spawn invocation) has no stable cross-process session-resume documented or PoC'd. `claude --continue` operates per-process within a single CLI run, not across separate `-p` invocations. Out of v0.1 scope.

Reference: PoC results were captured in the mediator's CONSULT #2 broadcast and ratified in planning round Q1-Q10.

## 3. Architecture contract

### 3.1 Trust-boundary carveout

Arch D introduces cross-spawn untrusted-state carryover: one batch's spawn output (and any prompt-injection / context contamination it accepts) flows into the next batch's spawn via the CLI vendor's session-store. v0 Architecture B is immune-by-construction; Arch D is not.

**Ratification lever (Q1 ratified):** the opt-in flag itself is sufficient ratification. Default `false` preserves v0 immunity. Operator opt-in to `true` is the explicit acceptance of cross-spawn carryover trade-off. **No §7.1 amendment to the v0 implementation-scope doc is required.**

### 3.2 SSOT opacity (acknowledged)

v0's session-state SSOT is `sessions` table in the inbox DB. Arch D extends this with `sessions.daemon_cli_session_id` (see §5) — the *pointer* to the CLI's session is daemon-owned. The CLI session's *contents* (prompt history, conversation state) live opaque inside the CLI vendor's store (`~/.codex/sessions/<UUID>` or equivalent for gemini). Daemon does not introspect those; cannot enforce content bounds.

This is a deliberate trade vs v1+ ContinuitySummary, which would put context contents inside the daemon-controlled envelope. Q3 reset primitives (§8) are how an operator reasserts control when CLI-side state goes wrong.

### 3.3 Reset primitives (ranked per Q3)

Three primitives, ranked PRIMARY → SECONDARY → TERTIARY:

- **PRIMARY (a) — operator verb.** New verb `peer-inbox daemon-reset-session --as <label>` and Python equivalent `peer receive --reset-session --as <label>`, both shipped in the same commit per the W1 Python+Go parity pattern (Python is the operator surface + parity anchor; Go is the daemon-call fast-path). NULLs `sessions.daemon_cli_session_id` for the named label. Next spawn does not pass `--resume`; CLI vendor allocates a fresh session-ID; daemon captures and persists per §6. **No CLI-side GC attempt** — daemon does not call `gemini --delete-session` or equivalent because orphaned CLI-side sessions are a separate janitor concern (operator's responsibility or CLI-vendor server-side GC).
- **SECONDARY (d) — auto-GC on L1 content-stop.** Wires into `transitionClosed` at `go/cmd/daemon/main.go:787-799`. When the daemon transitions to `daemon_state='closed'` via L1 content-stop (`§7.2 of Topic 3 v0 scope`), simultaneously NULL `sessions.daemon_cli_session_id`. Reopening an L1-closed daemon gets a fresh CLI session-ID by construction.
- **TERTIARY (c) — SQL escape hatch.** Documented in `docs/DAEMON-OPERATOR-GUIDE.md`: operator can `UPDATE sessions SET daemon_cli_session_id = NULL WHERE cwd = ? AND label = ?` directly. Always-available recovery for cases where the daemon is wedged or the verb path is broken.
- **REJECTED (b) — envelope-based reset.** Same trust-boundary concern as v0's rejection of in-band `envelope.state: closed`: cross-peer-controlled in-band signals widen the attack/footgun surface. Operator must initiate reset out-of-band.

## 4. CLI specifics

Parallel to Topic 3 v0 scope §4 (daemon-harness requirements). Each CLI's spawn helper in `go/cmd/daemon/main.go` gets resume-aware variants.

### 4.1 codex

- **Spawn form (resume disabled):** `codex exec --cd <CWD> --skip-git-repo-check < <(printf '%s' "$prompt")` (current Topic 3 v0).
- **Spawn form (resume enabled, with cached session-ID):** `codex exec resume <UUID> --cd <CWD> --skip-git-repo-check < <(printf '%s' "$prompt")`.
- **Spawn form (resume enabled, no cached session-ID — first batch):** identical to resume-disabled form above; daemon captures the new session-ID from stdout banner per §6 and persists for subsequent batches.
- **Workdir caveat:** `--cd` already in v0; unchanged.
- **Stdin caveat:** `< /dev/null` (or piped prompt) already in v0 to prevent codex hang on open stdin; unchanged.

### 4.2 gemini

- **Spawn form (resume disabled):** `gemini -p '<prompt>'` (current Topic 3 v0; cwd inherited from spawn process).
- **Spawn form (resume enabled, with cached session-ID):** `gemini --resume <UUID> -p '<prompt>'`.
- **Spawn form (resume enabled, no cached session-ID — first batch):** daemon snapshots `gemini --list-sessions` BEFORE the first spawn, then runs the spawn without `--resume`, then snapshots `gemini --list-sessions` AFTER. The set-difference yields the new session-ID. See §6 for race-tolerance rationale.
- **Workdir caveat:** cwd inheritance unchanged from v0.
- **Timeout-in-ms vs seconds caveat:** unchanged from v0 (handled in install dispatch per Option J fix `b5f9485`).

### 4.3 claude (asymmetry)

- **Spawn form:** unchanged from Topic 3 v0 (`claude -p --settings <SETTINGS> '<prompt>'`).
- **Resume behavior:** none. `cli_session_resume: true` combined with `cli: claude` triggers a runtime warning at daemon startup (per Q9): `"Claude has no cross-process session-resume; --cli-session-resume is a no-op for this daemon (see Arch B asymmetry note in operator guide)."` Daemon does not exit; spawn proceeds in fresh-invocation Arch B mode.
- **Doc surface:** `docs/DAEMON-OPERATOR-GUIDE.md` gets a "Known asymmetry: Claude" subsection explaining the why (no documented stable cross-process session-resume in `claude -p` form), the what (Arch D opt-in is silently ignored for Claude daemons), and the recovery (operator can use Arch B confidently; v1+ ContinuitySummary will be the path forward when needed).

## 6. Capture strategies per CLI

### 6.1 codex — stdout banner regex

- Daemon captures spawn stdout into a buffer.
- After spawn exits (regardless of exit code), scan buffer for the regex `(?i)session id:\s*([0-9a-f-]{36})` (UUIDv4 shape).
- On match: `SetDaemonCLISessionID(session, captured)`.
- On no-match: log warning (banner format may have drifted); leave session-ID NULL (next spawn will be fresh again, equivalent to no-op).
- **Drift-detect test:** `tests/daemon-codex-session-capture.sh` runs against a recorded fixture of the codex banner. Fails on banner-format change → operator must update the regex + bump fixture pin. Mirrors gamma #4 strengthening pattern (fixture-pin MANDATORY for regression coverage of CLI-version-coupled invariants).
- **Future upgrade path:** if `codex exec` exposes structured JSON output (e.g., `--json` flag emitting JSONL with a `session_id` key), upgrade to that for robustness. Note in §10.

### 6.2 gemini — `--list-sessions` delta snapshot (race-tolerant)

- BEFORE first spawn: run `gemini --list-sessions`, parse session-ID set into `before`.
- Run first spawn normally (no `--resume`).
- AFTER spawn exits: run `gemini --list-sessions`, parse into `after`.
- New session-ID = `after - before` (set difference).
- If exactly one new ID: `SetDaemonCLISessionID(session, new_id)`.
- If zero new IDs: log warning (gemini may have suppressed session creation on this invocation); leave NULL.
- If more than one new ID: log warning (concurrent gemini invocation in another shell created an additional session); pick the most-recent-mtime entry from gemini's session store, or leave NULL and warn loudly. The race-tolerance choice of set-diff means this case is detected, not silently corrupted.
- **Cost:** one extra `gemini --list-sessions` call before and after the first batch only. Subsequent batches use the cached UUID; no extra calls.
- **Drift-detect test:** `tests/daemon-gemini-session-capture.sh` against fixture of `--list-sessions` output format. Fails on serialization change.

### 6.3 claude — N/A

No capture; spawn proceeds Arch B. See §4.3.

## 7. Opt-in flag surface + precedence

### 7.1 Surfaces (Q8 ratified)

- **Config field:** `"cli_session_resume": true|false` in the daemon JSON config (per `~/.agent-collab/daemons/<LABEL>.json`).
- **CLI flag:** `--cli-session-resume` (boolean toggle on the daemon binary).
- **Env var:** `AGENT_COLLAB_CLI_SESSION_RESUME=1` (0/1 convention matches existing daemon env-vars `AGENT_COLLAB_DAEMON_SPAWN`, `AGENT_COLLAB_FORCE_PY`).
- **Default:** `false`. v0 Architecture B preserved by default. Opt-in is explicit operator action.

### 7.2 Precedence (matches commit 378acaa pattern)

CLI flag > config file > env > default.

Implementation: extends `daemonConfig` struct + `parseFlags` + `loadConfigFile` in `go/cmd/daemon/main.go`. Same shape as the existing TTL/sweep/poll-interval flags landed in commit 378acaa.

### 7.3 Granularity reservation (Q8 ratified)

Single global boolean ships in v0.1. Reserved for v1+: per-CLI granularity in config (`cli_session_resume: {codex: true, gemini: false}`). To support clean v1+ migration, the JSON config schema parses both forms (boolean OR object) but v0.1 only honors the boolean form; object form rejected at parse time with a clear error pointing at v1+. (Defers actual implementation while reserving the field name and shape.)

## 8. Reset primitives (detail)

Per §3.3 ranking. Implementation specifics:

### 8.1 PRIMARY — `peer-inbox daemon-reset-session --as <label>`

- Go binary subcommand in `go/cmd/peer-inbox/main.go`. Mirrors `peer-inbox daemon-complete` pattern from Topic 3 v0.
- Calls `ClearDaemonCLISessionID(session)`.
- Exit codes: 0 on success; 64 (EX_USAGE) on missing/invalid `--as`; 65 (EX_DATAERR) on session not found.
- Python parity: `peer receive --reset-session --as <label>` in `scripts/peer-inbox-db.py`.
- **No CLI-side GC.** Daemon does NOT call `gemini --delete-session <UUID>` or equivalent. The orphan in the CLI vendor's session store is the operator's or CLI vendor's GC concern.

### 8.2 SECONDARY — auto-GC on L1 content-stop

Wires into `transitionClosed` at `go/cmd/daemon/main.go:787-799`. After `SetDaemonState(closed)` succeeds, additionally call `ClearDaemonCLISessionID(session)`. Reopening (operator flips `daemon_state` back to `'open'`) gets a fresh CLI session by construction.

Does NOT trigger on L2 envelope-driven dormancy (because L2 is v1+ deferred per Topic 3 v0 §6 amendment 1) and does NOT trigger on L3 quiescence (because quiescence is "pause poll cadence", not "drop daemon state" — preserves the resume invariant).

### 8.3 TERTIARY — SQL escape hatch (doc-only)

Documented in `docs/DAEMON-OPERATOR-GUIDE.md` Arch D section:

```sql
-- Drop the captured CLI session-ID for a daemon-managed label.
-- Next spawn will allocate a fresh session.
UPDATE sessions
SET daemon_cli_session_id = NULL
WHERE cwd = '/path/to/cwd' AND label = 'reviewer-codex';
```

No new code. Always-available fallback if Go binary or Python verb is unreachable.

## 10. Open questions (for reviewer challenge round)

1. **Codex `--json` upgrade path priority.** §6.1 documents a future upgrade from regex-banner to structured JSON if codex adds it. Should v0.1 attempt to detect-and-prefer JSON output if available (auto-promote), or keep that as a v1+ commit? Recommend: stay regex-only in v0.1, defer auto-promote to a follow-up after merry-acorn confirms current codex-CLI version's structured-output state.

2. **Concurrent-gemini-race UX.** §6.2 says "leave NULL and warn loudly" when `--list-sessions` shows multiple new sessions after a single spawn. Alternative: proceed with most-recent-mtime entry. Owner / reason / merry-acorn / gemini-peer: which is the right default? Conservative (warn-and-NULL) means daemon never silently grabs a wrong session; permissive (mtime-pick) means daemon doesn't fail on a benign race.

3. **Reset verb authentication / authorization.** Operator reset is currently any-caller-with-DB-access. Is that sufficient? v1+ may want a session-key check or owner-label check, but v0.1 simplicity argues against it. Flag for alpha.

4. **Auto-GC scope expansion.** §8.2 wires auto-GC to L1 content-stop only. Should L3b pause-on-idle (timer-driven) also auto-GC after some longer threshold (e.g., daemon idle 6h → drop CLI session-ID)? Avoids long-orphaned CLI sessions; risks dropping a still-useful session right before traffic resumes. Recommend: defer to v1+ (separate timer + config field for the threshold).

5. **Per-CLI granularity v1+ shape.** §7.3 reserves the object-form config syntax for v1+. Is the proposed shape (`cli_session_resume: {codex: true, gemini: false}`) the right granularity, or should it be per-label (`labels.<LABEL>.cli_session_resume: true`)? Per-CLI is simpler; per-label is more flexible. v1+ scope question; flag for awareness only.

6. **Tag name.** §10 (this section) flags both options per Q10 ratification. Coder preference: `v3.x-topic-3-v0.1-shipped` for symmetry with `v3.x-topic-3-v0-shipped` and clearer version-progression than `v3.x-topic-3-arch-d-shipped` (which buries the version axis under the architectural axis). Owner pick.

7. **GOOSE migration ordering re-check.** Per W1 pattern, migration 0003 must land in the same commit as the GOOSE_VERSION_REQUIRED bump in `scripts/peer-inbox-db.py`, AND `tests/fail-open-fallback.sh` drift-sim must drop the new column to keep the rollback path testable.
