# v2.1 Handoff — Room Dynamics, Mediator Mode, and What's Next

Start a fresh session, load this file, pick up with context.

## TL;DR

v2.0 shipped group chats (broadcast, multicast, room view, room-level turn cap). Live testing revealed that **party mode can't produce convergent output** — agents freeze in groups and it's structural, not a model issue. We responded by building three things on top of v2.0:

1. **Room-state meta signals** so agents know whether they're in a pair, a party, or a mediated room.
2. **Mediator mode** — a Delphi-style protocol where one agent facilitates structured rounds.
3. **Join/leave system events** so the room reacts to membership changes (greetings, catch-up).

One live 3-round deliberation with a mediator produced a substantive, shippable synthesis in 9 room turns (budget is 100). The pattern works.

All of this is pushed to `origin/main` — latest commit `c3cfd37`.

## Three room states (this is the mental model)

Agents read incoming `<channel>` meta and pick behavior:

| Room state | Signal | Default behavior |
|---|---|---|
| **PAIR** | `meta.room_size=2` | Reply naturally — no ambiguity |
| **PARTY** | `meta.room_size>=3`, no `meta.has_mediator` | Broadcast by default; start threads; silence OK; flag if the room needs structure |
| **MEDIATED** | `meta.has_mediator=1`, `meta.mediator=LABEL` | Wait for mediator's prompt; reply 1:1 to mediator; don't address other participants directly |

Three message kinds within a mediated room:
- Mediator broadcasts (topic / summary / `[[converged]]`)
- Mediator 1:1 DMs (specific asks)
- Participant 1:1 replies to mediator

Three extra meta fields make the protocol robust:
- `meta.broadcast="1"` → sent to the room
- `meta.cohort="a,b,c"` → multicast to named subset
- `meta.mentions="a,b"` → tagged peers lead the reply (auto-parsed from `@label` in body, or explicit `mention` arg)
- `meta.system="join"` / `"leave"` → membership event, doesn't bump turn counter

## Shipped primitives you can use

### Three delivery modes (unchanged from v2.0 mechanics, expanded prompts)

- **Broadcast** — omit `to` in the tool, or `peer broadcast`.
- **Multicast** — `to: ["a", "b"]` or `peer broadcast --to a --to b`.
- **1:1 direct** — `to: "alice"` or `peer send --to alice`.
All count as ONE room turn regardless of recipient count. `peer_rooms.turn_count` is shared.

### `peer round` (v2.1)

`agent-collab peer round --round N [--label topic|summary|converged] --message "..."` is a thin sugar that broadcasts with a `[Round N]` or `[Round N:LABEL]` prefix. Warns when the caller isn't registered as `--role mediator` but does not block.

Mediator also has full access to `peer send --to <label>` for private per-participant prompts.

### `--role mediator`

Any session can register with `--role mediator`. Once set, every subsequent channel push in that room carries `meta.has_mediator=1` + `meta.mediator=<label>`. Participants adjust behavior automatically (per the updated prompt).

### `templates/peer-inbox-mediator-prompt.md`

Bootstrap prompt for a fresh Claude session being appointed as mediator. Describes the Delphi protocol: propose → DM each participant → synthesize (strip attribution) → broadcast summary → `[[agreed]]` / `[[open]]` / `[[converged]]` markers. Paste it verbatim as the first user message after `session register --role mediator`.

### Join/leave events

- On `session register` into a pair_key, if this is a genuinely new join (first-time or different session_key), auto-emit a `[system] <label> joined the room` broadcast with `meta.system="join"`.
- On `session close`, if the session was in a pair_key, auto-emit `[system] <label> left the room` with `meta.system="leave"`.
- System events land as inbox rows AND channel pushes, but do NOT bump the turn counter.
- Idempotent re-registers stay silent.

## How the live test went (single 3-round deliberation)

Topic: "In a multi-agent group chat collaborating on a code deliverable, WHO should write the code?"

Participants: `alpha` (claude), `gamma` (claude, joined mid-Round-1), `mediator` (claude, role=mediator).

- **22:20** — alpha joined. System event to mediator.
- **22:23** — mediator broadcast `[Round 1] Topic: ...` and DM'd alpha for private position.
- **22:23** — gamma joined mid-round. System event to mediator + alpha.
- **22:23** — **Mediator autonomously DM'd gamma a "Round 1 catch-up"** — repeated the topic and asked for private position. This was emergent behavior from the join event, not scripted. It's exactly the "people greet a new user and orient them" dynamic you described.
- **22:25** — mediator broadcast Round 2 summary with `[[agreed]]` markers + DM'd both for specific follow-ups.
- **22:27** — mediator broadcast `[[converged]]` with full synthesis.

The `[[converged]]` output rejected (a) and (c) outright, collapsed (b) and (d) as the same pattern, extracted 5 load-bearing mechanisms with `[[agreed]]` tags, scaled the rules to N=2 / N=3-4 / N=5+, and honestly flagged residual calibration deltas (LOC gate, hours cutoff) instead of faking consensus on them.

**Total room turns: 9 / 100.** Massively efficient thanks to DM-heavy pattern (DMs don't thundering-herd).

## Known slips and open questions

### Protocol slip: alpha occasionally cc'd gamma

In the live run, `alpha` sent one position to both the mediator AND gamma, when the prompt says "reply 1:1 to mediator only." Not destructive here, but a stricter deliberation would leak positions. Fix: tighten the mediated-mode rule in `peer-inbox-channel.py` instructions from:

> When the mediator DMs you, respond PRIVATELY — reply 1:1 back with `to: "<mediator>"`. Do not broadcast.

to explicitly add:

> Do NOT cc other participants, multicast, or include them in `to`. The mediator is the only recipient of your reply.

One-sentence prompt change. No code.

### Cross-LLM parity: Codex doesn't read the tool description

The channel MCP and tool description are Claude-only. Codex shells out to `agent-collab peer send` / `peer broadcast` — it never sees the mediated-mode rules. A Codex in a mediated room will still broadcast by default because its instructions are in `~/.codex/AGENTS.md`, not in the channel MCP.

**Fix:** update `templates/AGENTS.md` with the three-mode behavior description so Codex reads it at session start. Roughly a copy-paste of the Claude channel instructions with shell-command phrasing instead of tool phrasing. ~40 lines.

Gemini has the same gap (`~/.gemini/GEMINI.md`).

### Open design question: role rotation

The live test used a single Claude mediator. We evaluated dedicated vs rotating earlier (see `v2.0-scoping.md` extensions and the chat transcript leading to this handoff) and landed on **dedicated Claude** for a first test. If/when we want to test rotating mediators (each round, a different LLM takes the mediator seat), we need:

- A clean handoff: how does the new mediator catch up on prior round summaries? (Transcript replay; they'd need a "you are now the mediator, here's what happened" bootstrap.)
- Role mutation: can we re-register a session with a different role, or does the new mediator arrive as a separate session?

Defer unless the dedicated model fails in a specific way.

### Open: viewer doesn't render system events distinctively

The `peer web --pair-key K` viewer shows join/leave events in the main stream like regular messages (grouped into one `@room (N-1)` card since they land as N-1 inbox rows to other peers). They're legible but not visually distinct. A small CSS tweak + meta check would render them as muted grey inline annotations instead of full cards. Low priority.

### Open: mediated-to-party transition on `[[converged]]`

Currently `[[converged]]` is a convention in body text. The participants' prompt says "if the mediator broadcasts `[[converged]]`, the round is over — honor it." But nothing in the system state changes. The room's `has_mediator` flag stays set until the mediator session closes or re-registers without the role. If you want automatic state transition ("after converged, drop mediator role → back to party"), we'd need a round-end command. Defer unless we see churn.
