# v3.6 — Multi-room per session (proper)

## Context

v3.5 landed channel-as-URI and proved bidirectional real-time federation
works end-to-end (orange ↔ laptop over Tailscale, messages land in live
Claude contexts within ~1 RTT). During that test we observed that a single
Claude session **can already** be in multiple rooms today as a side-effect
of the schema — any number of `sessions` rows can share the same
`channel_socket`, and each row carries its own `pair_key`. Proof: my
laptop session was simultaneously in `federation-demo-1776779122` (as
`laptop-client`) and `v35-live-1776823765` (as `laptop`); HTTP pushes
to either landed in my context.

What doesn't work is the **identity resolution** inside the channel MCP
server. Both directions are leaky:

- **Incoming:** the `<channel>` system reminder that Claude sees carries
  a `pair_key=` attribute the MCP stamps from **its own self-resolution**
  (`SELECT ... WHERE channel_socket = ? LIMIT 1`), not from the actual
  push payload. Every message appeared under the first-registered
  room's pair_key regardless of which room it came from.
- **Outgoing:** `peer_inbox_reply` resolves self the same way — it always
  sends as the first-registered (cwd, label, pair_key) tuple. Replying
  into a different room requires the caller to remember which room they
  meant, which the MCP tool has no way to accept today.

This plan makes multi-room **explicit** without schema change. The work is
pure plumbing in the push payload and the channel MCP server.

## Scope

**In:**
- Sender-side: attach `meta.pair_key` to every channel push payload so
  the destination knows which room the message came from.
- HTTP gateway: `/api/channel-push` propagates pair_key through to the
  forwarded unix-socket payload.
- Channel MCP (Python): read `meta.pair_key` on incoming, stamp it onto
  the `<channel>` tag. Return `peer_inbox_reply` arguments with optional
  `pair_key`; default to "the room the last incoming message came from."
- Documentation: a short note in README / CLAUDE.md about registering one
  session into multiple rooms.

**Out:**
- Schema migration. The `(cwd, label)` primary key on `sessions` stays.
  Multiple rows per `channel_socket` is the mechanism; no junction table.
  (Could add one later for explicit membership queries, but not needed.)
- New verb for joining additional rooms. `session register --force
  --label X --pair-key Y` already works to add a second row. A future
  `peer join --pair-key Y` helper is ergonomic polish, not mechanism.
- Federation changes. v3.5 already carries pair_key via the `?pair_key=`
  query param on `/api/channel-push`; this just ensures it's in the
  payload meta too.

## Current breakage observed in v3.5 test

```
<channel source="peer-inbox" from="server-agent" to="laptop"
         pair_key="federation-demo-1776779122"              ← WRONG room
         room_size="1" members="laptop-client">             ← WRONG label
  v3.5 retry with updated binary ...
</channel>
```

The message originated in `v35-live-1776823765` but the MCP stamped the
header with `federation-demo-1776779122` because that row was ordered
first. Delivery worked (push arrived in context) but the recipient
cannot know which room to reply into.

## Design

### 1. Sender injects pair_key into meta

`go/pkg/store/sqlite/send.go` — both unicast and broadcast push paths:

```go
meta := map[string]string{"to": p.ToLabel}
if p.PairKey != "" {
    meta["pair_key"] = p.PairKey   // NEW
}
if len(mentions) > 0 {
    meta["mentions"] = strings.Join(mentions, ",")
}
```

`go/pkg/store/sqlite/system_event.go` — similarly. System events already
carry `meta.system=join|leave`; pair_key joins it.

Python `peer-inbox-db.py` already has the same gap; when the default
impl flipped to Go in v3.4 Phase 5.1 this became a single-location fix
on the Go side. Legacy Python path (AGENT_COLLAB_IMPL=python) gets the
same fix for completeness.

### 2. HTTP gateway passthrough

`go/cmd/peer-web/server/channel_push.go`'s `handleChannelPush`:

```go
// If the payload doesn't already carry meta.pair_key (legacy senders
// or non-Go origins), inject from the query-string for the forwarded
// unix-socket push. Payload-supplied values win over query string.
if meta, ok := payload["meta"].(map[string]any); ok {
    if _, has := meta["pair_key"]; !has && pairKey != "" {
        meta["pair_key"] = pairKey
    }
    payload["meta"] = meta
} else if pairKey != "" {
    payload["meta"] = map[string]any{"pair_key": pairKey}
}
```

This ensures downstream (the unix-socket receiver, then the channel MCP)
always sees `meta.pair_key` regardless of whether the original sender
was a v3.6 Go peer-inbox or an older binary.

### 3. Channel MCP resolves by incoming meta

`~/.agent-collab/scripts/peer-inbox-channel.py`:

**Inbound:** when writing the `<channel>` system-reminder text, read
`pair_key` from `meta` in the incoming push payload. Fall back to self
resolution only if meta.pair_key is absent (which v3.6 senders won't
do, but legacy senders might).

**State tracking:** cache the `(label, pair_key)` pair of the most
recently-received incoming message, keyed per-session. This becomes
the default "I'm replying to" context.

**`peer_inbox_reply` tool:**

- Add optional `pair_key` argument to the tool schema.
- On invocation:
  1. If `arguments.pair_key` given: validate it matches one of the
     session's registered rooms. Use it.
  2. Else: use the cached "last incoming pair_key" for this session.
  3. Else: if the session has exactly one registered room, use it
     (current behavior — backward-compatible single-room path).
  4. Else: return an error asking the caller to disambiguate.

- Pass the chosen pair_key via `agent-collab peer send --pair-key K`
  (the CLI already accepts it; just needs wiring).

**`_resolve_self_from_socket`** returns a LIST of (cwd, label, pair_key)
tuples. Callers pick the right one based on pair_key context.

### 4. Self-resolution for outbound

When `peer_inbox_reply` chooses a pair_key, it must also pick the
correct `(cwd, label)` for that room — because different rooms may
have different labels for the same session (e.g. `laptop-client` in
one pair and `laptop` in another).

```python
def _resolve_self_for_pair(socket_path, pair_key):
    row = conn.execute(
        "SELECT cwd, label FROM sessions "
        "WHERE channel_socket = ? AND pair_key = ?",
        (socket_path, pair_key),
    ).fetchone()
    return (row["cwd"], row["label"]) if row else None
```

The CLI invocation becomes:
```
agent-collab peer send --as <label> --cwd <cwd> --pair-key <pair_key> \
    --to <to> --message-stdin
```

## Test matrix

Three scenarios, all exercisable with tonight's topology (orange tmux
Claude + laptop me + v3.5 peer-web on both):

### M1 — Two rooms on same host, same socket

Setup: register one Claude session into pair A and pair B (two
`sessions` rows, shared `channel_socket`).

- Peer in room A sends to me.
- Peer in room B sends to me.

Expected: both messages arrive with correct `pair_key` in the `<channel>`
tag. Replies via `peer_inbox_reply` without `pair_key` target the room
that sent most recently. Replies with explicit `pair_key` target
that room.

### M2 — Multi-room across federation

Setup: I'm in pair A (home_host=laptop, all-local) and pair B
(home_host=orange, federated).

- Peer on orange (in pair B) sends to me — arrives via HTTP gateway.
- Peer on laptop (in pair A) sends to me — arrives via local unix push.

Expected: both have correct pair_key. Replying into A uses local path,
replying into B uses federated peer-send — the `--pair-key` I pass
determines which home_host the CLI routes to.

### M3 — Ambiguity + error path

Setup: same as M1 but try `peer_inbox_reply` with no pair_key and no
last-incoming (fresh session, or after every prior incoming is too old).

Expected: error message listing the registered rooms, asking caller to
pass `pair_key`.

## Migration

**Zero schema change.** Pure code:

1. Go store: add `meta["pair_key"] = ...` to push payloads in send.go
   and system_event.go. One line each.
2. Go peer-web: gateway passthrough in channel_push.go. ~5 lines.
3. Python channel MCP: two function changes (`_resolve_self_from_socket`
   returns list; `handle_tools_call` reads pair_key arg and cached
   context). ~30 lines.
4. MCP tool schema: add `pair_key` property to `peer_inbox_reply`
   input schema.

Rollout: v3.6 senders inject pair_key into meta. Legacy receivers
(Python MCP pre-v3.6) still work — they'll just ignore the new meta
field, same as any unknown meta. No coordination required between
sides during rollout.

Rollback: `git revert`. Meta fields are additive; revert leaves them
absent, channel MCP falls back to current behavior.

## Deferred to v3.7+

- `peer join --pair-key K` ergonomic verb (syntactic sugar over
  `session register --force --pair-key K` with the same label).
- `peer leave --pair-key K` to deregister a single membership
  without destroying the session.
- UI (peer-web) display of multi-room membership per session.
- Session-membership audit in `session list` — today it only shows
  the first-resolved row per (cwd, label).
