## 1. Goal + motivation

Add `pi` as a 4th `--cli` choice for `agent-collab-daemon`, with the same Arch D opt-in semantics as codex/gemini (cross-spawn context continuity behind `--cli-session-resume`) plus multi-provider routing via pi's built-in `--provider` flag. Three properties that distinguish pi from the existing trio:

1. **Session identity is operator-owned, not CLI-vendor-minted.** pi accepts `--session <PATH>` where PATH is a JSONL session file the daemon owns. No banner regex (codex), no `--list-sessions` delta-snapshot + UUID→index translation (gemini). Daemon mints a path on first batch (default `$HOME/.agent-collab/pi-sessions/<label>.jsonl`), persists it in `sessions.daemon_cli_session_id` (TEXT column reused unchanged from v0.1's migration 0003), reuses on every subsequent batch.
2. **No race conditions, no capture step.** Path-as-identity eliminates the entire capture-and-translate machinery. spawnPi reads the column → if non-NULL, use the path; if NULL, mint + persist.
3. **Multi-provider via `--provider <NAME>`.** pi routes to zai-glm / openai-codex / google-antigravity / anthropic / etc. Daemon passes `--provider` + `--model` from per-CLI config; pi handles the provider-specific auth via env vars (operator's responsibility — daemon never touches API keys).

**Architectural framing:** v0.2 is **additive**. codex, gemini, and claude code paths are unchanged; pi adds as a 4th branch. Operators who don't want pi see no difference. v0.3+ will revisit whether the daemon should collapse all CLI-direct paths into `pi-as-universal-harness` (let pi route to codex/gemini/anthropic providers internally and remove the daemon's per-CLI helpers entirely) — that's a deprecation + migration story that needs its own scoping round.

**Out of this doc's scope:**

- v0.3+ collapse (`pi-as-universal-harness` replacing codex/gemini direct paths) — deferred per Q3 ratification + §11.
- New schema migration — reuses existing `sessions.daemon_cli_session_id` TEXT column (v0.1 migration 0003); the column accepts UUID strings (codex/gemini) OR file paths (pi) interchangeably.
- New store methods — `Set/Get/Clear DaemonCLISessionID` work unchanged with path values.
- Provider auth implementation — daemon does not manage API keys; operator env (ZAI_GLM_API_KEY / OPENAI_API_KEY / ANTHROPIC_API_KEY) flows through `os.Environ()` inheritance per existing daemon practice.

## 3. Architecture contract

### 3.1 Trust-boundary parallel to codex/gemini

Same trust-boundary trade as v0.1: cross-spawn untrusted-state carryover lives in the persisted session file. v0 Architecture B is immune-by-construction (each spawn is fresh). v0.1 Arch D for codex/gemini opted operators in via `--cli-session-resume` to accept the trade. v0.2 pi rides on the same flag (per Q4 ratification — `cli_session_resume: true` enables session-file persistence for pi; default `false` means each pi spawn gets a fresh, ephemeral session).

The opt-in flag remains the explicit ratification lever. **No §7.1 amendment to the v0 implementation-scope doc is required.**

### 3.2 SSOT shape — daemon owns the file path

v0.1's SSOT-opacity caveat (daemon owns the *pointer* but not the *contents*) is preserved with a sharper boundary for pi: the **path itself is daemon-controlled**, but the *contents* (JSONL turns) are pi-vendor-controlled. Daemon writes the path to `sessions.daemon_cli_session_id` and passes it to pi via `--session`. Daemon does NOT read or parse the JSONL session file (per §3.4 invariant 2). pi reads + appends the file as part of its own session-management.

This is a stronger ownership model than codex/gemini (where daemon doesn't even own the path; the CLI vendor's session-store location is internal). For pi, the path is a daemon-controlled string. The trade: daemon's reset semantics for pi can include file deletion (see §8), which codex/gemini's reset cannot do.

### 3.3 Reset primitives (pi-specific subset of v0.1 §3.3 + §8)

Two of v0.1's three primitives apply unchanged; the file-delete side-effect is pi-specific:

- **PRIMARY (a) — operator verb.** `peer-inbox daemon-reset-session --as <label>` (Go) and `peer receive --reset-session --as <label>` (Python). NULLs `sessions.daemon_cli_session_id` AND **deletes the session file from disk** for pi-managed labels (codex/gemini reset stays NULL-only; the file-delete side-effect is gated on `cfg.CLI == cliPi`). The operator-facing message gains a "deleted session file" line for pi.
- **SECONDARY (d) — auto-GC on L1 content-stop.** Same `transitionClosed` extension as v0.1 §8.2; the additional pi-specific behavior is the file-delete on the same hook. Reopening an L1-closed pi daemon gets a fresh session file by construction.
- **TERTIARY (c) — SQL escape hatch + `rm`.** Documented in operator guide: `UPDATE sessions SET daemon_cli_session_id = NULL WHERE ...` + `rm $session_dir/$label.jsonl`. Two-step manual recovery; SQL-only without the rm leaves an orphan file (harmless but accumulates).
- **REJECTED (b) — envelope-based reset.** Same trust-boundary reasoning as v0.1.

**Why pi reset deletes the file:** the path is daemon-controlled, and a stale JSONL file with poisoned context would be re-read by the next pi spawn that uses the same label (since the path is deterministic). NULLing the column without deleting the file would mean the NEXT spawn mints a fresh path BUT a future operator reuse of the same label could resurrect the stale file. Hard-delete eliminates this footgun. (v0.1 codex/gemini reset doesn't delete because the CLI vendor's session-store location is internal — daemon has no path to delete.)

### 3.4 Normative constraints (pi-specific reading of v0.1's 5 invariants)

All 5 invariants from v0.1 §3.4 apply unchanged. Pi-specific notes:

1. **Envelope payload remains load-bearing per batch.** spawnPi MUST still serialize and pass the full claimed-batch envelope as the spawn's prompt input. The `--session <path>` is a *context-augmenter*, not a *content-source*. Same bug-prevention rationale as v0.1.
2. **Daemon MUST NOT introspect CLI session-store contents.** Daemon owns the path string but does NOT open, read, parse, or modify the JSONL session file. pi reads + appends the file; daemon only mints the path + persists it + (on reset) deletes it.
3. **Reset MUST be idempotent.** Calling `daemon-reset-session` against a pi label whose column is already NULL must succeed. The file-delete is best-effort: if the file is already absent, no error (use `os.Remove` with NotExist tolerance).
4. **`--cli claude --cli-session-resume` is non-fatal.** Unchanged from v0.1. Pi+flag is the happy path; no asymmetry warning for pi.
5. **Capture-failure MUST be non-fatal.** For pi, "capture" is just path-mint + persist. Failure modes: SetDaemonCLISessionID SQL error → log warning, leave column NULL, this batch proceeds with an ephemeral session (the spawn still happens; pi creates an unmanaged session file at the minted path). Next batch will retry the persist. No batch ever blocks on capture.

### 4.4 pi (NEW)

- **Spawn form (resume disabled, `cli_session_resume=false`):** `pi --provider <P> --model <M> -p '<prompt>'` — fresh ephemeral session per spawn, no `--session` flag. Pi creates a temporary session internally and discards it on exit (parallel to Arch B for codex/gemini).
- **Spawn form (resume enabled, with cached path):** `pi --provider <P> --model <M> --session <PATH> -p '<prompt>'` where PATH is the cached `sessions.daemon_cli_session_id` value. Pi appends to the existing JSONL file, preserving cross-spawn context.
- **Spawn form (resume enabled, no cached path — first batch):** daemon mints PATH = `$pi.session_dir/$label.jsonl` (default `$HOME/.agent-collab/pi-sessions/$label.jsonl`; operator-overridable via `pi.session_dir` config). Daemon ensures `$pi.session_dir` exists (`os.MkdirAll` with 0700). Persists PATH to the column via `SetDaemonCLISessionID`. Spawns pi with `--session <PATH>`. Pi creates the file on first write. Subsequent batches reuse the persisted path per the previous bullet.
- **Required config:** `pi.provider` + `pi.model` are MANDATORY; daemon startup exits 64 (EX_USAGE) with a clear error naming both fields if either is missing. `pi.session_dir` is optional (default applies).
- **Provider auth:** operator env, not daemon code. Daemon inherits env via `os.Environ()` (existing practice). Documented per-provider env vars:
  - `ZAI_GLM_API_KEY` for `provider=zai-glm`
  - `OPENAI_API_KEY` for `provider=openai-codex`
  - `ANTHROPIC_API_KEY` for `provider=anthropic`
  - `GOOGLE_API_KEY` (or per-vendor equivalent) for `provider=google-antigravity`
  - Other providers: see pi-mono README for canonical env-var names; daemon does not enforce or validate.
- **Stdin caveat:** `< /dev/null` (or piped prompt) consistent with v0 spawn-harness — prevents accidental hang on open stdin. pi's `-p` invocation is non-interactive; matches the existing `cliClaude / cliCodex / cliGemini` execSpawn shape.
- **Workdir:** cwd inherited from daemon (matches gemini behavior). Pi does not require a `--cd` flag.

## 6. Capture strategies

**No capture machinery for pi.** Daemon owns the path string by construction (it mints the path on first batch). The "capture" reduces to a single `SetDaemonCLISessionID` call with the minted path. No regex scanning (vs codex), no `--list-sessions` delta-snapshot or UUID→index translation (vs gemini), no race-tolerance subtests, no stale-UUID-fallback. This is the architectural simplicity of operator-owned session identity.

### 7.2 Surfaces

- **Required vs optional:** `pi.provider` + `pi.model` REQUIRED when `cli=pi` (daemon startup exits 64 with field-naming error if missing); `pi.session_dir` OPTIONAL (default `$HOME/.agent-collab/pi-sessions`, auto-created with 0700 perms on first spawn).

### 7.3 Precedence

CLI flag > config file > env > default. Matches the existing daemon-config-load precedence shipped in commit `378acaa`. Each pi sub-field resolves independently (operator can override `pi.session_dir` via env while leaving `pi.provider` + `pi.model` in the config file).

## 8. Reset primitives (detail)

Per §3.3 ranking. Pi-specific implementations:

### 8.1 PRIMARY — `peer-inbox daemon-reset-session --as <label>` (extended)

For pi-managed labels, the existing `peer-inbox daemon-reset-session` (Go) and `peer receive --reset-session` (Python) verbs gain a file-delete side-effect:

1. Read `sessions.daemon_cli_session_id` for the label.
2. If non-NULL AND `sessions.cli == 'pi'` (or label was registered with `--agent pi`): `os.Remove(path)` with NotExist-tolerance (no error if file already absent).
3. Call `ClearDaemonCLISessionID` to NULL the column.
4. Operator-facing message gains a line: `"deleted session file: <path>"` (only for pi labels; codex/gemini reset message unchanged).

The CLI verb's argv shape stays the same (`--cwd`, `--as`, `--format`); the file-delete is internal behavior driven by the daemon's per-CLI awareness, not a new flag.

### 8.2 SECONDARY — auto-GC on L1 content-stop (extended)

`transitionClosed` (the daemon-struct method on `go/cmd/daemon/main.go`) already calls `ClearDaemonCLISessionID` on L1 content-stop. v0.2 extends the hook for pi-managed labels: read the column value BEFORE clearing, then `os.Remove(path)` after `SetDaemonState(closed)` succeeds. Same NotExist-tolerance.

### 8.3 TERTIARY — SQL escape hatch + `rm` (doc-only)

Operator guide pi section documents the two-step manual recovery:

```sql
UPDATE sessions SET daemon_cli_session_id = NULL WHERE cwd = ? AND label = ?;
```

```bash
rm "$HOME/.agent-collab/pi-sessions/<label>.jsonl"
```

SQL-only without the `rm` leaves the file orphan (harmless — next mint at the same label path will overwrite if pi opens for write; or it stays as a manual-cleanup stub).

## 10. Open questions (for reviewer challenge round)

1. **Session-file GC posture on reset: hard-delete vs rename-to-`.deleted`.** §8.1 + §8.2 currently spec hard-delete (`os.Remove`). Alternative: rename to `<path>.deleted-<timestamp>` for operator recoverability. Coder lean: hard-delete (operator can re-create context via fresh batch; recovery from an LLM-poisoned session file is rarely valuable). Owner pick.

2. **pi-help fixture strictness: full-diff pin vs grep-pattern.** §9.2 gate 5 leans grep-pattern (asserts presence of the daemon-required flags `--provider`, `--model`, `--session`, `-p`) rather than a full-diff fixture. Less brittle to cosmetic pi-mono `--help` changes (whitespace, reorder, flag descriptions). Loses some drift-detection breadth. Coder lean: grep-pattern. If reviewers want stricter, switch to full-diff.

3. **Provider-specific default model.** Q4 ratification said "require both `pi.provider` + `pi.model`" — daemon startup exits 64 if either is missing. This avoids the "default model = glm-4.6 only valid for zai-glm provider" footgun. Re-confirming in this scope-doc: NO `provider_defaults` lookup; both required. Reviewer challenge: is requiring both the right ergonomic trade vs auto-defaulting to a sane per-provider model? Coder lean: keep both required (avoids silent provider/model mismatch).

4. **Multi-provider fake-pi test: single binary with switch-on-provider vs per-provider fakes.** §9.2 gate 2 leans single binary that inspects `--provider` argv to vary behavior. Simpler fixture maintenance (one bash file vs N). Loses per-provider isolation of test failure attribution. Coder lean: single fake. If reviewers want per-provider, split.
