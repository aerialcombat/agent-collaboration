## 1. Goal + motivation

Add `pi` as a 4th `--cli` choice for `agent-collab-daemon`, with the same Arch D opt-in semantics as codex/gemini (cross-spawn context continuity behind `--cli-session-resume`) plus multi-provider routing via pi's built-in `--provider` flag. Three properties that distinguish pi from the existing trio:

1. **Session identity is operator-owned, not CLI-vendor-minted.** pi accepts `--session <PATH>` where PATH is a JSONL session file the daemon owns. No banner regex (codex), no `--list-sessions` delta-snapshot + UUID→index translation (gemini). Daemon mints a path on first batch (default `$HOME/.agent-collab/pi-sessions/<label>.jsonl`), persists it in `sessions.daemon_cli_session_id` (TEXT column reused unchanged from v0.1's migration 0003), reuses on every subsequent batch.
2. **No race conditions, no capture step.** Path-as-identity eliminates the entire capture-and-translate machinery. spawnPi reads the column → if non-NULL, use the path; if NULL, mint + persist.
3. **Multi-provider via `--provider <NAME>`.** pi routes to zai-glm / openai-codex / google-antigravity / anthropic / etc. Daemon passes `--provider` + `--model` from per-CLI config; pi handles the provider-specific auth via env vars (operator's responsibility — daemon never touches API keys).

**Architectural framing:** v0.2 is **additive**. codex, gemini, and claude code paths are unchanged; pi adds as a 4th branch. Operators who don't want pi see no difference. v0.3+ will revisit whether the daemon should collapse all CLI-direct paths into `pi-as-universal-harness` (let pi route to codex/gemini/anthropic providers internally and remove the daemon's per-CLI helpers entirely) — that's a deprecation + migration story that needs its own scoping round.

**Out of this doc's scope:**

- v0.3+ collapse (`pi-as-universal-harness` replacing codex/gemini direct paths) — deferred per Q3 ratification + §11.
- New schema migration — reuses existing `sessions.daemon_cli_session_id` TEXT column (v0.1 migration 0003); the column accepts UUID strings (codex/gemini) OR file paths (pi) interchangeably.
- New store methods — `Set/Get/Clear DaemonCLISessionID` work unchanged with path values.
- Provider auth implementation — daemon does not manage API keys; operator env (per pi-mono's published env-var table, confirmed against `pi --help` at pin-version time) flows through `os.Environ()` inheritance per existing daemon practice.

## 3. Architecture contract

### 3.1 Trust-boundary parallel to codex/gemini + new multi-provider axis

Same trust-boundary trade as v0.1: cross-spawn untrusted-state carryover lives in the persisted session file. v0 Architecture B is immune-by-construction (each spawn is fresh). v0.1 Arch D for codex/gemini opted operators in via `--cli-session-resume` to accept the trade. v0.2 pi rides on the same flag (per Q4 ratification — `cli_session_resume: true` enables session-file persistence for pi; default `false` means each pi spawn gets a fresh, ephemeral session).

**New-in-v0.2 trust axis (reason's scope-round flag):** pi's multi-provider routing makes the daemon a **gateway to ANY provider pi supports** (zai-glm, openai-codex, google-antigravity, anthropic, groq, xai, mistral, etc.). This is a distinct trust surface from v0.1: in v0.1, a `--cli codex` daemon speaks exactly to OpenAI Codex's endpoint; in v0.2, a `--cli pi` daemon speaks to whichever provider `pi.provider` resolves to. Operators opting into pi implicitly trust pi-mono's provider routing + the operator's per-provider env-var setup. Mitigation: `pi.provider` + `pi.model` are REQUIRED at daemon startup (§4.4 / §7.2) — no silent provider default; mismatch between the two (see §10 Q3) rejects at startup. Operator acknowledgment is the opt-in surface.

The `--cli-session-resume` flag remains the explicit ratification lever. **No §7.1 amendment to the v0 implementation-scope doc is required.**

### 3.2 SSOT shape — daemon owns the file path (with blast-radius trade)

v0.1's SSOT-opacity caveat (daemon owns the *pointer* but not the *contents*) is preserved with a sharper boundary for pi: the **path itself is daemon-controlled**, but the *contents* (JSONL turns) are pi-vendor-controlled. Daemon writes the path to `sessions.daemon_cli_session_id` and passes it to pi via `--session`. Daemon does NOT read or parse the JSONL session file (per §3.4 invariant 2). pi reads + appends the file as part of its own session-management.

This is a stronger ownership model than codex/gemini (where daemon doesn't even own the path; the CLI vendor's session-store location is internal). For pi, the path is a daemon-controlled string. The trade: daemon's reset semantics for pi can include file deletion (see §8), which codex/gemini's reset cannot do.

**Blast-radius acknowledgment (gamma T2):** stronger ownership = stronger blast radius if gating is wrong. A reset verb that free-fires `os.Remove` against any `daemon_cli_session_id` value — without checking `sessions.agent == 'pi'` — could attempt to delete paths that don't exist (codex/gemini UUIDs interpreted as $CWD-relative paths) OR, in a pathological codebase where a file happens to match a UUID string literally, delete an unintended file. v0.2 mitigates via explicit `sessions.agent == 'pi'` gate (§8.1) PLUS `os.Remove` NotExist-tolerance as defense-in-depth; a cross-CLI reset-isolation regression gate (§9.2 (D)) locks the gate behavior so refactors cannot accidentally drop the agent-check.

### 3.3 Reset primitives (pi-specific subset of v0.1 §3.3 + §8)

Two of v0.1's three primitives apply unchanged; the file-delete side-effect is pi-specific:

- **PRIMARY (a) — operator verb.** `peer-inbox daemon-reset-session --as <label>` (Go) and `peer receive --reset-session --as <label>` (Python). NULLs `sessions.daemon_cli_session_id` AND **deletes the session file from disk** for pi-managed labels (codex/gemini reset stays NULL-only; the file-delete side-effect is gated on `sessions.agent == 'pi'`). The operator-facing message gains a "deleted session file" line for pi.
- **SECONDARY (d) — auto-GC on L1 content-stop.** Same `transitionClosed` extension as v0.1 §8.2; the additional pi-specific behavior is the file-delete on the same hook. Reopening an L1-closed pi daemon gets a fresh session file by construction.
- **TERTIARY (c) — SQL escape hatch + `rm`.** Documented in operator guide: `UPDATE sessions SET daemon_cli_session_id = NULL WHERE ...` + `rm $session_dir/$label.jsonl`. Two-step manual recovery; SQL-only without the rm leaves an orphan file (harmless but accumulates).
- **REJECTED (b) — envelope-based reset.** Same trust-boundary reasoning as v0.1.

**Why pi reset deletes the file (gamma N1 clarification):** the path is daemon-controlled AND deterministic — `$pi.session_dir/$label.jsonl` resolves to the SAME path every time for a given label. NULLing the column alone would cause the next spawn to re-mint the same path and find the stale JSONL file still on disk; pi would append to it, re-surfacing the poisoned context the reset was meant to discard. Hard-delete is required for reset to actually reset. This is distinct from v0.1 codex/gemini reset, where the CLI vendor's session-store location is internal and the daemon has no path to delete — NULLing the column is sufficient there because the vendor does not cross-reference the next-session mint with the stale one.

### 3.4 Normative constraints (pi-specific reading of v0.1's 5 invariants)

All 5 invariants from v0.1 §3.4 apply unchanged. Pi-specific notes:

1. **Envelope payload remains load-bearing per batch.** spawnPi MUST still serialize and pass the full claimed-batch envelope as the spawn's prompt input. The `--session <path>` is a *context-augmenter*, not a *content-source*. Same bug-prevention rationale as v0.1.
2. **Daemon MUST NOT introspect CLI session-store contents.** Daemon owns the path string but does NOT open, read, parse, or modify the JSONL session file. pi reads + appends the file; daemon only mints the path + persists it + (on reset) deletes it.
3. **Reset MUST be idempotent.** Calling `daemon-reset-session` against a pi label whose column is already NULL must succeed. The file-delete is best-effort: if the file is already absent, no error (use `os.Remove` with NotExist tolerance).
4. **`--cli claude --cli-session-resume` is non-fatal.** Unchanged from v0.1. Pi+flag is the happy path; no asymmetry warning for pi.
5. **Capture-failure MUST be non-fatal; missing-session-file-at-spawn MUST re-create at the same minted path.** For pi, "capture" is just path-mint + persist. Failure modes:
   - SetDaemonCLISessionID SQL error → log warning, leave column NULL, this batch proceeds with an **orphan-until-next-persist** session (the spawn still happens; pi creates an unmanaged session file at the minted path). Next batch will retry the persist. No batch ever blocks on capture.
   - Cached path exists in column, but file is missing on disk at spawn time (e.g., operator ran `rm` out-of-band) → spawnPi passes the cached path unchanged; pi is idempotent w.r.t. file existence and creates the file on first write. The deterministic-path invariant is preserved without column rewrite. This is the specified behavior for §9.2 gate 8a-stale-file.

### 4.4 pi (NEW)

- **Spawn form (resume disabled, `cli_session_resume=false`):** `pi --provider <P> --model <M> --no-session -p '<prompt>'` — fresh ephemeral session per spawn using pi's explicit `--no-session` flag (pi 0.67.68 ships this flag as "Don't save session (ephemeral)"). - **Spawn form (resume enabled, with cached path):** `pi --provider <P> --model <M> --session <PATH> -p '<prompt>'` where PATH is the cached `sessions.daemon_cli_session_id` value. Pi appends to the existing JSONL file, preserving cross-spawn context.
- **Spawn form (resume enabled, no cached path — first batch):** daemon mints PATH = `$pi.session_dir/$label.jsonl` (default `$HOME/.agent-collab/pi-sessions/$label.jsonl`; operator-overridable via `pi.session_dir` config). Daemon ensures `$pi.session_dir` exists (`os.MkdirAll` with 0700). Persists PATH to the column via `SetDaemonCLISessionID`. Spawns pi with `--session <PATH>`. Pi creates the file on first write. Subsequent batches reuse the persisted path per the previous bullet.
- **Spawn form (resume enabled, cached path + missing file on disk):** daemon passes the cached path as `--session <PATH>` unchanged; pi creates the file on first write (idempotent w.r.t. existence). Per §3.4 invariant 5. No column rewrite needed.
- **Required config:** `pi.provider` + `pi.model` are MANDATORY; daemon startup exits 64 (EX_USAGE) with a clear error naming both fields if either is missing. `pi.session_dir` is optional (default applies).
- **Model-provider coupling check (room-codex §10 Q3 ADD):** if `pi.model` contains a provider prefix (e.g., `openai/gpt-4o`) AND that prefix does NOT match `pi.provider`, daemon startup rejects with exit 64 + diagnostic (`"config error: pi.model 'openai/gpt-4o' provider-qualified as 'openai' but pi.provider is 'zai-glm'; provider/model must agree"`). Prevents a silent mis-route class where pi's own flag-parse would accept the model's embedded provider but daemon config contradicts it.
- **Provider auth:** operator env, not daemon code. Daemon inherits env via `os.Environ()` (existing practice). Documented per-provider env vars (confirmed against `pi --help` on pi-mono 0.67.68; future version bumps refresh this table):
  - `ZAI_API_KEY` for `provider=zai-glm`
  - `OPENAI_API_KEY` for `provider=openai-codex`
  - `ANTHROPIC_API_KEY` (or `ANTHROPIC_OAUTH_TOKEN`) for `provider=anthropic`
  - `GEMINI_API_KEY` for `provider=google` (standard Google Gemini path)
  - `provider=google-antigravity` — OAuth-based (no direct env-var in pi's 0.67.68 --help env-table); consult pi-mono README for current auth path, operator responsibility
  - Other providers per pi --help: `GROQ_API_KEY`, `CEREBRAS_API_KEY`, `XAI_API_KEY`, `OPENROUTER_API_KEY`, `MISTRAL_API_KEY`, `MINIMAX_API_KEY`, `KIMI_API_KEY`, `OPENCODE_API_KEY`, `AI_GATEWAY_API_KEY`; Azure via `AZURE_OPENAI_*`; Bedrock via `AWS_*` — daemon does not enforce or validate.
- **Stdin caveat:** `< /dev/null` (or piped prompt) consistent with v0 spawn-harness — prevents accidental hang on open stdin. pi's `-p` invocation is non-interactive; matches the existing `cliClaude / cliCodex / cliGemini` execSpawn shape.
- **Workdir:** cwd inherited from daemon (matches gemini behavior). Pi does not require a `--cd` flag.

Write-sites are all per-CLI (spawnCodex writes UUIDs; spawnGemini writes UUIDs; spawnPi writes paths); no cross-pollination possible.

## 6. Capture strategies

**No capture machinery for pi.** Daemon owns the path string by construction (it mints the path on first batch). The "capture" reduces to a single `SetDaemonCLISessionID` call with the minted path. No regex scanning (vs codex), no `--list-sessions` delta-snapshot or UUID→index translation (vs gemini), no race-tolerance subtests, no stale-UUID-fallback. This is the architectural simplicity of operator-owned session identity.

## 8. Reset primitives (detail)

Per §3.3 ranking. Pi-specific implementations:

### 8.1 PRIMARY — `peer-inbox daemon-reset-session --as <label>` (extended)

For pi-managed labels, the existing `peer-inbox daemon-reset-session` (Go) and `peer receive --reset-session` (Python) verbs gain a file-delete side-effect. **The reset verb is a separate process from the daemon** (see v0.1 §8.1); it reads the shared SQLite database directly, with no IPC to the daemon process. Control flow:

1. Resolve `(cwd, label)` from args.
2. Read `sessions.agent` AND `sessions.daemon_cli_session_id` for the row in a single SELECT. `sessions.agent` is `TEXT NOT NULL` (schema at `scripts/peer-inbox-db.py:303-306` + `migrations/sqlite/0001_*.sql`); populated at `session register` time via `--agent <name>` (INSERT at `scripts/peer-inbox-db.py:1250`). Every row is guaranteed to have a non-empty agent string.
3. If `sessions.agent == 'pi'` AND the column value is non-NULL: call `os.Remove(path)` with `errors.Is(err, os.ErrNotExist)` tolerance (no error if file already absent — satisfies §3.4 invariant 3 idempotency). Defense-in-depth: if `os.Remove` is ever called on a non-pi agent row by mistake (future refactor regression), NotExist tolerance keeps the call from surfacing as an error — a codex/gemini UUID string interpreted as a $CWD-relative path will almost certainly not exist on disk, and the no-op is safe. The `sessions.agent == 'pi'` gate is the correctness boundary; NotExist tolerance is resilience.
4. Call `ClearDaemonCLISessionID` to NULL the column.
5. Operator-facing message gains a line: `"deleted session file: <path>"` (only for pi labels with non-NULL path; codex/gemini reset message unchanged).

The CLI verb's argv shape stays the same (`--cwd`, `--as`, `--format`); the file-delete is internal behavior driven by the row's `sessions.agent` value, not a new flag. No daemon-IPC is needed because the reset verb has full read access to `sessions` via the same SQLite store the daemon writes to. Cross-CLI reset-isolation (resetting a codex/gemini label does NOT attempt `os.Remove` of the UUID string) is locked by §9.2 gate (D).

### 8.2 SECONDARY — auto-GC on L1 content-stop (extended)

`transitionClosed` (the daemon-struct method on `go/cmd/daemon/main.go`) already calls `ClearDaemonCLISessionID` on L1 content-stop. v0.2 extends the hook for pi-managed labels: read the column value BEFORE clearing, then `os.Remove(path)` after `SetDaemonState(closed)` succeeds. Same NotExist-tolerance. Because the daemon process itself owns this path (it's `cfg.CLI == cliPi` in that branch), no `sessions.agent` lookup is needed here — the CLI identity is known at the code-site.

### 8.3 TERTIARY — SQL escape hatch + `rm` (doc-only)

Operator guide pi section documents the two-step manual recovery:

```sql
UPDATE sessions SET daemon_cli_session_id = NULL WHERE cwd = ? AND label = ?;
```

```bash
rm "$HOME/.agent-collab/pi-sessions/<label>.jsonl"
```

SQL-only without the `rm` leaves the file orphan (harmless — next mint at the same label path will overwrite if pi opens for write; or it stays as a manual-cleanup stub).
