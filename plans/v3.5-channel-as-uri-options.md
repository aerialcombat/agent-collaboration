# v3.5 — Channel transport option enumeration

**Status:** research companion to `plans/v3.5-channel-as-uri.md`. This doc
is the full option enumeration + comparison that led to adopting option 9
(channel-as-URI). Kept as a record for future re-evaluation if deployment
assumptions change.

## Problem

Cross-machine federation (v3.3 + v3.4 phase 5.1 item 6) handles the **write**
path: `peer-send` with a room whose `home_host` is remote POSTs to that host's
`/api/send`. The write lands, `server_seq` is assigned, the row is durable.

The **read** path has no push. `sessions.channel_socket` holds a unix path
meaningful only on the host that created it. When orange writes an inbox row
for a laptop-client that lives behind NAT, orange has no way to push into the
laptop's live Claude context.

## Options enumerated

### 1. Everyone long-polls the hub
Every agent, including orange-hosted ones, runs a daemon that pulls
`GET /api/messages?after=<cursor>` from the hub. Writes go POST.
- Uniform code path, but orange-hosted agents burn HTTP loopback to reach
  their own DB; floor latency = poll interval + RTT; violates the
  "orange-agents first-class" constraint.

### 2. Unix socket on hub, long-poll for remotes
Hub-local agents keep the existing unix socket push. Remote agents run a
long-poll receiver.
- Preserves orange-local latency but asymmetric — two code paths; debugging
  "why didn't the push arrive" has different answers per receiver location.

### 3. WebSocket from every peer to hub
Persistent WS connection per agent. Writes multiplexed onto the same socket
or via POST. Hub pushes on new inbox rows.
- Lowest cross-host latency, but stateful server (connection tracking,
  reconnect, heartbeats); orange-local agents pay WS overhead for loopback;
  proxy/middlebox timeouts perennial; ~5x the code of HTTP.

### 4. SSE (Server-Sent Events)
Peer opens long-lived GET that hub keeps open; hub writes events on new rows.
Writes still POST.
- Simpler than WS (HTTP-level, no framing) and EventSource auto-reconnects,
  but same loopback objection for orange-local; one-way (server → client),
  writes need a separate path.

### 5. Message broker (NATS / MQTT / Redis pub-sub)
Hub publishes to topics, agents subscribe.
- Battle-tested but introduces an external dependency + ops surface for a
  10-agent deployment; brokers add a hop everyone pays including local.

### 6. gRPC bidi streaming
Persistent bidi streams per agent.
- Clean bidi model, same shape as WS; Python channel MCP can't easily
  consume; HTTP/2 + TLS config friction.

### 7. libp2p / P2P mesh
Agents discover each other and connect directly.
- NAT traversal needs STUN + relay anyway. Orange becomes the relay.
  Discovery protocol adds complexity. Defeats the "Tailscale handles
  reachability" advantage.

### 8. Shared filesystem (NFS / Ceph)
All agents mount the same DB.
- SQLite + network filesystem = write-lock corruption. Not feasible.

### 9. Channel-as-URI (selected)
`channel_socket` holds either a unix path or an HTTP URL. Write dispatcher
parses scheme. HTTP POSTs land at the receiving host's peer-web, which
forwards to the local unix socket.
- Orange-local unchanged (zero-latency unix push). Cross-host is one HTTP
  round-trip. Uniform auth model. One new endpoint + one new dispatcher
  branch. Requires every peer to be reachable (Tailscale/LAN/VPN assumption);
  not suitable for public-internet agents behind hostile NAT.

### 10. Hybrid: push-when-reachable, poll-else
Advertise a reachable URL if available; else long-poll.
- Best latency possible per-peer but two code paths and more failure modes.

### 11. WebSocket to hub + unix local (hybrid)
Inside a host: unix socket. Between hosts: WebSocket.
- Best possible latency both same-host and cross-host but hybrid complexity
  tax and WS connection management.

### 12. SQLite logical replication (rqlite / LiteFS / Turso)
Stream inbox rows from hub to replicas. Reads local, writes to hub.
- Uniform read model but massive dependency for ~10 agents, conflict
  resolution on broadcast rows, operational complexity.

## Decision matrix

| # | Model | Same-host latency | Cross-host latency | Orange-first-class | Deps | LOC estimate |
|---|---|---|---|---|---|---|
| 1 | Uniform long-poll | High (loopback HTTP) | High (poll interval) | no | none | ~500 |
| 2 | Unix + remote long-poll | Low (unix) | High (poll interval) | yes | none | ~400 |
| 3 | WS everywhere | Medium | Low | no | ws lib | ~1500 |
| 4 | SSE | Medium | Low | no | stdlib | ~700 |
| 5 | Broker | Medium | Low | no | broker | ~600 + ops |
| 6 | gRPC | Medium | Low | no | grpc | ~1500 |
| 7 | P2P | varies | varies | n/a | libp2p | ~3000 |
| 8 | Shared FS | n/a | n/a | n/a | NFS/etc | — |
| **9** | **Channel-as-URI** | **Low (unix)** | **Low (1 RTT)** | **yes** | **stdlib** | **~300** |
| 10 | Hybrid push+poll | Low | Low/High | yes | stdlib | ~800 |
| 11 | WS+unix hybrid | Low | Low | yes | ws lib | ~2000 |
| 12 | SQLite replication | Low | Medium | yes | rqlite/etc | ~800 + ops |

## Why 9 wins

The observation that reframes the problem: **agent-collaboration is a
toolkit for developers operating their own infrastructure**. Its actual
deployments are:
- Single-machine (two Claude sessions, a Codex, a Gemini) — 90% case
- Small team on Tailscale (laptops + a shared VPS) — 9% case
- Public-internet agents with hostile NAT — 1% case

Options 1-7 are sized for case 3. Option 9 is sized for cases 1 and 2 and
falls back cleanly (empty channel URI → receiver polls) for case 3 when
needed.

The key technical insight: the existing `channel_socket` column is a
transport hint. Generalizing from "unix path" to "URI" extends the existing
abstraction rather than bolting a new one alongside. The code delta is a
parser + dispatcher, not a new architectural component.
