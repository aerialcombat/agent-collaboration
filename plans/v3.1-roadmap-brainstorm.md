## Converged roadmap

### v3.1 — "Preservable AND safe" (~4-6 days total, ships right after coder's v3.0 W4)

Order matters. Redaction lands before export because redaction-at-write-to-SQLite makes export safe by default (and makes even local DB compromise non-leaky). Shipping export without redaction would turn the tool into a credential exfiltration vector — secrets leak into messages routinely (env dumps, debug output, error logs, `.env` contents pasted for context), and today they sit contained in local SQLite. Export amplifies the leak surface to PRs, shared transcripts, training data.

**Redaction implementation details** (gemini-peer proposal refined by gamma + alpha):
- **Write-time for new rows, back-redact-at-render for pre-v3.1 rows (dual path).** Sanitize on write to SQLite (in `RedactingTransport`) going forward. For rows written before v3.1's migration (identified by `schema_version < 11`), apply back-redaction at render-time on the export path (and any future cloud-sync path). Rationale: write-back correction of historical rows risks false-positive corruption of legitimate prose; leaving history raw means pre-v3.1 secrets leak through export the moment it ships. Dual-path preserves historical DB integrity while guaranteeing *nothing unsanitized leaves the DB boundary*. Adds ~1 day to the redaction cost estimate.
  - **Resulting safety property:** "All rows leaving the DB (via export or future cloud sync) are redacted. In-place DB contents predating v3.1 are unchanged but never exposed unsanitized."
- **Performance budget: <2ms p99 on 8KB messages.** Redaction is a write-path interceptor. The v3.0 hook budget is <10ms; redaction must not consume that budget for sends. If the regex catalog grows past the budget, cut catalog size before degrading perf — a thinner catalog is a feature-scope problem; blown latency is a user-visible regression.
- **Regex-first, not entropy-first.** Known patterns: `sk-…`, `ghp_…`, `xoxb-…`, AWS `AKIA`, GitHub PATs, JWT shape (3-segment base64 dotted), PEM blocks, `.env`-shaped lines. Entropy detector only as a warn-not-redact signal reviewer can inspect — entropy-based detection misflags ~3-5% of legitimate data (UUIDs, git hashes, session keys).
- **Opt-out: per-message flag, not env-var.** `peer send --allow-secrets` forces conscious per-send choice. Do NOT use env-var-based opt-out like `AGENT_COLLAB_REDACTION=off` — env vars stick across sessions and get left on by accident in long-running agents. Sandbox/dev token sharing for coordination is a real workflow, but should require explicit per-message intent.
- **Auditable markers.** Replace with `<REDACTED_BY_PEER_INBOX:pattern=github_token:audit=NONCE>` — reviewers see *that* something was redacted and *what class* without seeing the value. **Critical: do NOT use a hash of the secret as the audit ID.** Short hashes of known-format-prefix tokens are brute-forceable — a 4-byte hash of a 40-char alphanumeric GitHub PAT with known `ghp_` prefix narrows the search space dramatically. Use a per-redaction random nonce, or derive from `(idempotency_key, byte_offset)`. The goal is "did something get eaten here?" not "is this the same secret as that one over there."
- **Scope disclaimer (v3.1 MVP).** This is a watered-down version of production redaction (AWS Macie, GitHub Secret Scanning are entire products). The 4-6d estimate is for MVP only — ~20 format catalog, baseline FP tuning, per-message opt-in, nonce audit, core adversarial tests. A hardening pass (expanded catalog, provider-specific FP tuning, redaction observability) is an explicit v3.2 or v3.3 consideration, not part of v3.1 scope.

### v3.2 — "Continuity + reliability" (~5-7 days)

- **Room-digest carryover across pair resets and fresh sessions** (gamma, 3-5d). When a pair hits the 100-turn cap or a session ends, write a compact digest (converged decisions, open questions, member roster, converged commitments — positions the room staked that downstream decisions depend on; **not** git commits) to a persistent store keyed by a *semantic room ID*. New sessions joining the same semantic room auto-load the digest at first-turn. Addresses the real continuity gap — gamma hit the 100-turn cap mid-roundtable; a pair reset today would have lost 9 rounds of convergence. Alpha endorsed ranking this ahead of the papercut fixes.
  - **Dogfood datapoint (2026-04-17):** the pair_key `dusty-island-8114` that produced this very roadmap hit the 100-turn cap mid-broadcast while welcoming a new joiner; owner manually reset + raised `AGENT_COLLAB_MAX_PAIR_TURNS=300`. The reset preserved SQLite rows but there's no current mechanism for a future session to inherit our 11 rounds of convergence automatically — this doc (written to `plans/`) is the only retained artifact. Raises this item's priority above a papercut: it's the difference between a roadmap that survives a cap-hit and one that forces an owner to paste context manually into every new session. `reason` elevated this as evidence on re-entry.
  - **Semantic room ID definition:** a stable string (user-supplied at `session register`, or auto-derived from a hash of founding-members + first-topic) that persists across `pair_key` rotations. When a new `pair_key` is minted for the same semantic continuation, sessions joining it inherit the existing digest. Distinct from `pair_key`, which is ephemeral. Open design exercise: exact derivation rule is a v3.2 spec decision.
- **Per-message delivery semantics (`delivery=durable`)** (gamma backlog, 2-3d post-v3.0 idempotency). Ack-or-escalate for load-bearing messages (mediator convergence broadcasts, coder session reports). Best-effort stays default for chatter.
  - **Escalation target — split by recipient agent type** (idle-birch finding): Claude recipients have `UserPromptSubmit` auto-inject available; inject unacked durable as `<peer-inbox urgent="1">` on next prompt. **Codex / Gemini recipients do NOT have auto-inject today** (per `docs/PEER-INBOX-GUIDE.md` runtime table — they require manual `peer receive`). For them the escalation is: persist the `urgent=true` flag on the inbox row, surface on next `peer receive` with a distinct prefix, and print a warning at `session register` time if urgent-unread exists. Equivalent hook paths for Codex/Gemini are out-of-scope until vendor tooling supports them. Mediator is NOT a hardcoded escalation target — room may not have one.

### v3.3 — (removed as a phase)

The bash-level git-diff metadata item was always conditional ("ships as interim if demand materializes before v4"). Round-2 review concluded: committing to a stopgap we'll then replace adds cost without moving the destination. The context-diff capability folds into v4's `SessionReport` typed payload natively — `SessionReport { files_changed, loc_delta, tests_passed }` is exactly the differential metadata block. Shipping the typed version once is cheaper than shipping bash stopgap + typed version.

### v4 — "Mechanize the synthesis role" (weeks, high blast radius)

- **Typed structured payloads with a registered message schema** (gamma, 1w protocol + ongoing adoption effort). Extend `Envelope` (already frozen in v3.0 with `type` + `Any payload` built for this) with typed messages for common collaboration primitives: `ReviewFinding`, `ConvergenceDecision`, `ActionItem`, `OpenQuestion`, `SessionReport`, `RoundConvergence`. Agents emit prose + structured data. Informed by what v3.1's export reveals about recurring shapes — don't formalize patterns you haven't observed. `SessionReport` carries the differential metadata (files_changed, LOC delta, tests_passed) that the removed v3.3 stopgap was going to deliver.
- **Git-aware `ReviewFinding` binding** (gamma backlog, 1-2w, depends on typed payloads). Review payloads carry `{file, line_range, severity, claim, evidence}`. Aggregatable like inline PR comments.
- **Automated synthesis dashboards** (gamma's real long-game). Shows open questions, pending reviews, convergence state — mechanized, not hand-synthesized. Mediator's current manual job becomes a SQL query.
  - **Dependency: `cmd_peer_web` migration.** The current admin dashboard is a 453-LOC Python HTTP handler at `scripts/peer-inbox-db.py:1829`, not ported in v3.0. v4 dashboard work picks one path, names it explicitly in its own spec: **(a)** keep Python `cmd_peer_web` alive alongside Go forever (dual-maintenance cost); **(b)** port to Go as a v4 prerequisite (~3-5d, adds to v4 scope); **(c)** build a Go-native dashboard from scratch on top of typed payloads (~1w, throws away Python UX but gives clean typed-payload-native foundation). Leaning (c) as the cleanest fit for v4's typed-payload substrate, but this is a v4 spec decision, not a v3.1 lock-in.
- **CI integration.** `ReviewFinding` with `severity=blocking` blocks merge.
- **Conversation forking / subthreads** (gamma backlog, 1w). Turn-budgeted subtopics inside a room with merge-back semantics.
- **Agent-capability matching for routing** (gamma backlog, 1w). "Find a peer with `agent=codex` and `role=reviewer`" instead of name-addressing.

## Principle that emerged

**"Don't structure data you haven't seen the shape of yet."** (alpha)

Export reveals the shape. Typed payloads then formalize the recurring patterns. Sequencing is the win, not picking a single killer. v3.1 ships what's already in the DB as safe artifacts; v4 formalizes the patterns those artifacts reveal.

## Rejected / deferred

- **Context-differential broadcasts as a standalone v3.1 item OR as a v3.3 bash-level stopgap.** Gemini-peer proposed 2d; gamma and alpha agreed it overlaps significantly with typed payloads (`SessionReport.files_changed` IS the differential metadata). **Resolution: folds into v4 `SessionReport` typed payload, no interim.** Shipping a bash stopgap + typed version later would cost more than shipping the typed version once; v3.3 is no longer a phase.
- **Entropy-based secret detection as primary signal.** False-positive rate too high (~3-5% on UUIDs/git-hashes). Regex-first, entropy as warn-only.
- **VoteCast / ActionItem as first typed payloads.** Unproven in actual rooms; ship `ReviewFinding` + `RoundConvergence` first (clear winners based on this session's patterns); validate others against real export data before adding.
