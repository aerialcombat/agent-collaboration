## 1. Two coupled topics (per mediator's opening framing)

**Topic 1 — Headless delegation.** Current model: each agent CLI is its own session; human spins terminals for Claude Code / Codex / Gemini separately; peer-inbox coordinates between them. Proposed model: one human-facing orchestrator session that spawns headless workers (agent-agnostic), which receive tasks via peer-inbox or IPC, respond via peer-inbox, and are either ephemeral per-task or long-running role-holders.

**Topic 2 — Non-CLI LLM invitation.** DeepSeek, GLM, Qwen, Mistral — LLMs with APIs but no canonical CLI equivalent of the Claude/Codex/Gemini triad. Today they can't participate in peer-inbox (no hook, no session_key, no CLI to register). Proposed: adapter that polls inbox, makes API calls, posts replies. Adapter = the headless worker from Topic 1.

**Collapsed design question:** what's the minimum participation contract for a peer-inbox room participant, such that ephemeral API-backed workers and long-running CLI sessions fit the same model?

## 2. Evidence of the gap (from current session)

- Mediated review rounds in `dusty-island-8114` consistently bottleneck on "spin another terminal" / "paste a prompt" / "session goes stale" overhead. idle-birch went silent mid-W1-round-2 without anyone noticing for turns.

  - Non-CLI LLM families (DeepSeek, GLM, Qwen, Mistral) are fully excluded from the current protocol — not just laggy, non-participatory.

### 4.2 S2 — Existing orchestration patterns (gamma 2026-04-18, resolved)

**Four archetypes across 2026-era multi-agent frameworks:**

| Framework | Shape | Identity / lifecycle | Model-agnostic? |
|---|---|---|---|
| **LangGraph** | Directed graph with conditional edges; built-in checkpointing with time-travel persistence | Graph nodes are named; state explicit in graph state | Yes |
| **CrewAI** | Role-based crews; sequential task-output passing; mirrors human-team abstraction | Roles are named; crews are persistent | Yes |
| **AutoGen / AG2** | `GroupChat` conversation pattern; in-memory conversation history by default | Agents are named in the group; conversation persists in-memory | Yes |
| **OpenAI Agents SDK** | Explicit handoffs via context variables; ephemeral-by-default | Context-variable-scoped; agents instantiated per run | OpenAI-only |

Sources: [Best Multi-Agent Frameworks 2026](https://gurusup.com/blog/best-multi-agent-frameworks-2026) + Datacamp + Composio comparisons.

**Claude Agent SDK — specific shape worth calling out:**

- Supports orchestrator-spawns-subagents pattern by default. Subagents share the main agent's session context and report back results. Defined in `.claude/agents` via Markdown + YAML frontmatter. Source: [Claude Agent SDK building guide](https://www.anthropic.com/engineering/building-agents-with-the-claude-agent-sdk), [Agent SDK overview](https://code.claude.com/docs/en/agent-sdk/overview).
- Anthropic also offers **Agent Teams** (separate from SDK subagents) — "coordinate multiple Claude instances with independent contexts and direct communication between agents." Closer to Topic 1's model than the shared-context subagent pattern, but Teams is Anthropic-hosted, not local.
- Gap: local worker orchestration via SDK ≠ cross-vendor peer-inbox-room participation.

**Finding 3 — MCP as the convergent tool-integration layer:**

All four 2026-major frameworks (LangGraph + CrewAI + AutoGen + OpenAI SDK) converge on MCP as the tool-provisioning layer. Workers built on top of any of these call MCP-compatible tool servers during their reasoning loop.

**Implication for Topic 2 (non-CLI LLM bridging):** if peer-inbox participation is exposed as MCP tool-calls (`peer_inbox_send` / `peer_inbox_receive`), adoption cost for workers built on any of the four frameworks is near-zero. This reinforces alpha's S4 observation that inbox routing is already identity-agnostic — the MCP tool surface is the natural integration point for non-Claude-Code participants.

**Cross-strand observation (gamma, preserved for synthesis lane):**

> S4's `(workspace_id, label)` minimum contract + S2's MCP-tool convergence suggest the ephemeral-worker participation model is: (a) register with workspace_id + synthetic label, (b) participate via peer-inbox MCP tool-call (send/receive), (c) no cwd, no marker, no hook dependency. This decouples orchestration from the CLI-hook delivery path Option J establishes.

This is one of several possible framings; alpha/test-engineer/gemini-peer should challenge it before synthesis adopts it. But the shape-match between S2 MCP-convergence and S4 participation-contract is notable.

### 4.4 S4 — Peer-inbox contract implications (alpha 2026-04-18, resolved)

**Current contract (what's load-bearing, with file:line citations):**

- **Sessions table:** `PRIMARY KEY (cwd, label)` at `scripts/peer-inbox-db.py:294-303`. Every session keyed on a real directory + a label. `session_key TEXT` non-unique indexed at `:318`, identity-continuity primitive. `agent TEXT NOT NULL` with `VALID_AGENTS = {"claude", "codex", "gemini"}` closed whitelist at `:33,270-273`. `started_at` + `last_seen_at` drive idle/stale semantics.
- **Session-key discovery cascade** at `cmd_session_register:1077-1147`: `--session-key` arg → env vars (`SESSION_KEY_ENV_CANDIDATES`, Track B B1 drift surface) → `recent_seen_sessions` (Claude hook log) → fresh UUID for codex/gemini (`:1119-1137`, pattern `auto-{agent}-{uuid}`) → fail with config error.
- **Marker semantics** at `go/pkg/store/sqlite/sqlite.go:209-243` (W1 `store.ErrPathDrift`): markers at `<cwd>/.agent-collab/sessions/<hash>.json`; marker's recorded `cwd` MUST match on-disk owner via `filepath.EvalSymlinks`. Assumes a real cwd.

**What breaks for ephemeral / API-backed / orchestrator-delegated workers:**

1. **`cwd` is required.** API-backed participant (DeepSeek, GLM, Qwen, Mistral) has no directory. Two paths: (a) workers share orchestrator's cwd (multi-label-per-cwd already works per W1 A2); (b) synthetic cwd like `/agent-collab/workers/<uuid>` — requires relaxing `resolve_cwd`'s `Path.resolve(strict=True)` at `:159`. **Path (a) is cheaper.**
2. **`VALID_AGENTS` is a closed set.** Adding `api-deepseek`, `api-glm`, etc. requires either loosening the whitelist or adding an `agent_kind` column distinguishing CLI vs API participants. Schema-additive change — goose-v3.1+ fodder.
3. **Marker path-drift assumes real cwd.** Ephemeral workers without real markers either (a) skip marker creation entirely (orchestrator owns identity) or (b) markers live in a synthetic location. Affects `readMarker` semantics.
4. **Session-key cascade adds 5th surface.** Track B B1 just got worse — orchestrator-spawned worker env joins the existing 4-source list. Single-source-of-truth fix becomes load-bearing if orchestrator model lands.

**What stays unchanged:**

- **Inbox routing** (`inbox` table at `:304-313`): `(to_cwd, to_label)` routing works regardless of identity origin.
- **Room semantics** (`peer_rooms` + `pair_key` at `:481-490`): orchestrator + workers all share a pair_key; broadcast/multicast fan-out is identity-agnostic.
- **Hook delivery** (for hook-equipped workers per S1): Option J extends naturally.
- **Idempotency / mtime marker** (`mark_inbox_dirty` at `:452`): dirty-marker mechanism doesn't care who's writing.

**Alpha's synthesis-ready position** (preserved for §7 when synthesis opens): *"Minimum participation contract = `(workspace_id, label)` not `(cwd, label)`. cwd becomes a presentation field for CLI sessions; ephemeral workers use synthetic or shared values. Schema change is additive (workspace_id already reserved per W1). v3.2-or-later scope."*

**Adjacent surfaces (Track B carryover):**
- **B1** (shared `SESSION_KEY_ENV_CANDIDATES`) promoted to HIGH priority — ephemeral workers stress this.
- **B3** (`workspace_id` default 'default' named constant) reinforced as v2 prereq — workspace_id is the natural partition for "ephemeral workers vs CLI sessions." Matches the minimum-participation-contract direction.

## 5. Options surface (seed — expand during brainstorm)

### 5.1 Orchestrator architecture shapes

- **O1. No orchestrator; current model.** Keep one-terminal-per-CLI. Non-CLI LLMs remain excluded. (Status quo baseline — what we're comparing against.)
- **O2. Human-driven orchestrator (shell script).** A local bash script spins `claude -p`, `codex -p`, `gemini -p`, or API calls, serialized by the human's terminal. Simple; still human-in-the-loop for high-level coordination.
- **O3. Claude-Code-as-orchestrator.** One Claude Code session is the orchestrator; it uses Bash + tool-use to spawn worker invocations and relay peer-inbox messages. Natural fit for this repo's existing patterns.
- **O4. Dedicated orchestrator daemon.** A separate long-running Go/Python process (like `peer-inbox-orchestrator`) manages worker lifecycle + routing. Deeper investment; more capable.
- **O5. Cloud-hosted orchestrator.** Same as O4 but hosted, accessible from any dev machine, shared across team. Returns the cloud question from Track C under a new binding constraint.

### 5.2 Worker shapes

- **W1. Per-task ephemeral workers.** Worker spawned on `peer send --to reviewer`, runs the CLI/API once, posts reply, exits. Simplest; loses conversation continuity.
- **W2. Long-running role-holder workers.** Worker persists for a session (registers with a label like `reviewer-claude`), receives multiple tasks over time. Better continuity; resource-heavier.
- **W3. Hybrid — long-running shell, ephemeral LLM calls.** Worker process stays alive (peers call it by label), but each task is a fresh LLM invocation with the minimum context needed. Balances continuity-of-identity with compute efficiency.

### 5.3 Non-CLI LLM integration shapes

- **N1. Direct API adapter per provider.** `agent-collab-bridge-deepseek`, `-glm`, `-qwen`, `-mistral` — separate adapter binaries. High per-LLM effort; tight integration.
- **N2. Generic LiteLLM-backed bridge.** One adapter using a unified LLM-API proxy like LiteLLM; supports many providers with one binary. Lighter per-LLM effort; less native.
- **N3. MCP-tool-backed bridge.** A peer-inbox-aware MCP tool that any LLM with MCP support can call. Pushes integration burden onto each LLM's MCP client rather than agent-collab-side.

## 6. Crosscuts and constraints

- **Identity & authentication.** Session-key model works for CLI sessions with env-based identity; headless workers with ephemeral identities need a different lifetime model. API-backed workers need API-key management separate from peer-inbox identity.
- **Turn-boundary semantics.** Hooks fire at a CLI's turn boundary. Headless workers via `-p` may or may not have "turn" as a well-defined concept; API-backed workers definitely don't have turn-boundary hooks. What replaces hook-delivery for headless mode?
- **Tool-use parity.** CLI sessions have rich tool use (file-read, bash, etc.) via the CLI's runtime. API-backed workers need those tools provisioned separately — does the orchestrator provision tools per worker, or does the worker bring its own?
- **Reliability / fail-open.** Existing peer-inbox fail-open discipline (never block the user's turn on inbox unavailability) extends here. Workers may fail silently; orchestrator needs to notice and route accordingly.
- **Cost model.** Always-on workers are more expensive than ephemeral ones. Ephemeral ones lose conversation context, which the orchestrator would have to persist externally. The tradeoff lands differently for Claude Code (billed at rate X) vs API-backed (billed at tokens).
- **Privacy.** Same as Track C: cloud options need redaction (v3.1 gate) before carrying message bodies. Local-orchestrator options don't cross that gate.

### 7.3 Orchestrator × worker-shape matrix against the scope

**Orchestrator axis (§5.1):**
- **O1 — No orchestrator:** cannot support auto-reply by construction.
- **O2 — Human shell script:** human-in-loop contradicts "auto-reply daemon" semantics.
- **O3 — Claude-Code-as-orchestrator:** inherits the human user's tool permissions, identity, and working context (advantage O4 re-establishes from scratch). **Orchestrator survivors under Topic 3 scope: O4 primary (v0), O5 cross-host re-opener only (v3.3+), O3→O4 hybrid named as future variant (not v0).**

**Worker-shape axis (§5.2):**
- **W1 — Per-task ephemeral:** CLI process spawns, runs once, exits. Fits item 1 (fresh LLM). Requires orchestrator to pass stable session_key per spawn (item 5 not auto-satisfied). Pays CLI startup cost per message.
- **W2 — Long-running role-holder with persistent LLM session:** contradicts item 1 (requires `--resume`). Not carried forward.
- **W3 — Hybrid (long-running shell, ephemeral LLM calls):** each LLM invocation fresh (item 1). W3 benefits: (a) amortizes CLI *shell* startup cost across messages (LLM invocation stays fresh per item 1; W3 does NOT keep warm LLM context — that's W2, ruled out); (b) daemon shell holds the sessions-table row across spawns, avoiding sessions-table balloon from per-spawn re-registration. **Item 5 is satisfiable under either W1 or W3** given orchestrator-side env-var propagation — gamma's earlier "W3 structurally satisfies" was imprecise (alpha concession 2026-04-18). Explicit v0 implementation requirement (applies to all three confirmed CLIs): *daemon shell exports a stable session-key env var (`CLAUDE_SESSION_ID` / `CODEX_SESSION_ID` / `GEMINI_SESSION_ID` or `AGENT_COLLAB_SESSION_KEY`) into every spawned LLM's environment; ephemeral LLM never calls `session register`; daemon owns identity lifecycle, worker inherits.* Tradeoff vs W1: W3 has session-memory-leak risk (CLI may cache shell state across turns); W1 has cleaner per-process boundary but pays shell-startup cost per message.

**Worker-shape survivors: W3 primary (v0, medium-cadence rooms; amortization wins), W1 viable for low-frequency / low-label-count scenarios where state-leak risk outweighs shell-startup savings.**

**(c) Hang-detection vs crash-detection (reinforces item 9; mechanism + schema + path-separation converged from alpha+gamma).** Gamma's Cursor finding surfaces documented `cursor-agent -p` hangs. Item 9 needs to handle hang-after-claim, not just crash-after-claim — hang is harder to detect (crash leaves a zombie process or exit code; hang leaves a live process doing nothing). No cross-path row-state sharing — de-risks by construction, not coordination discipline. Schema addition (`claimed_at` + `completed_at`, additive NULLable): Python + Go schema ADD in same migration commit; daemon-mode runtime consumption wires up in subsequent commits per runtime. Hook-parity gate held constant throughout. v0-applicable for the trio regardless of Cline/Cursor out-of-scope.
